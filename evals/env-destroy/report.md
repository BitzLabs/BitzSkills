# env-destroy 評価レポート

- **評価日**: 2026-07-12
- **対象**: evals/env-destroy/（cases.md、runs 07〜13）

## 対象外の成果物について

`runs/01-baseline-tc01` 〜 `06-skill-tc03` は前セッションの中断により `log.md` が
存在しない破損成果物であり、**採点対象外**とする（`original` / `sandbox` のみが
残存し、手順の記録・アサーション判定の根拠が一切無いため）。本レポートは
`log.md` を持つ `runs/07-baseline-tc01` 〜 `runs/13-tc04` の7件のみを対象とする。

## サマリー

| 指標 | 結果 |
| --- | --- |
| ケース合格率 | 4/7（ベースライン3件は全て不合格、スキルあり3件＋発動判定1件は全て合格） |
| アサーション合格率 | 18/22 |
| 判定不能 | 0件 |

ベースライン（SKILL.md 未読の一般的なアシスタント）は TC-01〜03 のいずれでも
1件以上のアサーションを落としたのに対し、スキルあり（SKILL.md 準拠）は
TC-01〜03・発動判定（TC-04）のすべてで全アサーションに合格した。落ちた4件の
アサーションはいずれもベースライン側で発生しており、SKILL.md の該当手順が
その欠落を具体的に埋めていることを、原本との diff によって独立に確認できた。

## 検証方法

各ケースについて `log.md` の記述に加え、`original/` と `sandbox/` を
`diff -rq` および対象ファイル単位の `diff` で自分で突合し、ログの主張が
バイト単位で裏付けられるかを確認した（詳細は各ケースの根拠に記載）。

## ケース別結果

### TC-01: レジストリに基づく撤去 — ベースライン（run07）: 不合格 / スキルあり（run10）: 合格

**ベースライン（run07-baseline-tc01）**
- ✅ レジストリ記録に基づく撤去対象一覧の提示: log.md 手順4の記述どおり、3分類（新規生成ファイル／マーカー区間／settings.json エントリ）で提示
- ✅ 確認後に撤去を実行: log.md の記述と整合。確認前の変更を示す記録なし（本レポートでは確認プロセス自体はログの自己申告に依拠。実行結果側は独立確認済み）
- ✅ AGENTS.md / CLAUDE.md のマーカー区間のみ除去: `diff runs/07-baseline-tc01/original/AGENTS.md runs/07-baseline-tc01/sandbox/AGENTS.md` で除去行はマーカー区間内（8〜29行目）のみ、`diff .../CLAUDE.md` も同様（8〜38行目のみ）。区間外の差分はゼロ
- ✅ 新規生成ファイルの削除: `runs/07-baseline-tc01/sandbox/.claude/` に `agents/` ディレクトリが存在しない（`ls` で `settings.json` のみ残存を確認）
- ❌ レジストリ削除を独立した最終確認として実行: log.md 自己申告のとおり不達（一括確認に含めた）。成果物としてはレジストリは削除されている（`sandbox/.claude/` に `bitz-env.local.md` が無いことを確認済み）が、アサーションは「最後に単独で確認して実行しているか」という**手順**を問うものであり、手順の欠落自体がログに明記されているため不合格と判定
- ✅ 操作は sandbox 配下に限定: `diff -rq` で変更ファイルはすべて `runs/07-baseline-tc01/sandbox/` 配下（AGENTS.md・CLAUDE.md・.claude/settings.json、および agents/・bitz-env.local.md の消失）のみ
- **結果: 5/6 合格**

**スキルあり（run10-skill-tc01）**
- ✅ 撤去対象一覧の提示: log.md 手順3で3分類別に提示
- ✅ 確認後に実行
- ✅ マーカー区間のみ除去: `diff runs/10-skill-tc01/original/AGENTS.md runs/10-skill-tc01/sandbox/AGENTS.md`・`diff .../CLAUDE.md` ともに除去行はマーカー区間内のみで一致（run07と同一区間・同一内容）
- ✅ 新規生成ファイル削除: `runs/10-skill-tc01/sandbox/.claude/` に `agents/` が存在しないことを確認
- ✅ レジストリ削除を独立確認: log.md 手順4後半の記述どおり、他の撤去と分離した確認ステップを実施した旨が明記されており、run07 で欠落していた手順がここでは明示的に踏まれている
- ✅ sandbox 限定: `diff -rq` で変更ファイルは `runs/10-skill-tc01/sandbox/` 配下のみ
- **結果: 6/6 合格**

settings.json の deny 除去についても両runで独立確認した。`diff .../settings.json` は
両runとも `"deny": [...]`（9件）→ `"deny": []` の同一差分で、レジストリ記載の9件が
過不足なく除去されている。

**スキルあり/なし比較**: 唯一の差は「レジストリ削除を独立した最終確認にするか」。
SKILL.md 手順4の「すべての撤去が完了したら、最後にレジストリ自体の削除を確認して
実行する」という一文が、ベースラインが一般的な効率優先の判断（確認をまとめる）で
落としたポイントを明示的に埋めている。成果物（削除結果そのもの）は両者ともほぼ
同じだが、**手順の質**（確認の粒度）に明確な差がある。

### TC-02: レジストリ欠落時の安全側動作 — ベースライン（run08）: 不合格 / スキルあり（run11）: 合格

**ベースライン（run08-baseline-tc02）**
- ❌ 推測で削除せず候補報告に留める: log.md の自己申告どおり、状況証拠（マーカーコメントの文言）を根拠に削除まで実行しており不達
- ❌ sandbox 内が一切変更・削除されていない: `diff runs/08-baseline-tc02/original/AGENTS.md runs/08-baseline-tc02/sandbox/AGENTS.md` でマーカー区間（8〜29行目、TC-01と同一内容）が除去されていることを確認。また `runs/08-baseline-tc02/original/.claude/agents/` には `advisor.md` が存在するが、`runs/08-baseline-tc02/sandbox/.claude/agents/` はディレクトリごと消失（`ls` がエラーで確認）
- **結果: 0/2 合格**

**スキルあり（run11-skill-tc02）**
- ✅ 推測で削除せず候補報告に留める: log.md の記述どおり、`AGENTS.md` マーカー区間と `advisor.md` を「未記録（要ユーザー判断）」として報告するに留めた旨が明記
- ✅ sandbox 内が一切変更・削除されていない: `diff -rq runs/11-skill-tc02/original runs/11-skill-tc02/sandbox` の差分は `.git`（バージョン管理メタデータ）のみで、実ファイルの差分はゼロ
- **結果: 2/2 合格**

**スキルあり/なし比較**: このケースはスキルの効果が最も明確に観測できたケースの一つ。
ベースラインは「マーカーコメントの文言が説明的だから確信度が高い」という合理的だが
規律違反の判断で実削除に踏み切ったのに対し、スキルは SKILL.md 手順1の
「レジストリが存在しない場合は推測で削除せず候補報告に留める」という原則を機械的に
遵守し、確信度の高低に関わらず操作を止めた。これは ENVFR010-S4（安全側動作）が
明文化された規律として機能した典型例。

### TC-03: 部分承認 — ベースライン（run09）: 不合格 / スキルあり（run12）: 合格

**ベースライン（run09-baseline-tc03)**
- ✅ 承認された項目のみ撤去し AGENTS.md は残す: `diff runs/09-baseline-tc03/original/AGENTS.md runs/09-baseline-tc03/sandbox/AGENTS.md` の差分はゼロ（完全一致）
- ❌ 部分撤去のためレジストリを削除せず撤去済み項目だけ記録から消す: `ls runs/09-baseline-tc03/original/.claude/` には `bitz-env.local.md` が存在するが、`sandbox/.claude/` には存在しない（レジストリファイル自体が消失）。log.md の自己申告と一致
- **結果: 1/2 合格**

**スキルあり（run12-skill-tc03）**
- ✅ 承認された項目のみ撤去し AGENTS.md は残す: `diff runs/12-skill-tc03/original/AGENTS.md runs/12-skill-tc03/sandbox/AGENTS.md` の差分はゼロ（完全一致）
- ✅ 部分撤去のためレジストリを残し撤去済み項目だけ記録から消す: `runs/12-skill-tc03/sandbox/.claude/bitz-env.local.md` の内容を確認したところ、`.claude/agents/advisor.md` / `worker.md`（取り消し線＋「撤去済み」注記）・`CLAUDE.md` のマーカー区間（同）は撤去済みとして記録され、`AGENTS.md` のマーカー区間のみ「ユーザーの指示によりこの区間のみ残置」と明記されて有効エントリとして残っている。settings.json の deny も9件除去済みと記録あり、`diff runs/12-skill-tc03/original/.claude/settings.json runs/12-skill-tc03/sandbox/.claude/settings.json` の実差分（deny 9件→空）と一致
- **結果: 2/2 合格**

**スキルあり/なし比較**: ベースラインは TC-01（全撤去）の成功パターンをそのまま
部分撤去に流用し、レジストリを「全部消すか全部残すか」の二値でしか扱えず、
部分撤去にもかかわらずレジストリごと削除してしまった。スキルは SKILL.md 手順4の
「部分撤去の場合は残し、撤去済み項目を記録から消す」という一文により、
部分撤去を独立したケースとして正しく扱えている。

なお、テスト実行者が run12 の気づきとして記録したとおり、**レジストリの更新表現
（取り消し線＋残置注記）は SKILL.md に書式の規定が無く、実行者が可読性重視で
独自に構成したもの**である。今回は結果的にアサーションを満たしたが、書式が
明文化されていないため実行間でばらつく可能性がある。これは改善提案として
SI-ENV-020 に起票した（後述）。

### TC-04: 発動判定 — run13: 合格

- ✅ (a)「プラグインをアンインストールする前に後始末しておいて」→「使う」と判定: log.md の判定根拠（description のトリガー例「アンインストールの後始末」との表層一致）を確認。description 本文（`plugins/bitz-env/skills/env-destroy/SKILL.md` frontmatter）を独自に読み、該当フレーズが実在することを確認した
- ✅ (b)「ビルド成果物を消して」→「使わない」と判定: description のどのトリガーフレーズにも合致せず、判定は妥当
- **結果: 2/2 合格**

判定はいずれも description の記述と整合しており妥当。ただし log.md の気づきに
あるとおり、「プラグイン」という語のみでは bitz-env 固有かどうか曖昧になり得る
という懸念がテスト実行者自身から示されている。今回のテストでは唯一の候補が
bitz-env 生成物だったため誤発動は観測されていないが、複数プラグイン共存環境での
将来リスクとして SI-ENV-021 に起票した（後述）。

## 改善提案（優先度順）

1. **[中]** レジストリの部分撤去時における撤去済みエントリの記録書式を
   SKILL.md 手順4に明記する: run12（TC-03 スキルあり）では部分撤去後のレジストリ
   更新（取り消し線＋残置注記）をテスト実行者が独自に構成しており、SKILL.md に
   規定が無い。アサーション自体は満たしたが、実行者・実行間で書式がばらつく余地が
   あり、将来 env-doctor 等がレジストリを機械的に解析して残存物を追跡する用途を
   想定すると解析性を損なう。「部分撤去が発生した場合のレジストリエントリ状態
   表現」全般に効く一般化された修正として、SI-ENV-020 に起票した
   （`plugins/bitz-env/.spec/spec-issues/SI-ENV-020.md`）。
2. **[低〜中]** description の対象プラグイン限定を強化する: run13（TC-04）の
   気づきどおり、「プラグイン」という語のみに依拠したトリガー一致は、複数
   プラグインが共存する環境では対象の取り違えリスクを孕む。今回は誤発動は
   未観測（予防的措置）であり、過学習リスクに配慮し「bitz-env プラグインの」等
   対象限定語の強化」という一般化された文言改善として SI-ENV-021 に起票した
   （`plugins/bitz-env/.spec/spec-issues/SI-ENV-021.md`）。

## テストケース自体への提案

- **settings.json のテンプレート由来／ユーザー独自追加が混在するケースが未検証**:
  SKILL.md 手順2は「区別がつかないエントリは削除せずユーザー判断に委ねる」と
  規定しているが、全runで sandbox の deny 配列はテンプレートと完全一致しており
  この分岐は一度も実行されていない（run10 の気づきでも同旨の指摘あり）。
  この規律が実際に機能するかは未検証のため、settings.json にユーザー独自の
  deny/ask エントリが混在する sandbox 構成を追加したテストケースの新設を提案する。
- **非git管理下での `.bak` 作成が未検証**: SKILL.md 手順4は「展開先が git 管理外の
  場合は `.bak` を作成してから実行する」と規定しているが、全run（07〜13）の
  sandbox はいずれも git 管理下であり、この分岐は一度も実行されていない。
  git 管理外の sandbox を用いたテストケースの追加を提案する。
- 上記2点はいずれも SKILL.md 側は既に規律を記述済みであり、**実装済みだが未検証**
  という位置づけのため spec-issue ではなく `skill-tester` によるケース追加を提案する。
