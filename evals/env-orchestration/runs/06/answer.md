# TC-01 スキルあり回答（env-orchestration の指示に従う）

**入力プロンプト**: 30ファイルの API 移行があるんだけど、どこにどう委譲すべき？

## 前提の読み込み
- `.claude/bitz-env.local.md` を読んだ: adapters に `ex-delegate`（roles: [delegate]）が登録済み、
  routes.delegate: `ex-delegate`
- `.claude/agents/worker.md` あり（worker 役割）

## パターン選択の決定木の適用
- 30ファイルの API 移行 = 「量産・長文読解など外部エージェントの得意領域」に該当
  （手順が明確な定型作業でもあるが、規模的に外部アダプタの delegate 役割が適合）
- 決定木: 【委譲型】レジストリの能力宣言に合致するアダプタの delegate 役割へ
  （実スキル名はレジストリのルーティングテーブルで解決）

## 委譲先の解決（レジストリ経由）
1. 必要な役割: `delegate`
2. レジストリのルーティングテーブルを引く → `routes.delegate: ex-delegate`
3. エントリあり → フォールバック不要
4. **解決した実スキル名 `ex-delegate` で呼び出す**（役割名「delegate」や固定スキル名を
   決め打ちしていない）

## 委譲計画
- 呼び出し先: `ex-delegate`（レジストリで解決した実スキル名）
- 依頼内容: 30ファイルの API 移行パターンを1つのプロンプトに整理し、対象ファイル一覧・
  期待する DIGEST 形式（変更ファイル一覧・実行した検証の実出力・未解決事項）を明示して委譲
- 検収: 委譲完了後、DIGEST だけでなく中心（自分）が `git diff` 等で客観的に確認する

## 判断記録
- レジストリのルーティングテーブルを読んで委譲先を解決した（ex-delegate）
- 呼び出し名は役割名「delegate」ではなくレジストリが解決した実スキル名 `ex-delegate` を使用
- 選択理由: 30ファイルという量産系の規模であり、決定木の「量産・長文読解など外部エージェントの
  得意領域」に合致するため委譲型（delegate 役割経由）を選んだ
