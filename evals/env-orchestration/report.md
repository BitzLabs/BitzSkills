## 改善提案（優先度順）

1. **[高]** 決定木の「定型作業（worker 行き）」枝と「量産・長文読解など外部エージェントの
   得意領域（アダプタ delegate 行き）」枝が重複しうるケース（例: TC-01 の「30ファイルの
   API 移行」は手順は定型だが規模が量産系）について、優先順位・切り分け基準
   （例: アダプタの能力宣言にある break-even 目安との照合、対象ファイル数・作業範囲の目安）
   を決定木に追記する。runs/06/answer.md でも実行者自身が両枝への該当を留保しつつ選択して
   おり、モデル・実行者が変われば判断が割れうる。特定ケースの数値をそのまま条件に埋め込む
   のではなく、「両枝に該当する場合の判定基準（アダプタの能力宣言参照）」という一般化した
   形で追記することを推奨。→ **SI-ENV-014.md** に起票（詳細は spec-issue 本文参照）
2. **[中]** 「合議型の進め方」節にある「必要なら worker」という文言について、worker を
   合議メンバーに含めるべき条件が SKILL.md 本文に定義されていない。TC-03（runs/08）では
   worker を除外したが、その理由づけは決定木・合議節の記述からは導出できず実行者が独自に
   補ったものだった。合議に worker を含める／除外する一般的な判断基準
   （例: 争点に実装可否・工数見積もりなど実行観点が含まれるか）を明記する。
   → **SI-ENV-015.md** に起票
3. **[中]** 「DIGEST のみに依存しない客観的検収」（共通の運用ルール）は git diff / git status
   の取得を前提にしているが、委譲先の作業対象が git 管理外の場合や、DIGEST が実行ログ
   （検証の実出力）を含まない契約違反の場合に中心が取るべき具体的対応が定義されていない。
   代替の検収手段と、契約違反 DIGEST への定型対応（受理せず実出力の提示を要求する等）を
   共通運用ルールに追記する。→ **SI-ENV-016.md** に起票

## テストケース自体への提案

- TC-04 のテスト実行（runs/04, runs/09）は log.md 上「git init → 初期コミット → 未コミット
  変更を残す」と記録されているが、成果物として保存された sandbox には `.git` が含まれておらず、
  transcript 中の `git status`/`git diff` の出力を第三者が独立に再現・検証することができない。
  今後同種のシナリオ（「実行環境の状態を自ら取得したか」を検証するテスト）では、sandbox に
  実際の `.git` を含めて保存する（または実行時のコマンド出力ログを別ファイルとして保存する）
  ことで、証拠主義に基づく採点をより厳密に行えるようにすることを推奨する。
- TC-03 は「格下げの明記」をアサーションにしているが、「worker をメンバーに含めるか否かの
  判断過程を成果物に明記しているか」は現状アサーション化されていない。改善提案2
  （SI-ENV-015）の要件化後、対応するアサーションをケースに追加することを検討されたい
  （テスト実行者の気づき「合議メンバーに worker を含めるか否かの基準が SKILL.md に無い」に
  対応）。
