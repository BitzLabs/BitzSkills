# ペルソナと JTBD — plugin-creator

> 遡及的ディスカバリー。ユーザーインタビュー等の一次調査は未実施のため、感情・生の声は原則 **[proto / 未検証]** と明示する。人口統計・台詞は捏造しない。

## Jobs-to-be-Done（ジョブストーリー）

- **JTBD-1**: 新しいプラグインを作り始める**とき**、Claude Code と Antigravity 2.0 の両方で正しく発見・動作する雛形を**素早く立ち上げたい**、機能実装に集中**できるように**
- **JTBD-2**: コマンド / エージェント / フック / MCP / 設定のどれかを追加する**とき**、そのコンポーネント固有の正しい配置と frontmatter を**間違えずに書きたい**、「発見されず動かない」を避け**られるように**
- **JTBD-3**: 出荷前にプラグインを検証する**とき**、構造とマニフェストの不整合を**機械的に洗い出したい**、レビューの手戻りを減らせ**るように**
- **JTBD-4**: 既存プラグインに機能追加する**とき**、両プラットフォームの仕様差を**その場で確認したい**、公式ドキュメントを行き来する消耗を避け**られるように**

ジョブの3次元:
- **機能的**: 正しく動く両対応プラグインを作る
- **感情的**: 「配置ミスで動かない」不安から解放され、自信を持って出荷したい `[proto / 未検証]`
- **社会的**: OSS として「きちんと両対応された、作法の正しいプラグイン」を公開している開発者と見られたい `[proto / 未検証]`

## ペルソナカード

### ペルソナ A（主要）: 内製メンテナ「hide」

- **アーキタイプ / 1行**: BitzSkills モノレポを1人で保守する個人開発者。プラグインを日常的に増改築する当事者
- **文脈と行動**: Claude Code をオーケストレーター運用し、両プラットフォーム両対応を前提に開発。AGENTS.md 規約（自己完結・責務分離・一次情報）を強く内面化している
- **Jobs**: JTBD-1〜4 のすべて。特に JTBD-2/4（コンポーネント別の正確さ）と JTBD-3（検証）
- **Pains**: 両プラットフォームの仕様差の把握コスト、2マニフェスト同値維持の手間、配置ミスによる不発見、Antigravity 仕様の実測追随
- **Gains**: 調べ直し不要の雛形、コンポーネント別の的確なガイド、出荷前検証、skill-creator との明快な責務分離
- **生の声**: 「毎回ドキュメントを往復せず、作りたい機能に集中したい」`[proto / 未検証]`

### ペルソナ B（二次）: OSS 採用外部開発者「新規プラグイン作者」

- **アーキタイプ / 1行**: BitzSkills を導入し、自分のプラグインを作りたい Claude Code / Antigravity ユーザー
- **文脈と行動**: プラグイン機構の学習曲線を登り始めた段階。両対応かどうかは自分の利用環境次第で、片側だけ使うこともある
- **Jobs**: JTBD-1（立ち上げ）と JTBD-2（コンポーネント追加）が中心
- **Pains**: 「plugin.json が2つ要る」「コマンドは Antigravity ではスキル扱い」等の落とし穴を知らずに踏む、どのガイドを読めばいいか分からない
- **Gains**: 必要な時だけ発動する的確なガイド、create-plugin の対話ワークフローによる伴走、検証エージェントの安心感
- **生の声**: `[proto / 未検証]`（外部利用者の実声は未収集）

### ペルソナ C（served / anti）

- **served**: skill-creator を使うスキル作者 — plugin-creator は「スキルの同梱の作法」だけを提供し、実作業は skill-creator へ誘導する（重複させない）
- **anti-persona**: GUI ノーコードだけを求める非エンジニア、プラグイン機構を使わず単発スクリプトで済ませたい層。最適化しない

## 主要ペルソナ

**ペルソナ A（内製メンテナ hide）を主要ペルソナとする。** 遡及ディスカバリーの時点で唯一の実利用者かつ意思決定者であり、下流はまず A に最適化する。ペルソナ B は将来の拡大対象として設計に織り込むが、実声が取れるまでジョブ仮説は proto 扱い。

## ジャーニー（主要ペルソナ A・優先機会リスト）

段階を製品に合わせて調整: **着想 → コンポーネント設計 → 実装 → 検証 → 出荷**。

| 段階 | 行動 | 思考・感情 `[proto]` | ペイン | 機会（優先） |
|---|---|---|---|---|
| 着想 | 作るプラグイン種別を決める | 「両対応で作れるか？」（中） | 何から手を付けるか不明瞭 | create-plugin の発見フェーズで種別特定 |
| コンポーネント設計 | 必要な component を選ぶ | 「どの配置が正解か」（低・不安） | 仕様差・落とし穴 | コンポーネント別スキルの的確な発動（**最高レバレッジ**） |
| 実装 | 雛形・frontmatter を書く | 「発見されず動かないのが怖い」（低） | 2マニフェスト同値・${CLAUDE_PLUGIN_ROOT} | 雛形と両対応注記 |
| 検証 | 出荷前チェック | 「見落としがないか」（中） | 手動レビューの手戻り | plugin-validator + release_check の機械検証 |
| 出荷 | install / 配布 | 「作法通りに出せた」（高） | 配信手順の分散 | platform-paths 誘導（配信自体は Out-of-Scope） |

**決定的瞬間（MoT）**: 「コンポーネント設計」段階で正しいガイドが発動するか否かが、感情曲線が最も沈みやすく、かつ最もレバレッジの高い改善点。

> 全 Pain / 機会はペルソナ A のジョブに紐づけた。感情曲線はすべて proto であり、実声が届き次第 research-based へ昇格する。
