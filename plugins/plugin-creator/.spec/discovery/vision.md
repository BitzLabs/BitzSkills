# ビジョン — plugin-creator

> 遡及的ディスカバリー。plugin-creator は既にリリース済み（v1.2.1、7スキル + create-plugin コマンド + agent-creator/plugin-validator エージェント）であり、本ファイルは実装から逆算してビジョンを言語化したもの。根拠のない数値・ユーザー調査結果は `TBD` または `[proto / 未検証]` と明示する。

## Product Vision Board（Roman Pichler）— 5要素

### 1. Vision（究極の目的）

**Claude Code と Antigravity 2.0 の「両対応プラグイン」を、仕様差に迷わず一発で正しく作れる状態を当たり前にする。**

プラットフォーム間の仕様差（ディレクトリ規約・マニフェスト・コマンドとスキルの扱い・フック機構・MCP 設定形式）を開発者が毎回自力で調べ直す消耗をなくし、「作りたい機能」に集中できるようにする。

### 2. Target Group（ターゲット — セグメントする）

- **主要**: このモノレポ（BitzSkills）のメンテナ本人（hide、個人開発者）。BitzSkills 配下のプラグインを日常的に増改築する当事者
- **二次**: BitzSkills を OSS として採用し、自分のプラグインを作りたい将来の外部開発者（Claude Code / Antigravity 2.0 の両方またはいずれかを使う）
- **反ターゲット（anti）**: プラグイン機構を使わず単発スクリプトで済ませたい層、GUI ノーコードだけを求める非エンジニア層

### 3. Needs（解決する問題 / 便益）

- 両プラットフォームの**仕様差を横断的に把握するコストが高い** — 公式ドキュメントが分散し、Antigravity 側は実測が要る（`docs/調査報告/01.Antigravity/` が正とされる背景）
- 「plugin.json は2つ要る」「コマンドは Antigravity ではスキルに変換される」など**間違えやすい落とし穴**が多く、誤ると発見（自動発見）されず動かない
- コンポーネント（構造 / コマンド / エージェント / フック / MCP / 設定 / スキル同梱）ごとに**設計判断とベストプラクティスが必要**で、初回の学習曲線が急
- ゼロから雛形を書く**定型作業の反復**

### 4. Product（際立つ少数の差別化要素）

- **両プラットフォーム仕様差の一次情報化**: Claude Code と Antigravity 2.0 の差分を明示的に織り込んだガイド群（他の汎用プラグイン解説にはない切り口）
- **コンポーネント別の7スキル**による progressive disclosure（必要なガイドだけが発動する）
- **create-plugin コマンド**による発見 → 設計 → 実装 → 検証の一気通貫ワークフロー
- **plugin-validator エージェント**による構造・マニフェストの機械検証
- skill 本体の作成・検証・配置は skill-creator プラグインへ**明確に委譲**し、責務を混ぜない設計規律

### 5. Business Goals（提供側の便益）

- BitzSkills モノレポの**内製プラグイン開発速度と品質の底上げ**（ドッグフーディング。plugin-creator 自身が BitzSkills の他プラグインを生む土台）
- OSS 公開時の**採用障壁を下げる**（両対応の作り方が同梱されていること自体が差別化）
- 直接的な収益目標は設定しない（個人開発 OSS のため）。`[proto / 未検証]`

## Mission / Vision / Values

- **Vision（目的地）**: 両対応プラグイン開発が「調べ直し不要」で当たり前になった世界
- **Mission（日々の追求）**: Claude Code / Antigravity 2.0 の仕様差を一次情報として保守し、コンポーネント別ガイドと検証で「正しい雛形」を最短で届ける
- **Values（判断原則）**:
  1. **正確さ > 網羅**: 実測・一次情報に基づく（Antigravity は `docs/調査報告/` を正とし Gemini 生成解説は参照しない）
  2. **責務分離**: スキル本体の作成・検証・配置は skill-creator に委ね、plugin-creator は「プラグインとしての組み立て」に徹する
  3. **両対応を前提に置く**: 片方だけで完結させず、常に仕様差を明示する

## PR-FAQ 圧力試験

### プレスリリース（出荷済みの体で）

**見出し**: plugin-creator — Claude Code / Antigravity 2.0 の両対応プラグインを、仕様差に迷わず作る

**小見出し**: プラグイン開発者のための、コンポーネント別7ガイド + 作成ワークフローコマンド

**要約**: plugin-creator は、Claude Code と Antigravity 2.0 の両方で動くプラグインを設計・実装・検証するための開発ツール群である。構造・コマンド・エージェント・フック・MCP・設定・スキル同梱の各領域をカバーする7スキルと、発見から検証までを導く create-plugin コマンドを提供する。

**問題（顧客の言葉で）**: 「プラグインを作りたいが、Claude Code と Antigravity で微妙にディレクトリもマニフェストも違う。毎回ドキュメントを行き来して、結局動かない配置ミスで時間を溶かす」

**解決 / 差別化**: コンポーネントごとに独立したガイドが必要な時だけ発動し、両プラットフォームの差分を明示する。plugin-validator が配置・マニフェストを機械検証するため「発見されず動かない」を出荷前に潰せる。

**引用と CTA**:
- 責任者: 「調べ直しの消耗をなくし、作りたい機能に集中させたかった」（hide）
- 顧客の声: `[proto / 未検証]`
- 始め方: `/plugin install plugin-creator@bitzskills` → `/create-plugin`

### 内部 FAQ（荷重を受ける部材）

- **市場規模（TAM/SAM/SOM）**: `TBD`（個人開発 OSS。定量市場調査は未実施）
- **競合状況**: 公式ドキュメント、汎用的なプラグイン解説記事、手書き。両対応差分を体系化した競合は `TBD`（positioning.md で扱う）
- **ユニットエコノミクス**: 非該当（無償 OSS）。`[proto / 未検証]`
- **運用課題**: Antigravity 2.0 の仕様変更追随コスト（一次情報を実測で保守する必要）
- **主要リスク**: (a) 両プラットフォームの仕様が乖離・変化し続けガイドが陳腐化する、(b) 利用者が主要メンテナ以外にほぼ増えない、(c) skill-creator との責務境界が利用者に伝わらず混乱する
- **Go/No-Go 基準（反証可能）**: assumptions.md（PLG-DSC-006）で裁定する。遡及ディスカバリーのため既に「出荷継続（Go）」が実績として存在するが、崩壊クリティカル仮説はエビデンスとともに明示する
