# 対話的コマンドのパターン

AskUserQuestion ツールを使って、ユーザーの回答を集めながら進むコマンドの
作り方。

## 概要

単純な引数では扱いにくいユーザー入力がある:

- トレードオフの説明が必要な複数の選択肢からの選択
- リストからの複数選択
- 判断に説明が必要な決定
- 設定・好みの対話的な収集

こうした場合は、コマンド引数ではなく **AskUserQuestion ツール**を
コマンド実行中に使う。

## 使い分け

**AskUserQuestion を使う場面:**

1. 説明が必要な多肢選択の決定
2. 選ぶのに文脈が必要な複雑な選択肢
3. 複数選択のシナリオ
4. 設定・好みの収集
5. 回答に応じて分岐する対話的ワークフロー

**コマンド引数を使う場面:**

1. 単純な値（ファイルパス・数値・名前）
2. ユーザーが既に持っている既知の入力
3. 自動化可能であるべきワークフロー
4. 質問すると遅くなる高頻度の呼び出し

## AskUserQuestion の基本

```typescript
{
  questions: [
    {
      question: "どの認証方式を使いますか？",
      header: "認証方式",       // 短いラベル（最大12文字）
      multiSelect: false,       // true で複数選択可
      options: [
        { label: "OAuth 2.0", description: "業界標準。複数プロバイダ対応" },
        { label: "JWT", description: "ステートレス。API向き" },
        { label: "セッション", description: "従来型。サーバー側で状態管理" }
      ]
    }
  ]
}
```

**要点:**

- ユーザーは常に「その他」で自由入力できる（自動で付く）
- `multiSelect: true` で複数選択を許可
- 選択肢は1問あたり2〜4個
- 1回の呼び出しで1〜4問

## 基本パターン

### 対話的セットアップコマンド

```markdown
---
description: 対話的なプラグインセットアップ
allowed-tools: AskUserQuestion, Write
---

# 対話的セットアップ

## Step 1: 設定の収集

AskUserQuestion で以下を質問する:

**質問1 - デプロイ先:**
- header: 「デプロイ先」
- question: 「どのプラットフォームにデプロイしますか？」
- options: AWS / GCP / Azure / ローカル(Docker)

**質問2 - 環境構成:**
- header: 「環境」
- options: 単一（本番のみ）/ 標準（Dev・Staging・本番）/ 完全（Dev・QA・Staging・本番）

**質問3 - 有効化する機能:**
- header: 「機能」
- multiSelect: true
- options: オートスケール / 監視 / CI/CD / バックアップ

## Step 2: 回答の処理

回答を解析し、環境別設定を組み立て、選択された機能を有効化する。

## Step 3: 設定ファイルの生成

回答を基に `.claude/plugin-name.local.md` を生成する。

## Step 4: 確認と次のステップ

生成した設定を確認し、次の手順を案内する。
```

### 条件分岐フロー

```markdown
## 質問1: デプロイの複雑さ

question: 「デプロイはどの程度複雑ですか？」
options: シンプル（単一サーバー）/ 標準（複数サーバー・LB）/ 複雑（マイクロサービス）

## 回答に応じた追加質問

「シンプル」なら: 追加質問なし。最小設定を使う。
「標準」なら: ロードバランシング戦略・スケーリングポリシーを質問する。
「複雑」なら: オーケストレーション基盤（Kubernetes等）・サービスメッシュ・
監視・ログ集約について質問する。
```

### 反復収集

```markdown
## 質問: チーム人数

question: 「何人分をセットアップしますか？」
options: 2人 / 3人 / 4人 / 6人

## メンバーごとの反復

各メンバー（1〜N）について AskUserQuestion で役割を質問し、
全員分を集めてから設定ファイルを生成する。
```

### 検証ループ

```markdown
## 設定の収集 → 検証

設定を集めたら妥当性を確認する（依存の有無・設定同士の互換性・競合）。

検証に失敗したら AskUserQuestion で確認する:

question: 「設定に問題があります。どうしますか？」
options:
  - 修正（問題を解決するよう設定を調整）
  - 強行（警告を無視して続行）
  - 中止（セットアップを中断）
```

### コンテキスト対応の質問

```markdown
## 現在の状態を検出

- 言語: !`detect-language.sh`
- フレームワーク: !`detect-frameworks.sh`

## 検出結果に応じた質問

TypeScript なら: Strict Mode / デコレータ / パスマッピング を質問
Python なら: 型ヒント(mypy) / Black / Pylint を質問
```

## 質問設計のベストプラクティス

### 質問文

1. **具体的に**: 「どのデータベースを使いますか？」（「選んでください」は不可）
2. **トレードオフを説明**: 選択肢の description に利点・欠点を書く
3. **単体で意味が通る**: 質問文だけで文脈がわかるように
4. **簡潔に**: header は最大12文字、description は1〜2文

### 選択肢

1. **意味のあるラベル**: 1〜5語の具体的な名前
2. **説明的な description**: その選択が何を意味するか
3. **均等な詳しさ**: 全選択肢を同じ粒度で説明する
4. **2〜4個**: 多すぎる選択肢で圧倒しない

```markdown
✅ 良い例:
question: 「このプロジェクトのデータベースはどれにしますか？」
options:
  - PostgreSQL（リレーショナル・ACID準拠・複雑なクエリ向き）
  - MongoDB（ドキュメント指向・柔軟なスキーマ・高速な反復向き）
  - Redis（インメモリ・高速・キャッシュとセッション向き）

❌ 悪い例:
question: 「DB？」
options: 「選択肢1」「選択肢2」
```

### multiSelect の使い分け

```markdown
✅ 適切: 「どの機能を有効にしますか？」（ログ / メトリクス / アラート /
   バックアップ）— どの組み合わせもあり得る

❌ 不適切: 「どのデータベースエンジン？」— 1つしか使えないものは単一選択
```

### フロー設計

1. **自然な順序**: 質問が論理的に流れる
2. **前の回答を活かす**: 後の質問は前の回答に基づいて絞る
3. **質問は最小限**: 必要なものだけ聞く
4. **段階的開示**: 最初に「クイック / カスタム / ガイド付き」を聞き、
   選択に応じて質問量を変える
5. **実行前に確認**: 最後にサマリーを見せて確認を取る

## 引数と質問の併用

既知の値は引数で、説明が必要な選択は質問で受け取る:

```markdown
---
argument-hint: [project-name]
allowed-tools: AskUserQuestion, Write
---

プロジェクト名（引数から）: $1

続いて AskUserQuestion で以下を質問する:
- アーキテクチャパターン
- 技術スタック
- デプロイ戦略

これらは説明が必要なので、引数より質問が適している。
```

## 実例: マルチエージェント起動コマンド

```markdown
---
description: マルチエージェントスウォームを起動する
allowed-tools: AskUserQuestion, Read, Write, Bash
---

# スウォーム起動

タスクリストファイルが渡されなければ、対話的に作成を支援する。

### 質問1: エージェント数
options: 2体（小規模）/ 3体（中規模）/ 4体（標準）/ 6体以上（大規模）

### 質問2: タスク定義の方法
options: ファイル（用意済み）/ ガイド付き（対話的に作成）/ その他

### 質問3: 協調モード
options: リーダー方式 / 対等協調 / 自律（最小限の協調）

### ガイド付き作成（質問2で選択時）

各エージェント（1〜N）について:
- 名前（auth-agent / api-agent / ui-agent 等の候補を提示）
- タスク種別（認証 / API / UI / DB / テスト / ドキュメント）
- 依存（定義済みエージェントの一覧から multiSelect）
- PRのベースブランチ（main / staging / develop）

全員分を集めたらタスクリストファイルを生成し、パスを提示して起動に進む。
```

## トラブルシューティング

- **質問が表示されない**: `allowed-tools` に AskUserQuestion があるか、
  選択肢が2〜4個かを確認する
- **選びにくい**: ラベルの明確さ・description の有用性・選択肢の数・
  multiSelect 設定を見直す
- **フローが混乱する**: 質問数を減らす、関連する質問をまとめる、
  ステージ間に説明を挟む、進捗を示す
