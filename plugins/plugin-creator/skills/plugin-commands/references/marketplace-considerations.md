# マーケットプレイス配布時の考慮事項

配布用コマンドを設計するためのガイドライン。

## 概要

マーケットプレイスで配布するコマンドは、個人用と違って未知の環境・未知の
ユーザーで動く必要がある。環境の多様性・多様なユースケース・初見ユーザーの
体験を考慮する。

## 配布向けの設計

### クロスプラットフォーム対応

プラットフォームを検出して挙動を変える:

```bash
case "$(uname)" in
  Darwin*)  PLATFORM="macOS" ;;
  Linux*)   PLATFORM="Linux" ;;
  MINGW*|MSYS*|CYGWIN*) PLATFORM="Windows" ;;
  *)        PLATFORM="Unknown" ;;
esac
```

プラットフォーム固有コマンドの決め打ちは避ける:

```bash
# ❌ 悪い: macOS専用
pbcopy < file.txt

# ✅ 良い: 検出して使い分け
if command -v pbcopy > /dev/null; then
  pbcopy < file.txt
elif command -v xclip > /dev/null; then
  xclip -selection clipboard < file.txt
elif command -v clip.exe > /dev/null; then
  cat file.txt | clip.exe
else
  echo "このプラットフォームではクリップボードが使えません"
fi
```

### 依存を最小限に

必要なツールを先にチェックし、なければ導入先を案内する:

```bash
MISSING_DEPS=""
for tool in git jq node; do
  command -v $tool > /dev/null || MISSING_DEPS="$MISSING_DEPS $tool"
done

if [ -n "$MISSING_DEPS" ]; then
  # ❌ 不足している依存:$MISSING_DEPS
  # 各ツールのインストールURLを提示して終了する
fi
```

任意の依存（gh / docker 等）はコメントで文書化し、
「入っていれば機能が増える」設計にする。

### 段階的な機能低下（graceful degradation）

```markdown
利用可能な機能を検出する:
- gh があれば GitHub 統合をフル機能で提供
- なければ「⚠ 機能制限: GitHub CLI 未導入。gh を入れるとフル機能」と伝え、
  使える範囲の機能で続行する
```

## 初見ユーザーの体験

### 初回実行時のオンボーディング

初期化マーカー（`.claude/command-initialized`）がなければ、
コマンドの目的・クイックスタート・必要なセットアップを説明してから
本来の処理へ進む。

### ユーザーの間違いへの寛容さ

```markdown
<!-- よくある打ち間違いに対応 -->
$1 が「hlep」「hepl」なら「help のことですか？」とヘルプを表示する。

<!-- 未知のオプションには近い候補を提示 -->
❌ 不明なオプション: $1
もしかして: valid-option1（最も近い）/ valid-option2
全オプション: /command help
```

### 診断情報の提供

失敗時にはプラットフォーム・作業ディレクトリ・実行コマンド・
よくある問題のチェック結果（gitリポジトリか・書き込み権限・必要ファイル）を
まとめて表示し、サポートへの問い合わせに使えるようにする。

## 配布のベストプラクティス

### 名前空間への配慮

- 汎用的すぎる名前（`/deploy`, `/test`）は衝突しやすいので避ける
- プラグイン名プレフィックス・カテゴリ・動詞-名詞形式を検討する
- 命名の理由をコメントで文書化する（検討した代替案と選定理由）

### 設定可能性

ユーザー設定ファイル（`.claude/plugin-name.local.md`）を確認し、
なければ賢明なデフォルトを使う:

```markdown
デフォルト設定:
- Format: ${FORMAT:-json}
- Output: ${OUTPUT:-stdout}
- Verbose: ${VERBOSE:-false}

このデフォルトで8割のユースケースをカバーする。
引数または .claude/plugin-name.local.md で上書き可能なことを案内する。
```

### バージョン互換性

- コマンドにバージョンをコメントで記載し、必要なプラグインバージョンを
  チェックする（不足なら `/plugin update plugin-name` を案内）
- 非推奨オプションには警告を出し、削除予定時期と移行先を示す:

```markdown
⚠️ 非推奨警告
--old-flag は v2.0.0 で非推奨。v3.0.0 で削除予定。
代わりに --new-flag を使用。当面は旧動作で続行する。
```

## マーケットプレイスでの見せ方

### 発見されやすく

- **説明的な命名と description**: 「Do the thing」ではなく
  「セキュリティと品質チェック付きでPRをレビューする」
- **検索キーワード**: コメントに関連キーワード（security, code-review 等）を
  記載する

### 魅力的なデモ

コマンド冒頭に「試し方」とサンプル出力を載せ、何が得られるかを
すぐ理解できるようにする。

## 品質基準

### プロフェッショナルな仕上げ

- 絵文字・記号の一貫した使用
- 出力の桁揃え・数値の桁区切り
- 進捗表示・残り時間の目安
- 操作成功の明示的な確認

### 信頼性

**冪等性**: 完了フラグを確認し、既に完了していれば再実行しない
（再実行の手順を案内する）。

**アトミック操作**: 一時ディレクトリで変更を作り、検証に成功したら
まとめて適用、失敗したら破棄して「変更は未適用。再試行可能」と伝える。

## 配布前チェックリスト

**機能:**
- [ ] macOS / Linux / Windows(WSL) で動作
- [ ] 全引数をテスト済み
- [ ] エラーケース・エッジケースを処理

**UX:**
- [ ] 明確な description
- [ ] 役立つエラーメッセージ
- [ ] 例の提供・初回体験の設計・ドキュメント完備

**配布:**
- [ ] ハードコードされたパスなし
- [ ] 依存の文書化・設定オプションの明示
- [ ] バージョン設定・CHANGELOG更新

**品質:**
- [ ] TODOコメント・デバッグコードなし
- [ ] 性能・セキュリティ・プライバシーのレビュー済み

**サポート:**
- [ ] README・トラブルシューティング・連絡先・ライセンス

## ベータリリース

実験的なコマンドはベータであることを明示する:
バージョン（0.x）・安定性・既知の制限・フィードバック窓口を
コマンド冒頭に記載する。

## 保守と更新

**バージョン戦略**: major（破壊的変更・移行ガイド必須）/
minor（後方互換の機能追加）/ patch（バグ修正）。

**更新通知**: 新バージョンがあれば「📢 更新あり」と変更点・
更新コマンド・リリースノートを案内する。

## まとめ

1. **ユニバーサル**: プラットフォーム・環境を問わず動く
2. **自己完結**: 依存最小・要件明示
3. **段階的低下**: 機能が使えなくても丁寧に縮退する
4. **寛容**: ユーザーの間違いを予期して処理する
5. **ユーザー中心**: 良いUX・フィードバックへの対応
