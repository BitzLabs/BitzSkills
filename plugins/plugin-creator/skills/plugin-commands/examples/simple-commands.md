# 基本的なコマンド例集

よくあるユースケースの基本的なスラッシュコマンドパターン。

**重要:** 以下の例はすべて「Claudeへの指示」として書かれている
（ユーザーへのメッセージではない）。コマンドは Claude に何をするかを
指示するものであり、ユーザーに何が起きるかを説明するものではない。

## 例1: コードレビュー

**ファイル:** `.claude/commands/review.md`

```markdown
---
description: コードの品質と問題をレビューする
allowed-tools: Read, Bash(git:*)
---

このリポジトリのコードを以下の観点でレビューする:

1. **コード品質:** 可読性・保守性、スタイルの一貫性、適切な抽象度
2. **潜在的な問題:** ロジックエラー、未処理のエッジケース、性能の懸念
3. **ベストプラクティス:** デザインパターンの正しい使用、エラー処理、
   ドキュメントの十分さ

ファイルと行番号を添えて具体的なフィードバックを提示する。
```

## 例2: セキュリティレビュー

```markdown
---
description: コードのセキュリティ脆弱性をレビューする
allowed-tools: Read, Grep
model: sonnet
---

以下を対象に包括的なセキュリティレビューを行う:

**代表的な脆弱性:** SQLインジェクション、XSS、認証・認可の問題、
安全でないデータ処理、ハードコードされた秘密情報

**セキュリティのベストプラクティス:** 入力検証、出力エンコーディング、
安全なデフォルト、安全なエラーメッセージ、適切なログ（機密情報を含めない）

発見した問題ごとに: ファイル・行番号 / 重大度（Critical〜Low）/
脆弱性の説明 / 推奨する修正。重大度順に並べる。
```

## 例3: ファイル指定のテスト実行

```markdown
---
description: 指定ファイルのテストを実行する
argument-hint: [test-file]
allowed-tools: Bash(npm:*), Bash(jest:*)
---

$1 のテストを実行する:

テスト実行: !`npm test $1`

結果を分析する: 成功/失敗、カバレッジ、性能問題、フレーキーなテスト。
失敗があればエラーメッセージに基づく修正を提案する。
```

使い方: `> /test-file src/utils/helpers.test.ts`

## 例4: ドキュメント生成

```markdown
---
description: ファイルのドキュメントを生成する
argument-hint: [source-file]
---

@$1 の包括的なドキュメントを生成する。

含める内容:
**概要**: 目的と責務、主な機能、依存
**API**: 関数シグネチャ、パラメータと型、戻り値、例外
**使用例**: 基本的な使い方、よくあるパターン、エッジケース
**実装メモ**: 計算量、性能の考慮、既知の制限

プロジェクトドキュメントに使えるマークダウン形式で出力する。
```

## 例5: Gitステータスの要約

```markdown
---
description: Gitリポジトリの状態を要約する
allowed-tools: Bash(git:*)
---

リポジトリ状態の要約:

**現在のブランチ:** !`git branch --show-current`
**状態:** !`git status --short`
**最近のコミット:** !`git log --oneline -5`
**リモートとの差分:** !`git fetch && git status -sb`

変更の要約・推奨される次のアクション・警告や問題を提示する。
```

## 例6: デプロイ

```markdown
---
description: 指定環境へデプロイする
argument-hint: [environment] [version]
allowed-tools: Bash(kubectl:*), Read
---

$1 環境へバージョン $2 をデプロイする。

**デプロイ前チェック:**
1. $1 の設定の存在を確認する
2. バージョン $2 の妥当性を確認する
3. クラスタへの接続確認: !`kubectl cluster-info`

**デプロイ手順:**
マニフェスト更新 → 適用 → ロールアウト監視 → Pod健全性確認 → スモークテスト

**ロールバック計画:** 問題発生時に備えて現在のバージョンを記録する。

デプロイを続行するかユーザーに確認する。
```

使い方: `> /deploy staging v1.2.3`

## 例7: ファイル比較

```markdown
---
description: 2つのファイルを比較する
argument-hint: [file1] [file2]
---

@$1 と @$2 を比較する。

1. **差分**: 追加・削除・変更された行
2. **機能的な変更**: 破壊的変更、新機能、バグ修正、リファクタリング
3. **影響**: 影響を受けるコンポーネント、他所で必要な更新、移行の要否
4. **推奨**: レビューの重点、テスト要件、ドキュメント更新

構造化された比較レポートとして提示する。
```

## 例8: クイックフィックス

```markdown
---
description: よくある問題を素早く修正する
argument-hint: [issue-description]
model: haiku
---

以下を素早く修正する: $ARGUMENTS

**アプローチ:** 問題の特定 → 関連コードの発見 → 修正の提案 → 解決策の説明

重視する点: シンプルで直接的な解決、最小限の変更、既存パターンの踏襲、
破壊的変更なし。ファイルパスと行番号付きでコード変更を提示する。
```

使い方: `> /quick-fix ボタンがクリックに反応しない`

## 例9: リサーチ

```markdown
---
description: トピックのベストプラクティスを調査する
argument-hint: [topic]
model: sonnet
---

以下のベストプラクティスを調査する: $ARGUMENTS

1. **現状**: 現在の扱い方、既存の実装
2. **業界標準**: 一般的なパターン、推奨アプローチ、ツール・ライブラリ
3. **比較**: 現状と標準の差、必要な改善、移行の考慮点
4. **推奨**: 具体的なアクション、優先度と工数見積り、参考資料

調査に基づく実行可能なガイダンスを提示する。
```

## 例10: コード説明

```markdown
---
description: コードの仕組みを説明する
argument-hint: [file-or-function]
---

@$1 を詳しく説明する。

1. **概要**: 何をするか、なぜ存在するか、システム内での位置づけ
2. **ステップ解説**: 行単位の説明、主要なアルゴリズム、重要な詳細
3. **入出力**: パラメータと型、戻り値、副作用
4. **エッジケース**: エラー処理、特殊ケース、制限
5. **使用例**: 呼び出し方、よくあるパターン、連携ポイント

ジュニアエンジニアに適したレベルで説明する。
```

## 主要パターンまとめ

| パターン | 書き方 | 用途 |
| --- | --- | --- |
| 読み取り専用分析 | `allowed-tools: Read, Grep` | レビュー・ドキュメント・分析 |
| Git操作 | `allowed-tools: Bash(git:*)` + `` !`git status` `` | リポジトリ状態・コミット分析 |
| 単一引数 | `argument-hint: [target]` + `$1` | ファイル操作・対象指定 |
| 複数引数 | `[source] [target] [options]` + `$1 $2 $3` | ワークフロー・デプロイ・比較 |
| 高速実行 | `model: haiku` | 単純・反復的なコマンド |
| ファイル比較 | `@$1 と @$2 を比較` | 差分分析・移行計画 |
| コンテキスト収集 | `` !`git status` `` + `@file` | 情報に基づく判断 |

## 書き方のコツ

1. **基本から始める**: 単一責務・明確な目的。最初は frontmatter なしでもよい
2. **段階的に複雑化**: 機能を1つずつ追加して検証する
3. **説明的な名前**: コマンド名だけで目的がわかるように
4. **引数を文書化**: 常に argument-hint を付ける
5. **エラーを考慮**: 引数やファイルの欠落に備える
