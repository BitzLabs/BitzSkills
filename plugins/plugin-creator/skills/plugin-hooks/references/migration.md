# 基本フックから発展フックへの移行

基本的なコマンドフックから、保守性と柔軟性に優れたプロンプトベース
フックへの移行ガイド。

## なぜ移行するか

プロンプトベースフックの利点:

- **自然言語推論**: LLMが文脈と意図を理解する
- **エッジケースへの強さ**: 想定外のシナリオに適応する
- **bashスクリプト不要**: 書くのも保守するのも簡単
- **柔軟な検証**: コードなしで複雑なロジックを扱える

## 移行例1: bashコマンド検証

### 移行前（コマンドフック）

```json
{
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        { "type": "command", "command": "bash validate-bash.sh" }
      ]
    }
  ]
}
```

```bash
#!/bin/bash
input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command')

# ハードコードされた検証ロジック
if [[ "$command" == *"rm -rf"* ]]; then
  echo "危険なコマンドを検出" >&2
  exit 2
fi
```

**問題点:** 「rm -rf」の完全一致しか見ない（`rm -fr` や `rm -r -f` を
見逃す）、他の危険なコマンド（`dd`, `mkfs` 等）を見逃す、文脈を考慮しない、
bashの知識が必要。

### 移行後（プロンプトフック）

```json
{
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        {
          "type": "prompt",
          "prompt": "コマンド: $TOOL_INPUT.command。分析する: 1) 破壊的操作（rm -rf, dd, mkfs 等） 2) 権限昇格（sudo） 3) ユーザーの同意なしのネットワーク操作。説明付きで 'approve' または 'deny' を返す。",
          "timeout": 15
        }
      ]
    }
  ]
}
```

**利点:** あらゆる変種・パターンを捕捉、文字列でなく意図を理解、
スクリプトファイル不要、基準の追加が簡単、拒否時に自然言語の説明。

## 移行例2: ファイル書き込み検証

### 移行前

パターンのハードコード（`/etc/` の前方一致等）はシンボリックリンクや
エッジケース（`/etc` と `/etc/` の違い）を見逃し、ファイル内容も考慮しない。

### 移行後

```json
{
  "PreToolUse": [
    {
      "matcher": "Write|Edit",
      "hooks": [
        {
          "type": "prompt",
          "prompt": "ファイルパス: $TOOL_INPUT.file_path。内容プレビュー: $TOOL_INPUT.content（先頭200文字）。検証する: 1) システムディレクトリ（/etc, /sys, /usr）でない 2) 認証情報（.env, トークン, シークレット）でない 3) パストラバーサルがない 4) 内容が秘密情報を露出しない。'approve' または 'deny' を返す。"
        }
      ]
    }
  ]
}
```

**利点:** 内容も考慮した文脈対応、シンボリックリンクやエッジケースへの
対応、「システムディレクトリ」の自然な理解、内容中の秘密情報の検出。

## コマンドフックを残すべき場合

### 1. 決定的な性能チェック

```bash
# ファイルサイズの高速チェック
size=$(stat -c%s "$file_path" 2>/dev/null)
if [ "$size" -gt 10000000 ]; then
  echo '{"decision": "deny", "reason": "ファイルが大きすぎます"}' >&2
  exit 2
fi
```

検証が純粋に数値的・決定的なとき。

### 2. 外部ツール統合

```bash
# セキュリティスキャナの実行
scan_result=$(security-scanner "$file_path")
if [ "$?" -ne 0 ]; then
  echo "セキュリティスキャン失敗: $scan_result" >&2
  exit 2
fi
```

Yes/No を返す外部ツールと統合するとき。

### 3. 非常に高速なチェック（50ms未満）

```bash
if [[ "$command" =~ ^(ls|pwd|echo)$ ]]; then
  exit 0  # 安全なコマンド
fi
```

性能が重要でロジックが単純なとき。

## ハイブリッドアプローチ

両方を組み合わせて多段検証にする。コマンドフックが高速な決定的チェックを
行い、プロンプトフックが複雑な推論を担当する（`references/advanced.md` の
「多段検証」を参照）。

## 移行チェックリスト

- [ ] コマンドフックの検証ロジックを特定する
- [ ] ハードコードされたパターンを自然言語の基準に変換する
- [ ] 旧フックが見逃していたエッジケースでテストする
- [ ] LLMが意図を理解しているか確認する
- [ ] 適切なタイムアウトを設定する（プロンプトフックは通常15〜30秒）
- [ ] README で新フックを文書化する
- [ ] 旧スクリプトを削除またはアーカイブする

## 移行のコツ

1. **1つずつ移行する**: 一度に全部移行しない
2. **徹底的にテストする**: 旧フックが捕捉していたものを新フックも捕捉するか確認
3. **改善の機会にする**: 移行ついでに検証を強化する
4. **旧スクリプトは参照用に保管する**
5. **理由を文書化する**: なぜプロンプトフックが良いか README に書く

## 移行パターン集

### 文字列一致 → 自然言語

```bash
# 移行前
if [[ "$command" == *"sudo"* ]]; then ... fi
```

```
# 移行後
「権限昇格（sudo, su 等）をチェックする」
```

### 正規表現 → 意図

```bash
# 移行前
if [[ "$file" =~ \.(env|secret|key|token)$ ]]; then ... fi
```

```
# 移行後
「認証情報ファイル（.env, secrets, keys, tokens）への書き込みでないことを検証する」
```

### 複数条件 → 基準リスト

```bash
# 移行前
if [ 条件1 ] || [ 条件2 ] || [ 条件3 ]; then ... fi
```

```
# 移行後
「チェックする: 1) 条件1 2) 条件2 3) 条件3。いずれかに該当したら拒否する。」
```

## まとめ

プロンプトベースフックへの移行で、プラグインはより保守しやすく、柔軟で、
強力になる。コマンドフックは決定的なチェックと外部ツール統合に限定する。
