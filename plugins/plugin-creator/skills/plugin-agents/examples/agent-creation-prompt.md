# AI支援エージェント生成テンプレート

エージェント生成システムプロンプトを使って、Claude にエージェントを
生成させるためのテンプレート。

## 使い方

### Step 1: エージェントの要望を整理する

- どんなタスクを扱わせるか？
- いつトリガーされるべきか？
- proactive（能動的）か reactive（受動的）か？
- 主要な責務は何か？

### Step 2: 生成プロンプトを使う

（`references/agent-creation-system-prompt.md` のシステムプロンプトを
読み込んだ状態で）Claude に送る:

```
次の要望に基づいてエージェント設定を作成する: "[あなたの説明]"

JSONオブジェクトのみを返すこと。他のテキストは不要。
```

### Step 3: Claude が JSON を返す

```json
{
  "identifier": "agent-name",
  "whenToUse": "〜のときにこのエージェントを使用する。例: <example>...</example>",
  "systemPrompt": "あなたは... **中心的な責務:**..."
}
```

### Step 4: エージェントファイルに変換する

`agents/[identifier].md` を作成する:

```markdown
---
name: [JSONのidentifier]
description: [JSONのwhenToUse]
model: inherit
color: [blue/cyan/green/yellow/magenta/red から選択]
tools: ["Read", "Write", "Grep"]  # 任意: ツールを制限する場合
---

[JSONのsystemPrompt]
```

## 例1: コードレビューエージェント

**要望:**

```
コード変更の品質問題・セキュリティ脆弱性・ベストプラクティス準拠を
レビューするエージェントが欲しい。コードが書かれた後に呼ばれ、
具体的なフィードバックを提供すること。
```

**生成結果から作るファイル** (`agents/code-quality-reviewer.md`):

```markdown
---
name: code-quality-reviewer
description: ユーザーがコードを書いて品質レビューが必要なとき、または明示的にコード変更のレビューを求めたときにこのエージェントを使用する。例:

<example>
Context: ユーザーが新機能を実装したところ
user: "認証機能を追加しました"
assistant: "いいですね。コード品質をレビューします。"
<commentary>
コードが書かれたので code-quality-reviewer エージェントをトリガーする。
</commentary>
assistant: "code-quality-reviewer エージェントで変更を分析します。"
</example>

<example>
Context: ユーザーが明示的にレビューを依頼
user: "コードに問題がないかレビューして"
assistant: "code-quality-reviewer エージェントで徹底レビューを行います。"
<commentary>
明示的なレビュー依頼がエージェントをトリガーする。
</commentary>
</example>

model: inherit
color: blue
tools: ["Read", "Grep", "Glob"]
---

あなたはソフトウェア実装の問題特定を専門とするコード品質レビューの
専門家です。

**中心的な責務:**
1. コード変更の品質問題（可読性・保守性・性能）を分析する
2. セキュリティ脆弱性（インジェクション・XSS・認証問題）を特定する
3. プロジェクトのベストプラクティスとコーディング規約への準拠を確認する
4. 行番号付きの実行可能で具体的なフィードバックを提供する

**レビュープロセス:**
1. 利用可能なツールでコード変更を読む
2. 以下を分析する:
   - コード品質（重複・複雑さ・明確さ）
   - セキュリティ（OWASP Top 10・入力検証）
   - ベストプラクティス（エラー処理・ログ・テスト）
   - プロジェクト固有の規約（CLAUDE.md から）
3. 問題を重大度（critical/major/minor）付きで特定する
4. 例を添えた具体的な提言を行う

**出力形式:**
構造化されたレビューを提示する:
1. 要約（2〜3文）
2. 重大な問題（修正必須）
3. 主要な問題（修正推奨）
4. 軽微な問題（修正が望ましい）
5. 良かった点
6. 総合評価

すべての所見にファイル名と行番号を含める。
```

## 例2: テスト生成エージェント

**要望:**

```
コードのユニットテストを生成するエージェント。既存コードを分析し、
プロジェクトの慣習に従った網羅的なテストスイートを作成すること。
```

**生成結果の概要**: identifier `test-generator`、
「テストのない新規コードに能動的にトリガー」する example、
テスト生成プロセス（対象の読み込み → テスト可能単位の特定 →
テストケース設計 → プロジェクトパターンに従った生成 → アサーション追加）
を含むシステムプロンプト。

## 例3: ドキュメントエージェント

**要望:**

```
APIドキュメントを書き、更新するエージェント。コードを分析して
明確で包括的なドキュメントを生成すること。
```

**結果**: identifier `api-docs-writer` のエージェントファイル。

## 効果的な生成のコツ

### 要望を具体的にする

```
❌ 「コードを手伝うエージェントが欲しい」

✅ 「TypeScriptのPRを型安全性の観点でレビューするエージェントが欲しい。
   型注釈の適切さ・anyの回避・ジェネリクスの正しい使用をチェックすること」
```

### トリガーの希望を伝える

```
「テストを生成するエージェントを作って。明示的に頼まれたときだけでなく、
コードが書かれた後に能動的にトリガーされること。」
```

### プロジェクトの文脈を伝える

```
「コードレビューエージェントを作って。このプロジェクトは React と
TypeScript を使うので、React のベストプラクティスと TypeScript の
型安全性をチェックすること。」
```

### 出力の期待値を定義する

```
「性能を分析するエージェントを作って。ファイル名・行番号付きの具体的な
提言と、推定される性能への影響を提示すること。」
```

## 生成後の検証

```bash
# 構造の検証
bash scripts/validate-agent.sh agents/your-agent.md

# トリガーの確認: description の example にあるシナリオで実際に試す
```

## 生成結果の改善

生成されたエージェントに不足があれば手動で編集する:

1. 何が足りない・間違っているか特定する
2. description の example の改善 / システムプロンプトの具体化 /
   プロセスステップの明確化 / 出力形式の定義強化 に注力する
3. 再検証・再テストする

## AI支援生成の利点と手動編集のタイミング

**利点**: 網羅的（エッジケース・品質チェックを含む）・一貫性・高速・
example の自動生成・完全なシステムプロンプト構造。

**手動編集が必要な場合**: プロジェクト固有パターン・独自のツール構成・
固有のペルソナ・既存エージェントとの統合・精密なトリガー条件。

生成から始めて手動で磨くのが最良の結果につながる。
