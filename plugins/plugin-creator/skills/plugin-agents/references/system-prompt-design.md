# システムプロンプト設計パターン

自律的で高品質な動作を可能にする、エージェントのシステムプロンプトの
書き方の完全ガイド。

## 基本構造

すべてのエージェントのシステムプロンプトは、実績のある次の構造に従う:

```markdown
あなたは[特定のドメイン]を専門とする[特定の役割]です。

**中心的な責務:**
1. [主要な責務 — メインのタスク]
2. [副次的な責務 — 補助タスク]
3. [必要に応じて追加の責務]

**[タスク名]プロセス:**
1. [具体的な第一ステップ]
2. [具体的な第二ステップ]
3. [明確なステップを続ける]

**品質基準:**
- [具体性のある基準1]
- [具体性のある基準2]

**出力形式:**
以下の構造で結果を提示する:
- [構成要素1]
- [構成要素2]
- [具体的な書式要件を含める]

**エッジケース:**
- [ケース1]: [具体的な対処法]
- [ケース2]: [具体的な対処法]
```

## パターン1: 分析エージェント

コード・PR・ドキュメントを分析するエージェント向け:

```markdown
あなたは[分析種別]を専門とする[ドメイン]分析の専門家です。

**中心的な責務:**
1. [対象]を[特定の問題]について徹底的に分析する
2. [パターン/問題/機会]を特定する
3. 実行可能な提言を行う

**分析プロセス:**
1. **コンテキスト収集**: 利用可能なツールで[対象]を読む
2. **初回スキャン**: 明白な[問題/パターン]を特定する
3. **詳細分析**: [観点]を精査する:
   - [観点1]: [基準]をチェックする
   - [観点2]: [基準]を検証する
4. **所見の統合**: 関連する問題をグループ化する
5. **優先度付け**: [重大度/影響/緊急度]でランク付けする
6. **レポート生成**: 出力テンプレートに従って整形する

**品質基準:**
- すべての所見に file:line 参照を含める
- 問題を重大度で分類する（critical/major/minor）
- 提言は具体的で実行可能にする
- バランスのため良い点も含める

**出力形式:**
## 要約
[2〜3文の概要]

## 重大な問題
- [file:line] - [問題の説明] - [提言]

## 主要な問題 / 軽微な問題 / 提言
[...]

**エッジケース:**
- 問題なし: 肯定的なフィードバックと検証結果を提示する
- 問題が多すぎる: グループ化して上位10件に絞る
- 不明瞭なコード: 推測せず確認を求める
```

## パターン2: 生成エージェント

コード・テスト・ドキュメントを作るエージェント向け:

```markdown
あなたは高品質な[成果物の種類]の作成を専門とする[ドメイン]エンジニアです。

**中心的な責務:**
1. [品質基準]を満たす[成果物]を生成する
2. [規約/パターン]に従う
3. [正確性/完全性/明確さ]を保証する

**生成プロセス:**
1. **要件の理解**: 何を作る必要があるか分析する
2. **コンテキスト収集**: 既存の[コード/ドキュメント/テスト]を読んで
   パターンを把握する
3. **構造設計**: [アーキテクチャ/構成/フロー]を計画する
4. **生成**: 規約とベストプラクティスに従って[成果物]を作る
5. **検証**: [正確性/完全性]を確認する
6. **文書化**: 必要なコメント・説明を加える

**品質基準:**
- プロジェクト規約に従う（CLAUDE.md を確認）
- エラーハンドリングを含める
- 明確で十分な文書化

**エッジケース:**
- コンテキスト不足: ユーザーに確認を求める
- パターンの矛盾: 最新/明示的なパターンに従う
- 複雑な要件: 小さく分割する
```

## パターン3: 検証エージェント

検証・チェック・確認を行うエージェント向け:

```markdown
あなたは[品質観点]の保証を専門とする[ドメイン]バリデータです。

**中心的な責務:**
1. [対象]を[基準]に照らして検証する
2. 違反と問題を特定する
3. 明確な合否判定を出す

**検証プロセス:**
1. **基準の読み込み**: 検証要件を理解する
2. **対象のスキャン**: 検証対象を読む
3. **ルールチェック**: 各ルールについて検証方法を実行する
4. **違反の収集**: 各違反を詳細付きで記録する
5. **重大度の評価**: 問題を分類する
6. **判定**: [基準を満たす]場合のみ合格とする

**品質基準:**
- すべての違反に具体的な位置を含める
- 重大度を明示する
- 修正案を提示する
- 誤検出を出さない

**出力形式:**
## 検証結果: [PASS/FAIL]

## 要約 / 違反 [件数] / 提言
[...]

**エッジケース:**
- 違反なし: 検証パスを明示する
- 違反が多すぎる: 種類別にグループ化し上位20件を示す
- 曖昧なルール: 不確実性を記録し確認を求める
```

## パターン4: オーケストレーションエージェント

複数のツールやステップを調整するエージェント向け:

```markdown
あなたは[複雑なワークフロー]の調整を専門とする[ドメイン]
オーケストレータです。

**中心的な責務:**
1. [複数ステッププロセス]を調整する
2. [リソース/ツール/依存関係]を管理する
3. [完遂/統合]を保証する

**オーケストレーションプロセス:**
1. **計画**: 全体のワークフローと依存関係を理解する
2. **準備**: 前提条件を整える
3. **フェーズ実行**: 各フェーズを[ツール]で実行する
4. **監視**: 進捗を追跡し失敗に対処する
5. **確認**: 完了を検証する
6. **報告**: 包括的なサマリーを提示する

**エッジケース:**
- フェーズの失敗: リトライを試み、だめなら報告して停止する
- 依存の欠如: ユーザーに要求する
- タイムアウト: 部分的完了を報告する
```

## 記述スタイルの指針

### 二人称で書く

```
✅ あなたは〜に責任を持ちます / あなたは〜を分析します
❌ このエージェントは〜を分析する / 私は〜を分析します
```

### 具体的に書く

```
✅ 全データベースクエリのパラメータ化を調べてSQLインジェクションをチェックする
❌ セキュリティ問題を探す

✅ 各所見に file:line 参照を付ける
❌ 問題の場所を示す

✅ critical（セキュリティ）/ major（バグ）/ minor（スタイル）に分類する
❌ 問題の重大度を評価する
```

### 実行可能な指示にする

```
✅ Read ツールでファイルを読み、Grep でパターンを検索する
❌ コードを分析する

✅ test/path/to/file.test.ts にテストファイルを生成する
❌ テストを作る
```

## よくある落とし穴

| ❌ 悪い例 | ✅ 良い例 |
| --- | --- |
| 「ユーザーのコードを手伝う」「役に立つ」 | 「TypeScriptの型安全性の問題を分析する」「不適切な any の使用を特定する」 |
| 「コードを分析してフィードバックする」（方法なし） | 5段階の分析プロセスをステップで明記する |
| 「レポートを提供する」（形式なし） | 見出し・項目・file:line 形式まで定義する |

## 長さの目安

| 規模 | 語数目安 | 内容 |
| --- | --- | --- |
| 最小 | ~500語 | 役割 + 責務3つ + 5ステップのプロセス + 出力形式 |
| 標準 | 1,000〜2,000語 | 詳細な役割・責務5〜8・プロセス8〜12ステップ・品質基準・エッジケース3〜5 |
| 包括的 | 2,000〜5,000語 | 背景付きの役割・多段プロセス・複数の出力形式・多数のエッジケース・プロンプト内の例 |

10,000語超は避ける（効果が逓減する）。

## システムプロンプトのテスト

**完全性**: システムプロンプトだけで次を処理できるか？
典型的なタスク / 明記したエッジケース / エラーシナリオ / 不明瞭な要件 /
大きく複雑な入力 / 空・欠落した入力。

**明確さ**: 他の開発者が読んで、エージェントの役割・プロセス・出力形式・
品質基準を理解できるか？

**反復改善**: テスト後、つまずいた箇所を特定し、不足する指針を追加し、
曖昧な指示を明確にして再テストする。

## まとめ

効果的なシステムプロンプトは:

- **具体的**: 何を・どうやるかが明確
- **構造的**: 明確なセクションで整理されている
- **完全**: 通常ケースとエッジケースをカバーする
- **実行可能**: 具体的なステップを提供する
- **テスト可能**: 測定可能な基準を定義する
