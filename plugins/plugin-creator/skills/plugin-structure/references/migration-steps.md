# update のバージョン移行（マイグレーション）機構

標準スキル `<plugin名>:update`（`lifecycle-skills.md`）の**任意拡張**（CORE-CON-009）。
プラグインがバージョンアップで**配置先に残す状態**の形式を変えたとき、旧形式のまま
取り残された状態を、バージョン単位の累積マイグレーションステップで最新形式へ移行する。
状態の形式変更を伴わないプラグインは本機構を実装しなくてよい。

移行対象となる「状態」の例:

- 配置先スキルの frontmatter `metadata` スキーマ（キーの追加・改名・構造変更）
- 対象プロジェクトの `.spec` 系 frontmatter の書式（id 体系・必須キー等）
- `.claude/<plugin>.local.md` 等の設定ファイル（レジストリ・委譲マトリクス等の構造）

**責務外**: スキルファイル本体（SKILL.md・references）の置き換え可否判定。これは
skill-packager の安全判定（skill-creator の `skill-packager/references/lifecycle.md`）が担う。
本機構は「置き換えてよいか」を規定せず、「置き換え後の状態をどう直すか」だけを規定する
（二重規定の禁止）。

## バージョン軸

- マイグレーションを駆動するのは**プラグイン version**（`plugin.json`。3マニフェスト共通の
  semver）。skill-packager の安全判定が使うスキル単位の `metadata.version` とは**別の軸**。
- **D**（配置先の現在バージョン）= 配置先状態の自己記述から読む。`init` / `update` は
  `.claude/<plugin>.local.md` 等の状態記録に、展開時のプラグイン version を stamp しておくこと。
- **T**（移行先バージョン）= ライブラリ側（更新後プラグイン）の `plugin.json` の version。
- semver 比較の手順は skill-packager の規約（3整数分解・数値比較）を再利用する。
  D か T が欠落・不正形式なら比較不能→**安全側停止**。

## ステップのファイル配置（1ファイル1ステップ）

`update` スキルの `references/migrations/<from>-to-<to>.md`（例: `1.2.0-to-2.0.0.md`）に置く。

- ステップは**状態形式が実際に変わった版にのみ**作る。形式無変更の version bump に
  ステップは不要（チェーン上の飛び番は正常）。
- ステップは連続チェーンとして書く: あるステップの `to` は、次に形式が変わるステップの
  `from` と一致させる（連結リスト）。最初のステップの `from` はプラグイン初版（baseline）。
- **状態形式を変える PR は、対応するマイグレーションステップを同じ PR に同梱する**
  （書き忘れは機械検出できないため、これを規律とする）。

## ステップの書式（必須セクション）

各ステップファイルは以下をすべて持つ。**宣言的 Markdown**（エージェントが解釈実行する
手順書）であり、実行コードではない。

| セクション | 必須 | 内容 |
|---|---|---|
| `from` / `to` | ○ | 入力として受け付ける形式のバージョンと適用後のバージョン（プラグイン version の semver、`from < to`）。ファイル名と一致させる |
| `targets` | ○ | 変換対象（例: `deployed:SKILL.md#metadata` / `workspace:.spec/*/frontmatter` / `config:.claude/<plugin>.local.md`）。ファイル横断可 |
| `transform` | ○ | 変換内容（何を→何に）。before/after の例を必ず含める |
| `guard` | ○ | **適用済み判定＝冪等性の根拠**。「この条件が成立していれば適用済みとして no-op」を機械確認可能な形で書く（例: 新キー `x` が既に存在する）。ステップ本文は「guard 判定→未適用なら変換」の順で記述する |
| `verify` | ○ | 適用成功の機械確認（例: YAML パース可・新キー存在・旧キー不在） |
| `rollback` | ○ | 失敗時の復元手順（原則: 適用前バックアップからの復元。破壊的置換を避け加算的変換を推奨） |

## チェーン解決と適用

1. D と T を読む（上記バージョン軸）。読めなければ安全側停止。
2. `D >= T` なら移行不要（更新なしか降格）→終了。
3. `migrations/` の全ステップをファイル名から from/to 抽出して `to` 昇順にソートし、
   **適用候補** = `D < to <= T` を満たす集合を取る。
4. 適用候補が空 → D〜T 間に形式変更なし→移行不要で終了（正常）。
5. **連続性検査**（dry-run。書き込み前に実施）:
   - D が最古ステップの `from` より古く、その間を接続するステップが無い → **GAP**（配置先が
     古すぎて変換の入り口が無い）→安全側停止。
   - 隣接する候補ステップで `step[i].to != step[i+1].from` → **GAP**（チェーン断裂）→安全側停止。
6. PASS なら `from` 昇順に**逐次適用**。各ステップで guard 判定→変換→`verify`。verify 失敗は
   当該ステップの `rollback` を実行して停止・報告。
7. **全ステップ成功後に**、状態記録のバージョンを D→T へ stamp する（コミットマーカー。
   必ず最後に行う）。途中失敗時は D が旧値のまま残るため `update` は再実行可能で、
   guard により適用済みステップは no-op で通過し、修正後の再走で収束する。

### 安全側停止（変換せず停止し、状態と不足をユーザーへ提示する）

- (a) D または T が semver として解釈不能
- (b) D が最古ステップの `from`（baseline）より古く、その間を接続するステップが無い
- (c) 適用候補チェーンに断裂がある
- (d) ステップの `verify` が不成立、かつ `rollback` 後も回復不能
- (e) 変換対象ファイルが配置先に存在しない／パース不能で入力形式を判定できない

(a)〜(c) は dry-run 段階で検出でき、書き込み前に停止する。無視して先へ進めてはならない。
なお (a)〜(c) は読み取り専用で判定できるため、`doctor` が未適用マイグレーションや
チェーン断裂を事前検知して提示してもよい（任意）。

## 書き込み前の承認フロー

配置先は `~/.claude/skills/` や対象プロジェクトの `.claude/<plugin>.local.md` 等、
**リポジトリ外**になりうる。書き込み前に必ず:

1. dry-run で変換対象ファイル一覧と before/after 差分プレビューを提示する
2. 書き込み先の絶対パスと、それがリポジトリ内/外のいずれかを明示する
3. ユーザー承認を得るまで一切書き込まない（バックアップ取得を含む実書き込みは承認後）
4. 失敗時 rollback 用のバックアップ先を提示する

書き込み先が**作業中リポジトリ内**の場合（例: 開発リポジトリ自身の `.spec` を移行する
ドッグフーディング）は、リポジトリ外の承認フローは適用しないが、dry-run 差分の提示は
同様に行い、通常のブランチ規律（デフォルトブランチへの直接コミット禁止）に従う。

## 動作確認の指針（migrations を同梱するプラグイン側で実施）

合成フィクスチャ（旧形式を模した frontmatter / `.local.md` を一時ディレクトリに用意）で
少なくとも次を確認する:

- **冪等性**: 同一ステップの二重適用で状態が壊れない（2回目は guard で no-op になる）
- **安全側停止**: 中間ステップを欠落させたフィクスチャで、書き込みが起きずに停止・提示される
