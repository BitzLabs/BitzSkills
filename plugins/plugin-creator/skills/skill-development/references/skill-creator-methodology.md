# スキル設計方法論

anthropics/skills の skill-creator 方法論を基にした、スキル設計の詳細ガイド。
（体系的なスキル開発フローは `skill-creator` プラグインも参照）

## スキルとは何か

スキルは、専門知識・ワークフロー・ツールを提供してエージェントの能力を拡張する
モジュール式の自己完結型パッケージである。特定ドメインの「オンボーディング
ガイド」であり、どのモデルも完全には持ち得ない手続き的知識を装備させる。

**スキルが提供するもの:**

1. 専門ワークフロー — 特定ドメインの複数ステップ手順
2. ツール統合 — 特定のファイル形式・APIの扱い方
3. ドメイン知識 — 組織固有の知識・スキーマ・ビジネスロジック
4. バンドルリソース — 複雑・反復的なタスクのためのスクリプト・資料・素材

## 具体例駆動の設計

### なぜ具体例から始めるか

抽象的に「PDFスキルを作る」と考えると、何を同梱すべきか判断できない。
「このPDFを回転して」「このフォームに記入して」という具体的な発話から
逆算すると、必要なリソースが明確になる。

### 分析の手順

各具体例について:

1. **ゼロから実行するとどうなるか**を考える
2. **繰り返し発生する作業**を特定する
3. その作業を**どのリソース形式で固定化するか**決める

| 繰り返し発生する作業 | 固定化する形式 |
| --- | --- |
| 同じコードを毎回書き直す | `scripts/` の実行スクリプト |
| 同じ資料・スキーマを毎回調べ直す | `references/` のドキュメント |
| 同じ雛形を毎回作り直す | `assets/` のテンプレート |
| 同じ設定例を毎回考え直す | `examples/` のサンプル |

### 分析例

**pdf-editor スキル**（「このPDFを回転して」）:
回転処理は毎回同じコード → `scripts/rotate_pdf.py` を同梱。

**frontend-webapp-builder スキル**（「TODOアプリを作って」）:
毎回同じボイラープレート → `assets/hello-world/` テンプレートを同梱。

**big-query スキル**（「今日ログインしたユーザー数は？」）:
毎回スキーマを再発見している → `references/schema.md` を同梱。

**hooks 開発スキル**（プラグイン向け）:
hooks.json の検証とスクリプトのテストが毎回必要 →
`scripts/validate-hook-schema.sh` と `scripts/test-hook.sh` を同梱、
詳細パターンは `references/patterns.md` に分離。

## description 設計

description はエージェントがスキルを使うかどうかを判断する**唯一の材料**。

### 必須要素

1. **何をするか** — スキルの機能の要約
2. **いつ使うか** — 具体的なトリガーフレーズ（ユーザーが実際に言いそうな言葉）

### 良い例・悪い例

```yaml
# ✅ 良い: 第三者視点・具体的フレーズ・複数のトリガー
description: 「フックを作りたい」「PreToolUseフックを追加」「ツール使用を検証したい」と言われたとき、またはフックイベント（PreToolUse, PostToolUse, Stop）に言及されたときに使用する。フックAPIの包括的なガイダンスを提供する。

# ❌ 悪い: 曖昧・トリガーなし
description: フック関連の作業を支援する。
```

### 大きな references を持つ場合

references のファイルが大きい（1万語超）場合は、SKILL.md に grep 用の
検索パターンを載せて、必要な箇所だけ読めるようにする。

## 記述スタイル規則

### 命令形で書く

本文は動詞から始まる指示で書く。二人称（「あなたは〜」）は使わない。

```
✅ フックを作るには、イベント種別を定義する。
✅ 使用前に設定を検証する。
❌ あなたはイベント種別を定義する必要があります。
❌ Claudeは設定を検証すべきです。
```

### 客観的・指示的な言葉

「誰がやるか」ではなく「何をやるか」に集中する。
「〜できます」「〜かもしれません」より「〜する」と断定する。

## 検証チェックリスト

**構造:**
- [ ] SKILL.md が存在し、YAML frontmatter が正しくパースできる
- [ ] frontmatter に name / description がある
- [ ] 本文が実質的な内容を持つ
- [ ] 参照しているファイルが実在する

**description の品質:**
- [ ] 「何をするか」と「いつ使うか」の両方を含む
- [ ] ユーザーが実際に言いそうな具体的フレーズを含む
- [ ] 曖昧・汎用的でない

**コンテンツの品質:**
- [ ] 本文が命令形
- [ ] 本文がリーン（理想1500〜2000語、最大5000語）
- [ ] 詳細が references に分離されている
- [ ] 実例が完全で動作する
- [ ] スクリプトが実行可能でドキュメント化されている

**progressive disclosure:**
- [ ] コア概念が SKILL.md に、詳細が references/ にある
- [ ] SKILL.md がリソースを明示的に参照している
- [ ] ファイル間で情報が重複していない

## 反復改善のワークフロー

1. 実タスクでスキルを使う
2. つまずき・非効率に気づく
3. SKILL.md やリソースの更新点を特定する
4. 変更を実装し、再テストする

使った直後（スキルの挙動の記憶が新しいうち）に改善するのが最も効果的。

**よくある改善:**

- description のトリガーフレーズを強化する
- SKILL.md の長いセクションを references/ に移す
- 不足している実例・スクリプトを追加する
- 曖昧な指示を明確にする
- エッジケースの扱いを追加する
