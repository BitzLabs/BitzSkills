# プラグイン作成ワークフロー

初期構想からテスト済みの実装まで、完全で高品質な Claude Code /
Antigravity 2.0 プラグインの作成をユーザーとともに進める。体系的なアプローチに従う: 要件の理解 →
コンポーネント設計 → 詳細の明確化 → ベストプラクティスに沿った実装 →
検証 → テスト。

## 基本原則

- **確認の質問をする**: プラグインの目的・トリガー・スコープ・
  コンポーネントに関する曖昧さをすべて特定する。仮定せず具体的に質問し、
  回答を待ってから実装に進む
- **関連スキルを読み込む**: 必要に応じて Skill ツールで plugin-creator の
  スキル（plugin-structure, plugin-hooks, plugin-agents 等）を読み込む
- **専門エージェントを使う**: agent-creator / plugin-validator /
  agent-creator エージェントでAI支援開発を行う
- **ベストプラクティスに従う**: plugin-creator 自身の実装パターンを適用する
- **progressive disclosure**: references / examples を持つリーンなスキルを作る
- **TodoWrite を使う**: 全フェーズを通じて進捗を追跡する

**初期リクエスト:** $ARGUMENTS

## Phase 1: 発見（Discovery）

**目標**: 何のプラグインを作るのか、どんな問題を解決するのかを理解する

**アクション**:

1. 全フェーズのTODOリストを作成する
2. 引数から目的が明確なら: 理解を要約し、プラグイン種別
   （統合 / ワークフロー / 分析 / ツールキット等）を特定する
3. 不明確なら質問する:
   - このプラグインはどんな問題を解決するか？
   - 誰がいつ使うか？
   - 何をすべきか？
   - 参考になる類似プラグインはあるか？
4. **対象プラットフォームを確認する**: Claude Code のみ / Antigravity 2.0 のみ /
   両対応。両対応ならコンポーネント選定に制約が入る
   （スキル中心・コマンドはスキル変換されても成立する内容・
   フック/MCPは設定ファイル2系統）ことを伝える
5. 理解を要約し、ユーザーの確認を得てから進む

**成果**: プラグインの目的と対象ユーザーの明確な記述

---

## Phase 2: コンポーネント計画

**目標**: 必要なコンポーネントを決める

このフェーズの前に **plugin-structure スキルを必ず読み込む**。

**アクション**:

1. plugin-structure スキルでコンポーネント種別を把握する
2. 要件を分析して必要なコンポーネントを判定する:
   - **スキル**: 専門知識が必要か？（フックAPI・MCPパターン等）
   - **コマンド**: ユーザー起点の操作があるか？（deploy / configure / analyze）
   - **エージェント**: 自律的なタスクがあるか？（検証・生成・分析）
   - **フック**: イベント駆動の自動化が必要か？（検証・通知）
   - **MCP**: 外部サービス統合が必要か？（DB・API）
   - **設定**: ユーザー設定が必要か？（.local.md ファイル）
3. 必要な各コンポーネントについて、個数・役割・おおまかなトリガー/
   使用パターンを特定する
4. コンポーネント計画を表でユーザーに提示する:

   | コンポーネント | 個数 | 目的 |
   |----------------|------|------|
   | スキル         | 2    | フックパターン、MCPの使い方 |
   | コマンド       | 3    | deploy, configure, validate |
   | エージェント   | 1    | 自律的な検証 |
   | フック         | 0    | 不要 |
   | MCP            | 1    | データベース統合 |

5. ユーザーの承認または調整を得る

**成果**: 作成するコンポーネントの確定リスト

---

## Phase 3: 詳細設計と確認の質問

**目標**: 各コンポーネントを詳細に仕様化し、曖昧さをすべて解消する

**重要**: 最重要フェーズの1つ。**スキップ禁止。**

**アクション**:

1. 計画の各コンポーネントについて、仕様が曖昧な点を特定する:
   - **スキル**: 何がトリガーか？ どんな知識を提供するか？ どの程度詳細か？
   - **コマンド**: 引数は？ ツールは？ 対話的か自動か？
   - **エージェント**: トリガーは能動的/受動的？ ツールは？ 出力形式は？
   - **フック**: どのイベント？ プロンプト型かコマンド型か？ 検証基準は？
   - **MCP**: サーバー種別は？ 認証は？ どのツールか？
   - **設定**: フィールドは？ 必須/任意？ デフォルトは？
2. **質問をコンポーネント種別ごとのセクションに整理して提示する**
3. **回答を待ってから実装に進む**
4. 「お任せします」と言われたら、具体的な推奨案を提示して明示的な確認を得る

**スキルへの質問例**: どんなユーザーの発話でトリガーすべきか？
ユーティリティスクリプトを含めるか？ SKILL.md と references の詳細度の配分は？

**エージェントへの質問例**: 特定の操作の後に能動的にトリガーすべきか、
明示的な依頼のときだけか？ 必要なツールは？ 出力形式は？

**成果**: 各コンポーネントの詳細仕様

---

## Phase 4: プラグイン構造の作成

**目標**: ディレクトリ構造とマニフェストを作る

**アクション**:

1. プラグイン名を決める（kebab-case・説明的）
2. 作成場所をユーザーに確認する（カレント / ../new-plugin-name / カスタムパス）
3. bash でディレクトリ構造を作る:

   ```bash
   mkdir -p plugin-name/.claude-plugin
   mkdir -p plugin-name/skills     # 必要なら
   mkdir -p plugin-name/commands   # 必要なら
   mkdir -p plugin-name/agents     # 必要なら
   mkdir -p plugin-name/hooks      # 必要なら
   ```

4. Write ツールで `.claude-plugin/plugin.json` マニフェストを作る:

   ```json
   {
     "name": "plugin-name",
     "version": "0.1.0",
     "description": "[簡潔な説明]",
     "author": {
       "name": "[ユーザーから、またはデフォルト]",
       "email": "[メールまたはデフォルト]"
     }
   }
   ```

   **Antigravity 対応の場合**はルートにも `plugin.json` を作る
   （name / version / description。version は
   `.claude-plugin/plugin.json` と常に同じ値に保つ）

5. README.md のテンプレートを作る
6. 必要なら .gitignore を作る（`.claude/*.local.md` 等）
7. 新規フォルダなら git リポジトリを初期化する

**成果**: コンポーネントを受け入れる準備ができたプラグイン構造

---

## Phase 5: コンポーネント実装

**目標**: ベストプラクティスに従って各コンポーネントを作る

各コンポーネント種別の実装前に**関連スキルを読み込む**:
スキル→skill-creator プラグイン（同梱の考慮事項は plugin-skills）/ コマンド→plugin-commands /
エージェント→plugin-agents / フック→plugin-hooks /
MCP→plugin-mcp / 設定→plugin-settings

### スキルの場合

1. plugin-skills スキルを読み込む（作り方の方法論は skill-creator プラグイン）
2. 各スキルについて: 具体的な使用例を確認（Phase 3 の内容を使う）→
   リソース（scripts/ references/ examples/）を計画 → フォルダ構造を作成 →
   SKILL.md を書く（具体的トリガー付きの description・命令形のリーンな本文
   1500〜2000語・補助ファイルへの参照）→ references / examples / scripts を作る
3. skill-creator プラグインの skill-validator で各スキルを検証する

### コマンドの場合

1. plugin-commands スキルを読み込む
2. 各コマンドについて: frontmatter 付きマークダウンを書く。明確な
   description と argument-hint、最小限の allowed-tools、
   **ユーザー向けでなく Claude への指示**として本文を書く。
   関連スキルがあれば言及する

### エージェントの場合

1. plugin-agents スキルを読み込む
2. 各エージェントについて agent-creator エージェントを使う:
   役割の説明を渡す → identifier / example付きwhenToUse / systemPrompt が
   生成される → frontmatter 付きマークダウンを作る → model / color / tools を
   設定する → validate-agent.sh で検証する

### フックの場合

1. plugin-hooks スキルを読み込む
2. hooks/hooks.json を作る。複雑なロジックにはプロンプトベースフックを
   優先し、`${CLAUDE_PLUGIN_ROOT}` でポータブルにする。
   validate-hook-schema.sh と test-hook.sh でテストする
3. **Antigravity 対応の場合**はルートに Antigravity 書式の `hooks.json` も
   作る（イベント・matcher・入出力契約が別物。
   plugin-hooks の `references/antigravity-hooks.md` に従う）

### MCPの場合

1. plugin-mcp スキルを読み込む
2. `.mcp.json` を作る（ローカルは stdio、ホスト型は SSE。
   `${CLAUDE_PLUGIN_ROOT}` を使用）
3. **Antigravity 対応の場合**はルートに `mcp_config.json` も作る
   （stdio / SSE のみ。SSE は `serverUrl` キー）
4. 必要な環境変数を README に文書化し、セットアップ手順を書く

### 設定の場合

1. plugin-settings スキルを読み込む
2. README に設定テンプレートを載せ、フック・コマンドに設定読み取りを
   実装し、`.claude/*.local.md` を .gitignore に追加する

**進捗管理**: コンポーネント完了ごとにTODOを更新する

**成果**: 全コンポーネントの実装

---

## Phase 6: 検証と品質チェック

**目標**: プラグインが品質基準を満たし正しく動くことを保証する

**アクション**:

1. **plugin-validator エージェントを実行する**: マニフェスト・構造・命名・
   コンポーネント・セキュリティを包括的に検証し、レポートをレビューする
2. **重大な問題を修正する**
3. **skill-validator（skill-creator プラグイン）でレビューする**（スキルがある場合）: description の質・
   progressive disclosure・記述スタイルを確認し、提言を適用する
4. **エージェントのトリガーをテストする**（エージェントがある場合）:
   `<example>` の明確さ・トリガー条件の具体性を確認し、validate-agent.sh を実行する
5. **フック設定をテストする**（フックがある場合): validate-hook-schema.sh と
   test-hook.sh を実行し、`${CLAUDE_PLUGIN_ROOT}` の使用を確認する
6. **結果を提示する**: 検証結果の要約・残る問題・総合的な品質評価
7. **ユーザーに確認する**: 「検証完了。重大 [n]件、警告 [n]件。
   いま修正しますか、それともテストに進みますか？」

**成果**: 検証済みでテスト可能なプラグイン

---

## Phase 7: テストと動作確認

**目標**: 対象プラットフォーム上でプラグインが正しく動くことを確認する

**アクション**:

1. **導入手順を示す**:

   ```bash
   # Claude Code
   claude --plugin-dir /path/to/plugin-name

   # Antigravity 2.0（対応する場合）
   agy plugin validate /path/to/plugin-name
   agy plugin install /path/to/plugin-name
   ```

2. **確認チェックリスト**をユーザーに提示する:
   - [ ] スキルがトリガーで読み込まれる（トリガーフレーズで質問する）
   - [ ] コマンドが `/help` に出て正しく実行される
   - [ ] エージェントが適切なシナリオでトリガーされる
   - [ ] フックがイベントで発火する（該当時）
   - [ ] MCPサーバーが接続する（該当時）
   - [ ] 設定ファイルが機能する（該当時）
3. **テストの推奨**: スキルは description のトリガーフレーズで質問、
   コマンドは各種引数で実行、エージェントは example に合うシナリオを作成、
   フックは `claude --debug`、MCPは `/mcp` で確認
4. **ユーザーに確認する**: 「各コンポーネントのテストを一緒に進めますか、
   ご自身でテストしますか？」
5. ガイドを希望されたら、具体的なテストケースで各コンポーネントを
   順に確認する

**成果**: テスト・動作確認済みのプラグイン

---

## Phase 8: ドキュメントと次のステップ

**目標**: プラグインを文書化し、配布可能な状態にする

**アクション**:

1. **README の完全性を確認する**: 概要・機能・インストール・前提条件・
   使い方。MCPプラグインは必要な環境変数、フックプラグインは有効化方法、
   設定は設定テンプレートを記載する
2. **マーケットプレイス登録**（公開する場合）: marketplace.json への
   追加方法を示し、説明文・カテゴリ・タグを提案する
3. **サマリーを作る**: 全TODOを完了にし、作成物（プラグイン名と目的・
   コンポーネント数・主要ファイル・構造）と次のステップ（テストの推奨・
   公開・利用に基づく反復改善）を列挙する
4. **改善提案**（任意）: 追加できるコンポーネント・統合の機会・テスト戦略

**成果**: 完成し文書化された、利用・公開可能なプラグイン

## 重要な注意事項

### 全フェーズ共通

- **TodoWrite** で常に進捗を追跡する
- コンポーネント種別に応じて **Skill ツールでスキルを読み込む**
- **専門エージェント**（agent-creator / plugin-validator）と skill-creator プラグインの検証スキルを活用する
- 重要な判断ポイントで**ユーザーの確認**を得る
- **ベストプラクティスを適用する**: スキル description は第三者視点・
  本文は命令形・コマンドは Claude への指示・強いトリガーフレーズ・
  `${CLAUDE_PLUGIN_ROOT}`・progressive disclosure・セキュリティ第一
  （HTTPS・認証情報のハードコード禁止）

### ユーザーの確認を待つポイント

1. Phase 1 の後: プラグインの目的の確認
2. Phase 2 の後: コンポーネント計画の承認
3. Phase 3 の後: 実装への着手
4. Phase 6 の後: 修正するか進むか
5. Phase 7 の後: ドキュメントへ進むか

### フェーズ別に読み込むスキル

- **Phase 2**: plugin-structure
- **Phase 5**: plugin-skills / plugin-commands / plugin-agents /
  plugin-hooks / plugin-mcp / plugin-settings（必要に応じて）
- **Phase 6**: （エージェントが自動的にスキルを使う）

### 品質基準

すべてのコンポーネントが満たすべき基準:
実績あるパターンに従う / 正しい命名規則 / 強いトリガー条件（スキル・
エージェント）/ 動く実例 / 適切なドキュメント / ユーティリティでの検証済み /
Claude Code でのテスト済み

## ワークフローの例

### ユーザーのリクエスト

「データベースマイグレーションを管理するプラグインを作って」

### Phase 1: 発見

マイグレーション管理・スキーマのバージョニングと理解し、
作成・実行・ロールバックが必要なことを確認する。

### Phase 2: コンポーネント計画

スキル1（マイグレーションのベストプラクティス）、
コマンド3（create-migration / run-migrations / rollback）、
エージェント1（migration-validator）、MCP1（データベース接続）。

### Phase 3: 確認の質問

対象DBは？（PostgreSQL, MySQL 等）マイグレーションファイルの形式は？
（SQLかコードベースか）適用前にエージェントで検証するか？
必要なMCPツールは？（query / execute / schema）

### Phase 4〜8: 実装・検証・テスト・ドキュメント

---

**Phase 1: 発見 から開始する**
