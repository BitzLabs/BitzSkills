# ポジショニング（競合代替 → PoD/PoP → ステートメント）

> 差別化の宣言を導出する。競合の詳細調査は範囲外 — 簡易 intake で足りる分だけ。
> 事実が取れないものは `TBD`。

## 競合代替の簡易 intake

顧客（スキルを作る開発者）が skill-creator がなければ取る手段:

| 区分 | 代替 | 効く次元での比較 | 出典 |
|---|---|---|---|
| 現状維持 | **素の SKILL.md 手書き**（ドキュメントを見ながら自力で書く） | 自由度は高いが作法がブレ、検証・効果測定・配置が手作業 | Agent Skills 標準 <https://agentskills.io/specification> |
| 間接 | **汎用スキャフォールド／コードジェネレータ** | 雛形は出るが Agent Skills 仕様準拠・発動精度・効果測定は非対象 | `TBD`（具体ツールは未調査） |
| 間接 | **各自のアドホックなスニペット集・自作テンプレ** | 再利用はできるが検証・自己改善の仕組みがない | `TBD` |
| 直接 | 同種の「スキル開発一式」プラグイン | 実在・シェアとも `TBD`（未調査） |

目的は空白地帯の発見であって機能チェックリストではない。空白地帯は
**「作成の後工程（検証・効果測定・配置・配置後の自己改善）まで一続きで面倒を見る」**部分。

## PoP / PoD

- **PoP（同等性のポイント／過剰投資しない）**
  - Agent Skills 仕様準拠の SKILL.md + フォルダ一式を生成できる。
  - frontmatter 必須項目・命名・自己完結を満たす。
  - → これらは「検討に入るための当たり前」。手書きでも達成可能なので、
    ここで超過しても勝てない。

- **PoD（差別化のポイント／資源を集中する）**
  1. **効果を測る**: skill-tester のスキルあり/なしベースライン比較 + skill-evaluator の
     採点。「効いた気がする」を排除する（他の雛形ツールにない）。
  2. **配置後まで面倒を見る自己改善ループ**: instrumenter→observer→improver が
     実運用の失敗を observations.jsonl に貯め、`.spec/spec-issues/` へ起票して
     人間の裁定に載せる。作りっぱなしにしない。
  3. **全工程のオーケストレーション**: pipeline が「どの工程から始めるか」を案内し、
     10 スキルを 1 つの規律として束ねる。

## ポジショニング・ステートメント（Moore テンプレ・短形式）

> **エージェントスキルを作り・保守する開発者（主要: 個人開発者本人）** のための
> **skill-creator** は、**スキルの品質を勘ではなく規律で担保したいという課題** に対する
> **スキル開発ライフサイクル支援プラグイン** であり、**作成から検証・効果測定・配置・
> 自己改善までを一続きで提供する** ことを届ける。**素の SKILL.md 手書きや汎用
> スキャフォールド** と違い、**効果をあり/なしで測定し、配置後の失敗を観察ログから
> 拾って改善に回す**。

## 規律と反証可能な仮説

- 主張するのは**価値（顧客の成果 = 効いていると分かる／劣化に気づける）**であって
  機能リストではない。PoP（雛形が出せる）の土俵では戦わない。
- 崩壊クリティカルな差別化主張「効果測定 PoD が実際にワークする」は仮説であり、
  assumptions.md（SKC-DSC-006, A-D2）へ転記して検証対象にする。
