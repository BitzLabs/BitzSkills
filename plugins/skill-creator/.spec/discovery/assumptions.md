# 仮説検証ゲート（分類 → 崩壊影響 → テスト+閾値 → Go/No-Go 裁定案）

> 戦略を反証可能な賭けの集合に変え、最も危険な仮説から検証する。これは遡及 discovery
> であり、対象はすでにリリース済み（v0.5.3）で実運用中のプラグイン。したがって
> 「崩壊クリティカル仮説の一部は実運用の稼働実績で部分的に検証済み」という前提で読む。
> **最終的な Go/No-Go 裁定は人間が PR レビューで行う。以下はエビデンス付きの裁定案。**

## 崩壊クリティカル（★★★）仮説の裁定状況

- **A-D1**: 部分検証（主要ペルソナで成立、外部は未検証だが No-Go 要因ではない）。
- **A-D2**: **未検証**。ただし効果測定の仕組み（tester/evaluator）は実装済みで、
  テストは「既存スキルで あり/なし比較を実走する」ことで安価に実施可能。
  事前閾値は `TBD`（要人間定義）。
- **A-F1**: 部分検証。配置ロジックは実装済み、実環境発動確認が既知の残作業
  （MEMORY のリファクタリングプラン進捗と整合）。テスト経路は明確。

## Go / No-Go 裁定案（最終裁定は人間が PR レビューで行う）

**裁定案: Go（条件付き）**

**根拠（エビデンス）**:
1. 本プラグインはすでに v0.5.3 でリリース・実運用され、主要ペルソナ（本人）による
   全 10 スキルの採用が継続している（A-D1 部分検証、A-V2 検証済み前提）。
2. 崩壊クリティカル仮説のうち機械検証系（A-F2/A-F3）は validator / release_check で
   検証済み。残る未検証の A-D2・A-F1 は**いずれもテスト経路が明確で安価に検証可能**
   （既存スキルでの あり/なし比較実走、実環境配置確認）であり、
   sdd-discovery の Go 条件「崩壊クリティカル仮説がすべて検証済み or テスト+閾値定義済み」に
   テスト定義側で接続できる。

**条件（Go 後に追跡する open リスト）**:
- A-D2 の事前閾値を人間が定義し、既存スキル数件で効果測定を実走する（最優先）。
- A-F1 の実環境 `/plugin install` 発動確認を完了する（人間側の既知残作業）。
- A-D1 外部ユーザー需要は `TBD`。OSS 公開後のリサーチで昇格させる。

**No-Go に振れる場合**: 上記 A-D2 の効果測定が「あり/なしで差を示せない」と判明した
場合は PoD 1 が崩れるため、positioning / scope（効果測定の位置づけ）を人間に差し戻す。

> エビデンスが届いたらゲートは再訪する（一度きりの儀式にしない）。docs/ 該当ドラフトへの
> 影響が生じたら sdd-core の再伝播プロトコルに乗せる。
