# ビジョン（Vision Board + PR-FAQ 圧力試験）

> 遡及的 discovery。skill-creator はすでにリリース済み（plugin.json version 0.5.1）の
> 10 スキルからなるプラグインであり、本ファイルは「なぜこれを作ったか・誰のためか・
> 何をやらないか」を事後に明文化するもの。根拠のない数値・調査結果は `TBD` または
> `[proto / 未検証]` と明示する。

## Product Vision Board（Roman Pichler）— 5要素

1. **Vision（究極の目的）**
   エージェントスキルの開発を「思いつきで書く手作業」から「作成→検証→テスト→評価→
   最適化→配置→自己改善が回る規律あるパイプライン」へ変える。スキルの品質を
   属人的な勘ではなく、機械検証と効果測定に載せる。

2. **Target Group（必ずセグメントする）**
   - **主要**: このモノレポでスキルを量産・保守する個人開発者本人（hide）。
     Claude Code / Antigravity 2.0 双方向けにスキルを作り続ける当事者。
   - **二次**: 本プラグインを OSS として導入する将来の外部開発者
     （Agent Skills 標準に沿ってスキルを作りたいが、作法・検証・配置で迷う層）。
   - 「みんな」は対象にしない。プロンプトを書くだけの一般利用者、
     スキルを消費するだけのエンドユーザーは対象外。

3. **Needs（解決する問題／届ける便益）**
   - SKILL.md の frontmatter 仕様・description の発動精度・progressive disclosure
     といった作法が暗黙知で、毎回ブレる。
   - 「スキルが効いているか」を確かめる手段がなく、体感で良し悪しを判断してしまう。
   - 配置先（Claude Code / Antigravity）ごとのパスやインストール手順が煩雑。
   - 一度配置したスキルが劣化・空振りしても気づけず、改善の起点が残らない。
   → 便益: 一貫した作法・効果の可視化（スキルあり/なし比較）・ワンストップ配置・
     観察ログ起点の継続改善。

4. **Product（際立つ少数の差別化要素）**
   - **全工程の分業**: 単一の巨大スキルではなく、責務ごとに 10 スキルへ分割し
     pipeline が統括する。
   - **効果測定を内蔵**: skill-tester がスキルあり/なしのベースライン比較を行い、
     skill-evaluator が採点する（「良くなった気がする」を排除）。
   - **自己改善ループ**: instrumenter→observer→improver が実運用の失敗を
     observations.jsonl に貯め、.spec/spec-issues/ へ起票して人間の裁定に載せる。
   - （機能全リストではない。上記が他のスキル雛形ツールと際立つ点）

5. **Business Goals（開発者／プロジェクトにとっての便益）**
   - このモノレポ（BitzSkills）のスキル生産・保守コストを下げ、品質を平準化する。
   - OSS 公開時の初期採用の入り口（「スキル開発の作法が一式そろっている」）になる。
   - 収益目的の商用プロダクトではない（個人開発者の道具＋OSS 公開）。

**Mission**: スキル開発者に、作法・検証・効果測定・配置・自己改善を一続きで提供する。
**Vision**: エージェントスキルの品質が勘ではなく規律で決まる状態。
**Values**: 機械検証を信じる（自己申告を信用しない）／progressive disclosure（薄く保つ）／
自己完結（スキルはフォルダ単位でコピーできる）。

## Working Backwards — PR-FAQ（圧力試験・要旨）

**すでに出荷された体で書く。**

- **見出し**: 「スキルを、思いつきから規律ある成果物へ — skill-creator」
- **小見出し**: エージェントスキルを作る開発者向けに、作成から自己改善までの
  全工程を 10 スキルで支える。
- **要約**: skill-creator は Agent Skills 標準準拠のスキルを、対話設計→仕様検証→
  効果測定→最適化→両プラットフォーム配置まで一貫して支援するプラグイン。
- **問題（顧客の言葉）**: 「SKILL.md の書き方は分かったが、これが本当に効いているのか、
  仕様に沿っているのか、どう配ればいいのかが毎回あいまいだ」。
- **解決／差別化**: 効果をスキルあり/なしで比較採点し、配置後の失敗を観察ログから
  拾って改善に回す点が、単なる雛形ジェネレータと異なる。
- **引用（責任者）**: [proto / 未検証] — 実ユーザー引用は未収集。

### 内部 FAQ（荷重を受ける部材）

- **市場規模（TAM/SAM/SOM）**: `TBD`（Agent Skills 標準の採用者数は未調査）。
- **競合状況**: 素の SKILL.md 手書き、汎用スキャフォールドツール、`skill-creator` を
  持たない各自のアドホックな手順（詳細は positioning.md）。
- **ユニットエコノミクス**: 非商用のため対象外。コストはトークン消費と保守工数。
- **Go/No-Go 基準（反証可能な形で）**:
  - G1: 主要ユーザー（本人）が、素の手書きより skill-creator 経由の方が
    スキルを速く・仕様準拠で作れると判断できる（現状: 実運用で採用継続中 → 満たす見込み）。
  - G2: skill-tester / evaluator が「スキルあり/なし」で有意な差を測定できる
    （効果測定の仕組みが機能する）。→ 崩壊クリティカル、assumptions.md で扱う。
