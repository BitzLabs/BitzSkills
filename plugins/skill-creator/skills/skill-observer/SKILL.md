# skill-observer

## 目的

スキル実行の直後に「その実行がうまくいったか」をAI自身が観察し、問題があった
場合だけ事実を構造化ログに残す。ここで採取された観察が自己改善ループ
（`skill-improver` によるスキル修正）の唯一の情報源になる。

このスキルの責務は「観察と記録」のみ。ログの分析・スキル修正は
`skill-improver`、観察ステップの組み込みは `skill-instrumenter` が行う。

## 原則

- **観察は本処理を邪魔しない**: 記録に失敗しても、観察対象スキルの本来の
  作業結果は完了として扱う。観察のためにユーザーを待たせる質問もしない。
- **成功は記録しない**: outcome が success なら何も書かない（ノイズ削減）。
  「観察済み・問題なし」と一言添えるだけでよい。
- **事実を書く**: observation には解釈や言い訳ではなく、起きた事実を書く。
- **改善案は具体化する**: suggested_fix は「スキル名・ステップ・変更内容」まで
  特定して書く（例:「skill-tester の Step 3 のパス例を修正」）。抽象論は書かない。

## ワークフロー

### 1. 観察対象の確定

直前に実行したスキルについて以下を特定する。

- スキル名（`name` フィールドの値）
- 実行したワークフローステップの範囲
- ユーザーの当初の依頼と、実際に残した成果

### 2. 自己観察（outcome 判定）

- **success**: スキルの手順どおりに進み、約束した成果物が揃った
  → 記録せず終了。
- **partial**: 完了はしたが、手順の迷い・回避策・手戻り・ユーザーの
  訂正介入があった → Step 3 へ。
- **fail**: 手順どおりに進められなかった、または成果物が揃わなかった
  → Step 3 へ。

判定に迷ったら partial とする（「スキルの記述が原因で迷った」こと自体が
観察に値する事実）。

### 3. 観察ログへの追記

`references/observation-schema.md` のスキーマに従い、観察1件を1行のJSONに
まとめ、**カレントプロジェクト直下**の `evals/observations/observations.jsonl`
に追記する。

- ディレクトリ・ファイルがなければ作成する
- 追記は必ず `>>`（シェルのアペンド）で行い、既存の行を書き換えない
- 追記に失敗した場合は、記録できなかった旨を報告して終了する
  （本処理の完了報告は取り消さない）

### 4. 報告

記録した観察（スキル名・severity・observation の要旨）を1〜2行で報告する。
同じスキルの open な観察が複数件溜まっていることに気づいたら、
`skill-improver` での分析を提案する。

## スキーマの正典

観察ログのフィールド定義・severity 基準・記入例は
`references/observation-schema.md` を参照。`skill-improver` はこの書式を
前提に Ingest するため、書式を勝手に変えないこと。
