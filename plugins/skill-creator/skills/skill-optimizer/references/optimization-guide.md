# スキル最適化ガイド

`skill-optimizer` が改善を行う際の詳細な基準と例。

## descriptionの書き方

descriptionはエージェントがスキルを使うかどうかを判断する**唯一の材料**。
Discovery段階では本文は読まれず、`name` と `description` だけが読み込まれる。

### 必須要素

1. **何をするか** — スキルが行う作業と成果物
2. **いつ使うか** — 発動すべき状況。ユーザーの発言例（「〜して」と言われた場合）
   や具体的なキーワードをそのまま含めると発動精度が上がる
3. **（任意）いつ使わないか** — 隣接するスキルと混同されやすい場合、境界を明記
   する（例: 「新規作成はskill-creatorを使う」）

### 悪い例 → 良い例

悪い例（何をするかしか書いていない・抽象的）:

```yaml
description: PDFを処理するスキル。
```

良い例（作業内容＋トリガー発言＋キーワード）:

```yaml
description: PDFのテキスト抽出・フォーム入力・結合を行う。「PDFからテキストを取り出して」「PDFを結合して」と言われた場合や、フォーム・文書抽出に関する作業のときに使う。
```

### 最適化の手順

1. スキル本文を読み、発動すべき場面を5〜10個列挙する（ユーザーの発言として書く）
2. 発動すべきでない隣接場面も列挙する（誤発動の防止）
3. 両方を判別できるdescriptionを書き、改善前後を並べてユーザーに提示する
4. 適用後、`skill-tester` で発動判定テスト（この発言で使う/使わないの判定）を
   行うことを提案する

## progressive disclosure（本文分離）の目安

| 状態 | 対応 |
| --- | --- |
| 本文 < 500行 かつ 読み切れる分量 | 分離不要 |
| 本文 ≥ 500行 または 5000トークン超 | 詳細を `references/` へ分離する |
| 特定の状況でしか使わない長い手順・仕様表・例の集まり | 行数に関わらず分離候補 |

分離のルール:

- 本文には「何があるか」と「いつ読むべきか」を1〜2行で残す
  （例: 「frontmatterの全フィールドは `references/spec.md` を参照」）
- 参照は SKILL.md からの相対パスで、できるだけ1階層以内にとどめる
- 毎回必要な指示は本文に残す。分離するのは「必要なときだけ読む」内容のみ

## トークン効率の改善対象

- 同じ指示の言い換え・繰り返し
- エージェントの挙動を変えない前置き・修飾（「〜することが重要です」等）
- 本文に埋め込まれた長大なサンプル出力（references/かassets/へ）
- 箇条書きで足りる内容の冗長な段落

削ってはいけないもの: 制約条件、安全確認の手順、判断基準、良い例/悪い例の対比。

## バージョンbumpの判定

変更を適用したら `metadata.version`（semver）を上げる。規約の正式な定義は
skill-creator の spec.md（metadata運用規約）にあり、要点は次の通り。

| bump | このスキルの作業でいうと |
| --- | --- |
| patch | (c) トークン効率の圧縮など、挙動を変えない修正 |
| minor | (a) description最適化、(b) references分離、(d) decision tree導入など、責務の範囲内の改善 |
| major | 責務・トリガー条件そのものの変更（ユーザーの明示的な承諾が必要な変更） |

`metadata.updated` も忘れずに当日日付へ更新する。`version` フィールドが
そもそも無い旧形式のスキルだった場合は、`"0.1.0"` を付与してから作業する。

## decision treeへの構造化

条件分岐が3つ以上あり、それぞれで手順が変わる場合は、条件を先頭に集めた
決定木セクションを設ける。

```markdown
## どのフローを使うか

- 新規作成 → 「作成フロー」へ
- 既存の修正で、仕様違反の指摘あり → 「修正フロー」へ
- 既存の修正で、動作はするが品質が低い → 「改善フロー」へ
```

本文のあちこちに「ただし〜の場合は…」が散らばっている状態が、構造化のサイン。
