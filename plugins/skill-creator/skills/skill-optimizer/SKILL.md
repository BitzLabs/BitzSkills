# skill-optimizer

## 目的

既存スキルの品質を改善する。descriptionの良い例/悪い例、分離判断の目安などの
詳細は `references/optimization-guide.md` を参照。

このスキルの責務は「既存スキルの改善」のみ。新規作成は `skill-creator`、
検査だけなら `skill-validator` を使う。

## ワークフロー

### 1. 対象と課題の特定

- 対象スキルフォルダのパスを確認する
- 何を改善したいかを確認する。ユーザーが明確に指定しない場合は、SKILL.mdを
  読んで以下の4観点で課題を洗い出し、見つかった課題を提示して優先順位を
  ユーザーと合意する

### 2. 改善の実施（4観点）

**(a) description最適化** — スキルが発動すべき場面のユーザー発言・キーワードを
列挙し、それらを含む形でdescriptionを書き直す。「何をするか」と「いつ使うか」の
両方を必ず含める。**改善前後を並べて提示し、承諾を得てから適用する。**

**(b) progressive disclosure** — 本文が長い（目安: 500行/5000トークン超）場合、
詳細な手順・参考資料・大量の例を `references/` 配下のファイルに分離し、本文
からは相対パスで参照させる。本文には「いつそのファイルを読むべきか」を残す。

**(c) トークン効率** — 同じ内容の繰り返し、回りくどい言い回し、エージェントの
挙動を変えない飾りの文章を削る。意味を変えないことを差分で確認する。

**(d) 構造改善** — 状況によって取るべき手順が変わるのに条件が本文に散らばって
いる場合、決定木（decision tree）形式のセクションに整理する。

### 3. バージョンbump

変更を適用したら、同じ作業の中で `metadata.version` を上げ、`metadata.updated`
を当日日付にする。bumpの粒度（patch/minor/major）の判定基準は
`references/optimization-guide.md` の「バージョンbumpの判定」を参照。

### 4. 検証と次工程の提案

- 変更後、改善前後の差分の要約（何をどう変えたか・なぜか・新しいversion）を
  報告する
- `skill-validator` での再検証を提案する
- descriptionを変更した場合は、`skill-tester` で発動判定のテストを行うことを
  提案する

## 注意

- **意味を変えない**: 最適化はスキルの挙動を保ったまま行う。責務の追加・削除が
  必要だと気づいた場合は、勝手に変えず、ユーザーに提案する
- **過学習を避ける**: 特定のテストケースだけを通すための場当たり的な文言追加
  ではなく、一般化された表現で改善する
