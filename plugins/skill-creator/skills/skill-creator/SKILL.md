# skill-creator

## 目的

Agent Skillsオープン標準に準拠した新しいスキルを、仕様に合ったフォルダ構成・
frontmatter・本文で作成する。frontmatterの細かい制約の正式な一覧は
`references/spec.md` にまとめてあるので、迷ったらそちらを参照する。

このスキルの責務は「新規スキルの設計と雛形作成」のみ。作成後の検証・テスト・
配置は後述の関連スキルに委ねる。

## ワークフロー

### 1. ヒアリング

ユーザーに次を確認する。推測で埋めず、不明な点は必ず質問する。

- **スキル名**: 英小文字とハイフンのみの名前（例: `pdf-processing`）。フォルダ名と
  一致させる必要があるので、この時点で確定させる。
- **トリガー**: どんな発言・状況でこのスキルを使ってほしいか（そのまま `description`
  の材料になる）。
- **単一責務か**: 複数のことをやろうとしていないか確認する。2つ以上の異なる作業を
  1つのスキルに詰め込もうとしている場合は、スキルを分割するよう提案する。
- **複雑度**: 指示だけで完結する単純なものか、`scripts/`（実行コード）や
  `references/`（詳細ドキュメント）や `assets/`（テンプレート等）が必要になりそうか。

### 2. ディレクトリ作成

スキルフォルダ `<skill-name>/` を作成する。置き場所はリポジトリの構成に従う
（BitzSkills のようなプラグインモノレポでは `plugins/<plugin名>/skills/<skill-name>/`。
どのプラグインに属するかをユーザーに確認する）。複雑度に応じて以下も作成する。

- `scripts/` — 実行コード（Python/Bash等）を置く場合
- `references/` — SKILL.md本文に収まらない詳細ドキュメントを置く場合
- `assets/` — テンプレートや画像などの静的ファイルを置く場合

### 3. `SKILL.md` の作成

frontmatterのルール（詳細は `references/spec.md`）:

- `name`: 1〜64文字、英小文字・数字・ハイフンのみ、先頭末尾ハイフン禁止、
  連続ハイフン禁止、**親フォルダ名と一致させる**
- `description`: 1〜1024文字。「何をするか」と「いつ使うか」の両方を、
  ヒアリングで聞いたトリガーの言葉を活かして具体的に書く
- `metadata`: ライフサイクル管理用のメタ情報を必ず書く（規約は
  `references/spec.md` の「metadata運用規約」）
  - `version`: `"0.1.0"`（新規作成時の初期値）
  - `author`: 作成者名（ユーザーに確認する）
  - `created` / `updated`: 作成日（両方とも当日の `"YYYY-MM-DD"`）

本文は簡潔に保つ（progressive disclosureの考え方。目安は500行未満）。
長くなりそうな内容（詳細な手順、参考資料、大量の例）は本文に書かず
`references/` 配下のファイルに逃がし、本文からは相対パスで参照する。

### 4. 完了報告と次工程の提案

作成したファイルのパス一覧を伝えたうえで、次の工程を提案する。

- **検証**: `skill-validator` で仕様準拠（命名規則・description・行数など）を
  チェックする
- **テスト**: `skill-tester` でテストケースを設計・実行し、`skill-evaluator` で
  結果を評価する
- **配置**: 実際の環境（ワークスペース/グローバル）で使えるようにするには
  `skill-packager` で配置する

## スキル設計原則

- **単一責務**: スキルは1つのことだけを完璧にこなす。2つのことをしているなら
  2つのスキルに分ける。
- **観測可能な成果**: チャットへのテキスト回答だけでなく、ファイル・コミット・
  図など具体的な成果物を残す指示を優先する。
- **不明点は止めて聞く**: 必要なファイル・権限・情報が足りない場合は、推測で
  進めず、ユーザーに確認する。
- **安全確認**: コード実行・ファイル削除・ネットワークアクセスを伴うスキルには、
  「安全確認・検証」の手順を明記する。

## 詳細仕様

`references/spec.md` を参照。
