# 採点基準とレポート雛形

`skill-evaluator` が使用する採点基準の詳細と、`report.md` の雛形。
テストケース・実行ログの書式は `skill-tester` の `references/test-design.md` で
定義されたものを前提とする。

## アサーションの判定

| 判定 | 基準 |
| --- | --- |
| 合格 | 成果物または実行ログから、アサーションが真であることを直接確認できた |
| 不合格 | 成果物または実行ログから、アサーションが偽であることを直接確認できた |
| 判定不能 | 記録が不足していて真偽を確認できない（採点者の推測で埋めない） |

原則:

- **証拠主義**: 判定には必ず根拠（ファイルパス・該当箇所・ログの記述）を添える。
  根拠を示せない判定は「判定不能」にする
- **アサーションの文言に忠実に**: 実行結果が「惜しい」場合でも、アサーションの
  文言を満たしていなければ不合格。アサーション自体が不適切だと感じたら、判定は
  文言通りに行い、別途「ケースの改善提案」として書く

## ケースの合否

- ケース内の全アサーションが合格 → ケース合格
- 1つでも不合格 → ケース不合格
- 不合格はないが判定不能がある → 「要再実行」

## 定性評価の観点（比較所見用）

スキルあり/ベースラインの比較では、アサーションの合否に加えて以下を見る。

- **手順の質**: 確認すべきことを確認したか、危険な操作の前に承諾を得たか
- **成果物の質**: 構成・網羅性・仕様準拠の度合い
- **一貫性**: 同種の入力に対して安定した挙動が期待できそうか
- **効率**: 不要な回り道・冗長な出力がないか

## report.md の雛形

```markdown
# <skill-name> 評価レポート

- **評価日**: YYYY-MM-DD
- **対象**: evals/<skill-name>/（cases.md vN, runs 1〜M）

## サマリー

| 指標 | 結果 |
| --- | --- |
| ケース合格率 | X/Y |
| アサーション合格率 | X/Y |
| 判定不能 | N件 |

<全体所見を2〜3文で>

## ケース別結果

### TC-01: <名前> — 合格 | 不合格 | 要再実行
- ✅ <アサーション>: <根拠>
- ❌ <アサーション>: <根拠>
- **スキルあり/なし比較**: <所見>

## 改善提案（優先度順）

1. **[高]** <提案>: <どのケースのどの結果に基づくか。一般化された修正か>
2. **[中]** ...

## テストケース自体への提案

<アサーションの不備・不足しているケースがあれば>
```

## 過学習の警告

改善提案を書くとき、次のパターンは避ける。

- テストケースの入力文をそのままdescriptionに書き足す提案
- 特定ケースの期待成果物名をSKILL.md本文にハードコードする提案

代わりに「その入力が属するカテゴリ全体」に効く修正を提案する。判断に迷う
場合は、提案に「過学習リスクあり」と注記してユーザーに委ねる。
