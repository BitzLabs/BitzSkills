# skill-evaluator

## 目的

`evals/<skill-name>/` に保存されたテスト実行結果を採点し、
`evals/<skill-name>/report.md` として評価レポートを作成する。
採点基準とレポートの雛形は `references/grading-rubric.md` を参照。

このスキルの責務は「採点とレポート作成」のみ。テストの実行は `skill-tester`、
改善の実施は `skill-optimizer` に委ねる。

## ワークフロー

### 1. 材料の確認

`evals/<skill-name>/` の `cases.md` と `runs/` を読む。次の場合は採点を始めず
ユーザーに確認する。

- `cases.md` または `runs/` が存在しない → 先に `skill-tester` の実行を提案
- 一部のケースが未実行 → 部分評価でよいか確認

### 2. ケース毎の採点

各ケースについて、`runs/` の記録と成果物をアサーションに照らして判定する。

- アサーション単位で **合格 / 不合格 / 判定不能** を付け、必ず根拠
  （成果物のどこを見たか、ログのどの記述か）を添える
- 判定不能（記録不足で真偽が確認できない）は不合格と区別し、`skill-tester`
  での再実行を提案する

### 3. スキルあり/なしの比較

同一ケースの「スキルあり」と「ベースライン」の成果を比べ、スキルが挙動を
どう変えたかを所見として書く。差がない場合は「このケースではスキルの効果が
観測できない」と正直に書く（スキルが無意味なのか、ケースが弱いのかも考察する）。

### 4. レポート作成

`references/grading-rubric.md` の雛形に従い、`evals/<skill-name>/report.md` を
作成する。含める内容:

- 合格率サマリー（ケース数・アサーション数ベース）
- ケース毎の合否と根拠
- スキルあり/なしの比較所見
- 改善提案（優先度付き）

改善提案は**一般化を優先**する。特定のテストケースを通すためだけの文言追加は
提案せず、「同種の入力全般で挙動が良くなる修正」を提案する。

### 5. 次工程の提案

- 不合格・改善点があった場合: `skill-optimizer` での改善と、改善後の
  `skill-tester` 再実行を提案する
- 全て合格の場合: `skill-packager` での配置を提案する
