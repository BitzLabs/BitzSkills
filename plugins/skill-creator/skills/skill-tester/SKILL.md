# skill-tester

## 目的

対象スキルのdescription・本文からテストケースを設計し、実行結果を
`evals/<skill-name>/` に保存する。テストケースの書式と設計観点の詳細は
`references/test-design.md` を参照。

このスキルの責務は「テストの設計と実行」まで。結果の採点とレポート作成は
`skill-evaluator` に委ねる。

## ワークフロー

### 1. 対象スキルの理解

対象スキルの `SKILL.md` を読み、次を列挙する。

- このスキルが発動すべきユーザー発言・状況（トリガー場面）
- スキルが約束している成果物・挙動
- エッジケースになりそうな入力

### 2. テストケースの設計

`references/test-design.md` の書式に従い、次の3種類からケースを設計する。

- **正常系**: 典型的なトリガー発言に対して、期待する成果物・挙動が得られるか
- **エッジケース**: 曖昧な指定・情報不足・境界値など
- **発動判定**: 「この発言でスキルを使うべき/使うべきでない」の判別
  （descriptionの品質テスト）

各ケースには **入力プロンプト** と **合格基準（アサーション）** を必ず付ける。
アサーションは実行後に真偽を判定できる具体的な文にする。

設計したケースをユーザーに提示し、追加・修正がないか確認してから
`evals/<skill-name>/cases.md` に保存する（`evals/` はこのリポジトリの
ルート直下。スキルフォルダ内には置かない）。

### 3. テストの実行

各ケースを2通りで実行し、比較材料を作る。

- **スキルあり**: 対象スキルのSKILL.md本文の指示に忠実に従って、入力プロンプト
  のタスクを遂行する
- **ベースライン（スキルなし）**: スキルの指示を参照せず、同じ入力プロンプトの
  タスクを遂行する

実行ごとに `evals/<skill-name>/runs/<n>/`（`<n>` は連番）を作成し、以下を保存する。

- `log.md`: ケースID、実行日時、スキルあり/なし、実行時の判断・手順の記録
- 生成した成果物（ファイルを作るスキルの場合、このディレクトリ配下に出力する）

コード実行・ファイル削除・ネットワークアクセスを伴うテストは、実行前に内容を
ユーザーに提示して承諾を得る。テストの副作用は `evals/` と一時ディレクトリの
外に出さない。

### 4. 次工程の提案

全ケースの実行が終わったら、保存先のパス一覧を報告し、`skill-evaluator` で
採点・レポート作成を行うことを提案する。
