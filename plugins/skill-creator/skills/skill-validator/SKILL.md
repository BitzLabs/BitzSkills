# skill-validator

## 目的

対象スキルフォルダを受け取り、Agent Skills仕様への準拠を機械的に検査して、
✅/❌付きのチェックリストと修正提案を報告する。検査項目の正式な一覧と判定基準は
`references/checklist.md` を参照。

このスキルの責務は「検査と報告」のみ。修正の実施は行わない（修正が必要な場合は
`skill-optimizer` の利用を提案する）。

## ワークフロー

### 1. 対象の特定

検査対象のスキルフォルダのパスをユーザーに確認する。指定がなく候補が複数ある
場合は、どのスキルを検査するか質問する。「全部」と言われた場合は1スキルずつ
順に検査する。

### 2. 検査の実施

対象の `SKILL.md` と同フォルダ内のファイルを読み、`references/checklist.md` の
全項目を判定する。主な検査項目:

- frontmatterがYAMLとして正しくパースできるか
- `name` の命名規則（1〜64文字、英小文字・数字・ハイフンのみ、先頭末尾/連続
  ハイフン禁止）と親フォルダ名との一致
- `description` の存在、1024文字以内、「何をするか」と「いつ使うか」の両方を
  含むか
- 本文が500行未満か
- 本文中の相対パス参照（`references/` 等）が実在するファイルを指しているか
- 単一責務か（2つ以上の異なる作業を扱っていないか）
- `metadata` のライフサイクル情報（`version` がsemver形式、`author`、
  `created`/`updated` が日付形式、ライブラリ内に `installed-*` が混入して
  いないこと）

### 3. 報告

スキルごとに以下の形式で報告する。

```
## skill-name の検査結果

- ✅ name: 命名規則を満たし、フォルダ名と一致
- ❌ description: 「いつ使うか」が書かれていない
- ✅ 本文行数: 87行 (< 500)
...

### 修正提案
- description に「〜のときに使う」というトリガー条件を追記する
```

判定に迷う項目（単一責務かどうか等）は、❌と断定せず ⚠️ として根拠と併せて
提示し、ユーザーの判断を仰ぐ。

### 4. 次工程の提案

- ❌/⚠️ があった場合: `skill-optimizer` での改善を提案する
- 全項目 ✅ の場合: `skill-tester` でのテスト、または `skill-packager` での
  配置を提案する
