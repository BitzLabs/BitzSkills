# ブラウンフィールド評価（MMI + DDD 12基準）

既存コードベースの成熟度を2つの独立した枠組みで採点し、統合改善計画に落とす。前提として現状分析（モジュール一覧・ドメイン対応表）を `.spec/design/analysis/` に揃えておく。2評価は独立なので**並列実行できる**（サブエージェントが使えるなら1メッセージで2体起動）。採点は assets/evaluation-scorecard.md の書式で記録する。

## 共通の採点尺度（1〜5）

5=模範的（問題なし・ベストプラクティス準拠）/ 4=良好（軽微な改善のみ）/ 3=許容（いくつか問題あり）/ 2=懸念（重大な問題あり）/ 1=致命的（根本的な問題）

## 評価1: MMI（Modularity Maturity Index）— モジュールごとに4軸

| 軸 | 重み | 見るもの | score 5 / 1 のアンカー |
|---|---|---|---|
| 凝集度 Cohesion | 0.30 | モジュール内の機能的まとまり | 5: 単一の明確な責務 / 1: god class・ユーティリティごみ箱 |
| 結合度 Coupling | 0.30 | モジュール間依存の少なさ（低いほど高得点） | 5: 最小依存・明確な IF・循環なし / 1: 循環依存・単独変更不能 |
| 独立性 Independence | 0.20 | 単独でデプロイ・テストできるか | 5: 完全に独立してデプロイ/テスト/バージョン可 / 1: モノリシックに絡み単独不能（DB/ランタイム共有は 2） |
| 再利用性 Reusability | 0.20 | 別文脈で使い回せるか | 5: 文脈非依存で再利用可能 / 1: 特定文脈に癒着 |

```
MMI = (0.3×凝集 + 0.3×結合 + 0.2×独立 + 0.2×再利用) / 5 × 100
```

| MMI | 判定 |
|---|---|
| 80–100 | 成熟 — マイクロサービス移行可能 |
| 60–80 | 中程度 — 部分リファクタリング後に移行可能 |
| 40–60 | 要改善 — 大規模リファクタリングが必要 |
| 0–40 | 未成熟 — 根本的な再設計が必要 |

各モジュール×各軸に score と1〜2文の根拠を記録する。

## 評価2: DDD 準拠 — 3層12基準

| 層（重み） | # | 基準 | 見るもの |
|---|---|---|---|
| 戦略設計（0.30） | 1 | ユビキタス言語 | ドメイン用語の一貫性、コードとの対応 |
| | 2 | 境界づけられたコンテキスト | 境界の明確さ、責務分離 |
| | 3 | サブドメイン分類 | Core/Supporting/Generic の識別と投資配分 |
| 戦術設計（0.45） | 4 | Value Object | 不変性・等価性・ドメインルールの内包 |
| | 5 | Entity | 識別子・ライフサイクル管理 |
| | 6 | Aggregate | トランザクション境界・整合性保証 |
| | 7 | Repository | 永続化の抽象化・コレクション意味論 |
| | 8 | Domain Service | ステートレス・複数集約の調停 |
| | 9 | Domain Event | イベント駆動・状態変化の通知 |
| アーキテクチャ（0.25） | 10 | レイヤリング | 層構造の明確さ・依存方向 |
| | 11 | 依存方向 | 内向き依存・DIP の適用 |
| | 12 | Ports & Adapters | 外部接続の抽象化 |

```
DDD スコア = (0.30×戦略平均 + 0.45×戦術平均 + 0.25×アーキ平均) / 5 × 100
```

80 以上=強い準拠 / 60–80=概ね準拠 / 40–60=弱い準拠（大幅リファクタリング要）/ 40 未満=非準拠。基準ごとに score と根拠を記録する。

## 統合改善計画

1. **両評価で低スコアの領域を最優先の改善対象にする**（例: MMI 結合度が低い × DDD Aggregate が低い → 集約境界の再設計が最優先）
2. 各改善項目を**ビジネス影響 × 技術的実現性**の2軸でランク付けする
3. **quick wins（短期・小工数で効く）**と**構造改善（中長期）**に分類する
4. 計画を人間に提示し、承認された方針を再設計（グリーンフィールド手順 2〜4）の入力にする。恒久的な裁定は ADR ドラフトへ

評価の採点表・根拠は `.spec/design/evaluation/` に残す。docs/ に書くのは再設計の結論（proposed ドラフト）だけ。
