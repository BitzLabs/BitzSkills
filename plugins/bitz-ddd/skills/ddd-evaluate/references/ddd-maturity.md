# DDD 成熟度評価（DDD Maturity Evaluation）

本ドキュメントは、対象システムの Domain-Driven Design（ドメイン駆動設計、以下 DDD）原則への準拠度を定量的に評価するための基準と手順を定義します。DDD の成熟度を測ることで、システムの保守性やビジネスとの整合性を評価し、アーキテクチャの改善点を浮き彫りにします。

## 評価観点と重み付け

DDD の評価は、ソフトウェア設計の根幹を成す以下の3つのレイヤーと、それらに属する合計12の基準（Criteria）で行います。

### Strategic Design（戦略設計） - 全体の30%
ビジネスの要件とソフトウェアのモデルを一致させるための高度な設計アプローチです。
1. **Ubiquitous Language（ユビキタス言語）**: 
   - ビジネスサイドと開発チームの間で、ドメイン用語が一貫して使用されているか。
   - 同じ概念が、ドキュメントからコードの変数名に至るまで同じ名前で呼ばれているか。
2. **Bounded Context（境界づけられたコンテキスト）**: 
   - モデルの適用範囲（コンテキストの境界）が明確に定義され、責任の分離が明示的か。
   - 異なるコンテキスト間の関係性（コンテキストマッピング）が適切に文書化されているか。
3. **Subdomain Classification（サブドメイン分類）**: 
   - ビジネス領域が Core（中核） / Supporting（支援） / Generic（汎用）な Subdomain（サブドメイン）に正しく特定されているか。
   - 重要度に応じたリソースや投資の割り当てが行われているか。

### Tactical Design（戦術設計） - 全体の45%
ドメインモデルを具体的なコードとして実装するための詳細なパターン群です。
4. **Value Objects（値オブジェクト）**: 
   - Value Objects は不変（Immutable）として実装されているか。
   - ドメインのルールや振る舞いをカプセル化し、等価性が属性の値ベースで定義されているか。
5. **Entities（エンティティ）**: 
   - Entities はライフサイクル全体を通じて安定した一意の識別子（ID）を持っているか。
   - 生成から破棄までのライフサイクル管理が明示的に行われているか。
6. **Aggregates（集約）**: 
   - トランザクションの境界が Aggregate Root（集約ルート）によって正しく定義されているか。
   - データの一貫性の境界（Invariants）が集約の内部で厳密に強制されているか。
7. **Repositories（リポジトリ）**: 
   - Repositories はデータ永続化の詳細を抽象化し、ドメイン層をインフラ技術から保護しているか。
   - データベースのクエリビルダーを直接露出させず、インメモリのコレクションセマンティクスを提供しているか。
8. **Domain Services（ドメインサービス）**: 
   - 複数の Aggregates にまたがるステートレスなドメインロジックが Domain Services に配置されているか。
   - ユースケースを調整するだけのアプリケーションサービスと明確に区別されているか。
9. **Domain Events（ドメインイベント）**: 
   - ドメイン内で発生した重要な状態の変化が Domain Events として発行されているか。
   - 異なる Bounded Context 間の通信に、疎結合なイベント駆動設計が使用されているか。

### Architecture（アーキテクチャ） - 全体の25%
システム全体の構造的完全性と、ドメイン層の保護を評価します。
10. **Layering（レイヤリング）**: 
    - レイヤー構造（Domain / Application / Infrastructure / Presentation）が明確であり、プロジェクト全体で一貫して適用されているか。
11. **Dependency Direction（依存の方向性）**: 
    - 依存関係が常に内側（ドメイン層の方向）を向いているか。
    - 依存関係逆転の原則（Dependency Inversion Principle）が適用され、ドメインがインフラに依存していないか。
12. **Ports & Adapters（ポートとアダプター）**: 
    - 外部システムやデータベースとの接続がインターフェース（Ports）で抽象化されているか。
    - 具象実装（Adapters）がインフラストラクチャ層に隔離されているか。

## 評点基準（レベル定義）

各基準は 1〜5 の5段階で評価します。

- **5 (Exemplary)**: 模範的。対象の概念が完全に適用されており、違反や例外が見られない状態。
- **4 (Good)**: 良好。概ね適切に適用されているが、軽微な技術的負債やわずかな改善の余地がある状態。
- **3 (Acceptable)**: 許容範囲。原則は適用されているものの、境界が曖昧であったり、ルールが一部不完全である状態。
- **2 (Concerning)**: 懸念あり。複数の責任の混在、不適切な依存関係、またはドメインロジックの漏出が目立つ状態。
- **1 (Critical)**: 致命的。DDD の原則が全く適用されておらず、単なるデータ構造の操作（トランザクションスクリプトなど）やアンチパターンに陥っている状態。

### 全体スコアと成熟度レベルの算出

各レイヤーの基準スコアの平均点に重みを掛け合わせ、100点満点の全体スコアを算出します。
`DDD Score = (Strategic層の平均 × 0.30 + Tactical層の平均 × 0.45 + Architecture層の平均 × 0.25) / 5 × 100`

算出したスコアに基づき、以下の成熟度レベルを判定します。
- **80-100**: Strong DDD alignment（DDD への強力な準拠）
- **60-80**: Moderate alignment（中程度の準拠、対象を絞った改善が必要）
- **40-60**: Weak alignment（弱い準拠、大規模なリファクタリングが必要）
- **0-40**: Minimal DDD adoption（DDD の採用が最小限、根本的な設計見直しが必要）

## 評価手順

1. **入力情報の収集**: 
   対象システムの要件定義や既存の分析ドキュメントを一式読み込み、システムの全体像を把握します。
2. **並列レイヤー評価**: 
   Strategic Design、Tactical Design、Architecture の各レイヤーについて独立して評価を実施します。各基準（1〜12）を1〜5のスケールで採点し、その評価に至った理由（Rationale）と具体的な発見事項（Findings）を記録します。
3. **スコア計算と総括**: 
   各レイヤーの評価結果を統合して全体スコアを計算し、成熟度レベルを判定します。評価結果から、アーキテクチャの改善に向けた優先度の高いアクションアイテムを導出します。

## 判定結果の記録フォーマット

評価結果は `.spec/reviews/` 配下（bitz-sdd の書式に準拠）にMarkdown形式で保存します。
生成するレポートには以下を含める必要があります：
- 全体スコア（100点満点）および成熟度レベル
- レイヤーごとのスコアと、各基準の点数・評価理由
- コードベース全体から得られた具体的な発見事項
- 実行可能で優先度の高い改善案（トップ3〜5個程度）
