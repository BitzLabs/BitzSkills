# ドメインモデリング（DDD 戦略設計 + 2パス導出）

docs/02-design/domain-model.md の proposed ドラフトを作るための手法。**ドメインをモデル化する。DB スキーマをモデル化しない**（ストレージはモデルに従わせる）。

## 構成要素（戦術）

- **Entity** — 状態が変わっても持続する同一性（キー）を持つ。永続化され・識別され・複数の操作を受ける概念だけをエンティティに昇格させる
- **Value Object** — 属性で定義され、同一性なし・不変（Money, Address, DateRange 等）。過剰正規化を避けるため、迷ったら Value Object を選ぶ
- **Aggregate** — 1つの整合性単位として扱うエンティティ/VO の集まり。**集約ルートが唯一の入口**で、不変条件は集約境界の内側で守る
- **Command → Aggregate → Event** — コマンドが変更を要求し、集約が不変条件を守って適用し、イベントが事実を記録する。この連鎖がエンティティ発見を駆動する

**トランザクション境界 = 集約境界。** 集約は小さく保ち、他の集約へは ID で参照する（オブジェクトを直接抱き込まない）。

## 2パス導出

**2パスは必須**。1パス目の見落としを機械的に拾うのが2パス目。

1. **Pass 1（明示的）** — 上流成果物（ドメインストーリー・ペルソナ/ジャーニー・UI 案・機能一覧）から: 画面/概念 → エンティティ、フォーム項目 → 属性、操作の結果 → 状態/イベント
2. **Pass 2（暗黙的）** — 機能×エンティティの **CRUD マトリクス**を作り、多対多の中間概念・履歴・監査・状態機械を洗い出す。**追加した各要素に「なぜ必要か（機能上の関係）」の根拠を必ず記録する**

停止条件: 明示的概念をすべて捕捉し、CRUD マトリクスを回し、関係と多重度を定義し、ER 図（mermaid `erDiagram`）が描けたら完了。

## 戦略設計（コンテキスト分割）

- **Ubiquitous Language（ユビキタス言語）** — コンテキストごとに1つの正確な共有語彙。全用語を docs/01-context/glossary.md と整合させる（新語は glossary への proposed 追記）
- **Bounded Context（境界づけられたコンテキスト）** — モデルと言語が一貫する明示的境界。同じ語が別コンテキストで別の意味を持つのは正常
- **サブドメイン分類と投資判断**:
  - **Core** — 差別化の源泉。**ここに投資する**（買わずに作る）
  - **Supporting** — 必要だが差別化しない。実利的に作る
  - **Generic** — 解決済みの問題（認証・課金・通知）。**買う/採用する**。過剰設計しない
- **コンテキストマップ** — コンテキスト間の関係を型で明示する: Partnership / Customer-Supplier / Conformist / **Anticorruption Layer** / **Open Host Service / Published Language** / Shared Kernel。コンテキスト間は疎結合を目指し、Core は ACL で守る
- **境界のサイズは将来の機能を吸収できるように** — 現在の画面ではなくビジネスケイパビリティに沿って引く
- **整合性ヒント** — 各コンテキストに `Strong` / `Eventual` / `TBD` を1行の理由つきで付ける（不変条件を持つ金銭・在庫・予約 → Strong、読み取りモデル・分析・通知 → Eventual）。これは**ヒント**であり、ACID/Saga の確定判断は architecture ステップで行う

## 規律

- 便利さのために貧血モデルへ逃げない。ただし要件にない振る舞いも発明しない
- 各集約・エンティティ境界の**理由を記録する**（変更再伝播のときに再検討できるように）
- 曖昧な境界は `TBD` として明示し、Open Question に残す（勝手に埋めない）
