# ペルソナとジョブ（JTBD → ペルソナカード → 主要ペルソナ）

> 遡及 discovery。実ユーザー調査は未実施。作者本人（1名）以外の感情・生の声は
> すべて **[proto / 未検証]**。人口統計・台詞は捏造しない。

## Jobs-to-be-Done（ジョブストーリー）

- **J1（機能的・主）**: 複雑なドメインの新機能を SDD で設計するとき、軽量デフォルト設計では概念を取りこぼしそうなので、
  ドメインを言葉から正しくモデル化して要件派生の根拠にしたい。
  → 「複雑ドメインの設計に着手した**とき**、sdd-design 単体では境界が恣意的になるので、
  ドメインストーリーからモデルを2パスで導出**したい**、暗黙概念を取りこぼさず要件に繋げられる**ように**」。
- **J2（機能的）**: 既存コードをリファクタリングする前に、現状の DDD 成熟度とモジュール性を客観的に測って
  改善の優先順位を決めたい。
  → 「レガシーコードの再設計を判断する**とき**、勘ではなく基準で現状を測りたい**とき**、DDD 12基準と MMI で採点し
  改善トップ3〜5を出せる**ように**」。
- **J3（感情的）**: DDD の教科書を毎回思い出す不安なしに、正しい作法（集約境界＝トランザクション境界等）を
  エージェントに担保してほしい（安心して設計したい）。
- **J4（社会的）**: OSS として公開したとき、成果物が SDD ライフサイクルに綺麗に流れる「規律ある設計」として
  見られたい（我流でない設計プロセスの証跡）。

## ペルソナカード

### ペルソナ A — 「ドッグフーディングする個人開発者」（主要ペルソナ）
- **1行要約**: BitzSkills 自身を BitzSDD で開発する作者（hide）。1人で設計〜実装〜検証を回す。
- **文脈と行動**: モノレポで複数プラグインを SDD 準拠で開発。設計工程で複雑ドメインに当たると DDD 手法を呼ぶ。
  トークン/手間の節約に敏感（司令塔＋委譲運用）。
- **Jobs**: J1, J2, J3, J4。
- **Pains**: 軽量設計だけでは複雑ドメインで境界が曖昧／DDD 作法の想起コスト／設計成果物が下流と繋がらないと二度手間。
- **Gains**: 2パス導出で暗黙概念を機械的に洗い出せる／成果物が `.spec/design/` 契約で下流に直結／1人でも規律を維持。
- **生の声**: 「複雑ドメインのとき軽量設計だと集約境界が勘になる」`[proto / 未検証：作者の設計意図に基づく推定]`。

### ペルソナ B — 「bitz-sdd 導入済みの OSS 利用者」（二次ペルソナ）
- **1行要約**: BitzSDD を採用し、DDD の設計規律を求めて bitz-ddd を追加インストールする外部開発者。
- **文脈と行動**: 個人〜小規模チーム。DDD をやりたいが本格ワークショップは重い。`[proto / 未検証]`。
- **Jobs**: J1, J3（J2 は既存コード保有者のみ）。
- **Pains**: DDD 学習コスト／既存 SDD フローに手法を組み込む手間。`[proto / 未検証]`。
- **Gains**: 併用前提で契約が揃っているため導入が軽い（想定）。`[proto / 未検証]`。
- **生の声**: `TBD`（外部ユーザー調査なし）。

### ペルソナ C — 「ブラウンフィールド改善の担当開発者」（served ペルソナ）
- **1行要約**: レガシーコードの再設計判断で ddd-evaluate（DDD/MMI 採点）を使う開発者。A/B と重なりうる。
- **Jobs**: J2。
- **Pains**: 現状評価が主観的で改善順が決まらない。`[proto / 未検証]`。
- **生の声**: `TBD`。

### アンチペルソナ — 「多人数イベントストーミング運用チーム」
- 付箋・リアルタイム協調の対面ワークショップを主軸にするチーム。本プラグイン（AI 主導・軽量導出）は不適合。
  スコープ（DDD-DSC-003）の Won't に対応。

## 主要ペルソナの特定

**主要ペルソナ = A（ドッグフーディングする個人開発者）**。下流はまず A に最適化する。
B/C は A と契約・成果物を共有するため、A 向けの最適化がそのまま効く設計とする。
A のみ実在（1名）、B/C は proto。外部証拠が届いたら proto を引用つき事実へ昇格させる。

## 優先順位付き機会リスト（下流入力）

1. J1 の「2パス導出で暗黙概念を取りこぼさない」体験の担保（最高レバレッジ、主要ペルソナの核）。
2. 成果物の SDD 契約準拠による下流直結（G1 と連動、二度手間の除去）。
3. J2 のブラウンフィールド採点の客観性（served ペルソナ向け、Should 帯）。
4. B の導入容易性（外部 OSS 需要が検証されたら優先度上昇。現状 `TBD`）。
