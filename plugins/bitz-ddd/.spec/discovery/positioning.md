# ポジショニング（競合代替 → PoD/PoP → ステートメント）

> 遡及 discovery。競合の詳細調査は範囲外。簡易 intake で空白地帯だけ確認する。
> 出典が取れないものは `TBD`。

## 競合代替の簡易 intake（顧客が bitz-ddd 無しでやること）

| 区分 | 代替 | 効く次元での比較 |
|---|---|---|
| 現状維持 | **sdd-design の軽量デフォルト設計のみ**で済ませる | 導入ゼロだが、複雑ドメインで集約境界が恣意的・暗黙概念の取りこぼしリスク |
| 現状維持 | DDD を頭の中／手書きメモで適用し成果物化しない | SDD ライフサイクルに繋がらず二度手間・トレース欠落 |
| 間接競合 | DDD の書籍・ブログを都度参照して手動適用 | 知識源としては豊富だが、毎回の想起・成果物化コストが高く、SDD 契約と非連携 |
| 間接競合 | 汎用 AI にプロンプトで「DDD で設計して」と依頼 | 手軽だが、2パス導出の強制・根拠記録・DSN- 契約準拠がなく再現性・下流連携が弱い `[proto / 未検証]` |
| 間接競合 | 独立系のモデリング/図ツール（例: PlantUML 等での手描きモデル） | 図は描けるが JTBD 起点の導出規律や SDD 契約への流し込みはない `TBD`（個別ツール比較は未実施） |

**目的は空白地帯の発見**: 「DDD の設計規律を、SDD の設計工程に契約準拠で差し込み、成果物が下流に直結する」領域が空白。

## PoP / PoD

- **PoP（同等性のポイント、過剰投資しない）**
  - Entity/VO/Aggregate・境界づけられたコンテキストという DDD の標準語彙を扱える（当たり前品質）。
  - ドメインモデルを何らかの成果物として残せる。
- **PoD（差別化のポイント、資源集中）**
  - **D1: bitz-sdd との完全な契約準拠**（artifact-frontmatter / DSN- 体系 / `.spec/design/` 配置）で、
    設計成果物が要件派生・実装・検証へそのまま流れる。独立ツールにはこの連携がない。
  - **D2: AI エージェント向けに規律を強制**（2パス導出必須・根拠記録必須・自己完結スキル）。
    汎用プロンプトや書籍参照より再現性が高い。
  - **D3: 手法を「差し替え可能なプロバイダ」として提供**（sdd-design 軽量設計 ⇄ bitz-ddd 本格 DDD を必要時に切替）。

## ポジショニング・キャンバス（April Dunford）

1. **競合代替**: 軽量設計のみ／DDD 書籍の手動適用／汎用 AI プロンプト。
2. **独自の属性**: SDD 契約準拠での成果物出力・2パス導出の強制・DDD/MMI 評価の内蔵。
3. **価値（成果）**: 個人開発者でも、複雑ドメインを取りこぼしなくモデル化し、設計が下流に直結して二度手間が消える。
4. **ターゲットセグメント**: bitz-sdd 導入済みの個人〜小規模開発者（主要ペルソナ A）。
5. **市場カテゴリ**: 「BitzSDD の DDD 設計手法プロバイダ（プラグイン）」。

## ポジショニング・ステートメント（Moore 短形式）

> **bitz-sdd を使う個人〜小規模開発者**のための **bitz-ddd** は、複雑ドメインを規律ある形で設計したいという
> **ニーズ**に対する **SDD の設計手法プロバイダ（プラグイン）**であり、ドメインストーリー → ドメインモデル →
> 成熟度評価を契約準拠で提供する。**軽量デフォルト設計や汎用 AI プロンプト**と違い、
> **2パス導出を強制し成果物を SDD ライフサイクルにそのまま流す（bitz-sdd 契約準拠）**。

## 規律

- 主張するのは**価値（取りこぼしなき設計・下流直結・再現性）**であって機能リストではない。
- 崩壊クリティカルな差別化仮説（D1 の契約準拠が本当に採用理由になるか、D2 の規律強制が品質を上げるか）は
  assumptions.md（DDD-DSC-006）の仮説表へ転記する。
