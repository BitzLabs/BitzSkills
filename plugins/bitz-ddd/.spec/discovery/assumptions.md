# 仮説検証ゲート（Desirability / Viability / Feasibility → Go/No-Go）

> 遡及 discovery。bitz-ddd はリリース済みのため Feasibility は実在で検証済み。
> 未検証なのは主に Desirability（外部需要）と Viability（保守持続性）。
> **最終裁定は人間が PR レビューで行う。** 以下は裁定案とエビデンス。

## 仮説表（3分類 × 崩壊影響 × テスト+事前閾値）

| ID | 分類 | 仮説 | 崩壊影響 | 状態 | テスト + 事前閾値 |
|---|---|---|---|---|---|
| H-D1 | Desirability | 個人開発者は sdd-design 軽量設計だけでは足りず、複雑ドメインで DDD 手法を実際に呼び出す | **崩壊クリティカル**（無ければプラグインの存在理由が消える） | **部分検証**（作者のドッグフーディング前提。外部は未検証） | ドッグフーディングで ddd-story→ddd-model の一連が実運用で最低1回使われる／閾値: 使われなければ Vision 再考 |
| H-D2 | Desirability | 外部 OSS 利用者に DDD 手法プロバイダの需要がある | 中（無くても主要ペルソナ A で成立） | **未検証** `TBD` | GitHub の install/star・issue の声を観察／閾値未定 `TBD` |
| H-D3 | Desirability | 2パス導出の強制・契約準拠が「汎用 AI プロンプト」より選ばれる理由になる（D1/D2 差別化） | 高（PoD の中核） | **未検証**（proto） | 作者が両者を比較し、取りこぼし/再現性で優位を体感／閾値: 優位を感じなければ差別化見直し |
| H-V1 | Viability | メンテナ1名で3スキル＋今後の SI-CORE-014 改修を保守し続けられる | 高（放棄すれば価値ゼロ） | **部分検証**（現状 v0.1.1 まで維持できている） | 保守が破綻せず SI-CORE-014 まで到達／閾値: 破綻したらスコープ縮小 |
| H-V2 | Viability | bitz-sdd の契約変更に追従するコストが持続可能な範囲に収まる | 中〜高（強結合リスク） | **未検証** | 契約変更発生時の追従工数を観察／閾値 `TBD` |
| H-F1 | Feasibility | 3スキルが bitz-sdd 契約下で成果物を破綻なく生成できる | 崩壊クリティカル | **検証済み**（リリース済み・実在） | `release_check.py` / `spec_inspect.py` で契約準拠を機械検証 |
| H-F2 | Feasibility | AI エージェント主導の DDD 導出が実用に足る品質を出せる | 高 | **部分検証**（作者利用で成立、外部品質は未検証） | ドッグフーディングでの設計欠陥の早期発見実績／閾値: 重大な誤導出が頻発しないこと |

## 崩壊影響ランク（Riskiest Assumption 順）

1. **H-D1**（崩壊クリティカル・部分検証）— 存在理由。作者利用で部分的に支えられているが外部は未検証。
2. **H-F1**（崩壊クリティカル・検証済み）— リリース済みで実在。機械検証可能。リスク低。
3. **H-D3 / H-F2**（高・未検証〜部分）— 差別化と導出品質。proto。
4. **H-V1 / H-V2**（保守持続性）— 1名運用と強結合の追従コスト。

## Go/No-Go 判定基準（sdd-core ゲート規約が正）

- **No-Go**: 崩壊クリティカルな仮説のいずれかが「未検証」かつ「テスト+閾値も未定義」。
- **Go**: それ以外。未解決仮説は open リストで追跡。

## Go/No-Go 裁定案（最終裁定は人間が PR レビューで行う）

**裁定案 = 条件付き Go（Conditional Go）。**

- **根拠1（F1 検証済み）**: プラグインはリリース済み（v0.1.1）で、崩壊クリティカルな Feasibility 仮説 H-F1 は
  実在により検証済み。契約準拠は `release_check.py` / `spec_inspect.py` で機械確認できる。
- **根拠2（D1 に事前閾値あり）**: もう一つの崩壊クリティカル仮説 H-D1 は未完全検証だが、
  ドッグフーディングというテストと「実運用で最低1回使われる」事前閾値が定義済み＝No-Go 条件（テスト未定義）に当たらない。
- **条件（open で追跡）**: H-D2/H-D3/H-F2（外部需要・差別化・導出品質）は未検証。設計続行は許容するが、
  外部エビデンスが届いた時点でゲートを再訪する（一度きりにしない）。SI-CORE-014 は構造/動作変更のみで
  本 discovery の結論を変えない見込み。

> 補足: これは既存プラグインの遡及裁定であり、実質は「既存の設計判断の追認」。純新規なら H-D2/H-D3 の
> 未検証を重く見るべきだが、既にサンクした実装があり主要ペルソナ（作者）で価値が出ているため条件付き Go が妥当。
> ただし**最終判断は人間の PR レビューに委ねる（要ユーザー判断）**。
