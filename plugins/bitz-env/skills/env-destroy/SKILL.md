# env-destroy

## 目的

プラグイン同梱フック（即効層）はプラグイン無効化で消えるが、env-init が生成した
恒久層（生成層）はアンインストール後もプロジェクトに残留する。本スキルは
レジストリ `.claude/bitz-env.local.md` の生成物トラッキング記録を根拠に、
残留物を**ユーザー確認付きで**撤去し、環境を bitz-env 導入前の状態へ戻す。

## 原則

- **レジストリの記録が撤去の唯一の根拠**。記録に無いものを推測で削除しない
- **撤去は必ずユーザー確認付き**。一覧を提示し、承認を得てから実行する
- **マーカー区間のみ除去**。`AGENTS.md` / `CLAUDE.md` などの既存ファイルは
  `<!-- bitz-env:begin -->` 〜 `<!-- bitz-env:end -->` の区間だけを取り除き、
  ユーザー自身の記述は一切変更しない
- **対象はプロジェクト内限定**。ホームディレクトリ等プロジェクト外には触れない

## ワークフロー

### 1. レジストリの読み込み

対象プロジェクト直下の `.claude/bitz-env.local.md` を読み、撤去対象を分類する:

| 分類 | 例 | 撤去方法 |
| --- | --- | --- |
| 新規生成ファイル | `.claude/agents/advisor.md` / `worker.md` | ファイルごと削除 |
| マーカー区間 | `AGENTS.md`（ガードレール節）、`CLAUDE.md`（委譲マトリクス節） | 区間のみ除去 |
| permissions | `.claude/settings.json` の deny/ask エントリ | bitz-env 由来のエントリのみ除去（後述） |
| アダプタ登録 | レジストリ内の `routes` / `priority` | レジストリ自体の削除に含まれる |

**レジストリが存在しない・記録が欠落している場合**: 推測で削除せず、手順2の
スキャンで検出できた候補を「未記録（要ユーザー判断）」として報告するに留める。

### 2. 現状スキャンと突合

記録と実ファイルの現状を突き合わせ、ズレを検出する:

- 記録にあるが実体が無い → 「撤去済み/手動削除済み」として報告のみ
- マーカー区間がユーザーにより編集されている（env-doctor の差分検出と同じ観点）→
  区間除去の diff を提示し、個別に確認を取る
- `settings.json` は生成時のテンプレート（env-init の `references/permissions.md`）
  由来のエントリのみ除去候補とし、ユーザーが独自に追加したエントリは残す。
  区別がつかないエントリは削除せずユーザー判断に委ねる

### 3. 撤去計画の提示と確認

撤去対象の全一覧（ファイル削除 / 区間除去 / エントリ除去の別、各 diff）を提示し、
ユーザーの承認を得る。部分的な承認（これは残す等)にも対応する。

### 4. 撤去の実行

- 承認された項目のみ実行する。展開先が git 管理外の場合は、書き換え対象ファイルの
  `.bak` を作成してから実行する（env-init の書き込み前チェックと同じ規律）
- すべての撤去が完了したら、最後にレジストリ `.claude/bitz-env.local.md` 自体の
  削除を確認して実行する（部分撤去の場合は残し、撤去済み項目を記録から消す）

### 5. 報告

- 撤去した項目・スキップした項目・未記録のため残した候補を報告する
- プラグイン自体の無効化/アンインストール（`/plugin` 操作）はユーザーの操作領域
  なので、必要なら手順を案内する（フック＝即効層はそれで消える）

## してはいけないこと

- レジストリに記録の無いものの推測削除
- マーカー区間の外にある既存記述の変更・削除
- ユーザー確認なしの一括削除
- プロジェクト外（ホームディレクトリ等）への操作
