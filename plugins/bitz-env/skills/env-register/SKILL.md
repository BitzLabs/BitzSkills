# env-register

## 目的

契約（v2）準拠の協調アダプタをプロジェクトのレジストリへ登録し、
役割（delegate / review / status）→ 実スキル名のルーティングテーブルを記録することで、
env-orchestration と CLAUDE.md の委譲マトリクスが固定スキル名に依存せず解決できるようにする。

## ワークフロー

### 1. アダプタの検出

次の順で試み、取得できた手段を使う（すべて失敗したら手動登録へフォールバック）:

1. インストール済みプラグインの列挙（`claude plugin list` / `agy plugin list` 等が
   使える環境ならその出力）
2. ユーザーへの質問: 「導入した協調アダプタのプラグイン名（またはパス）は？」

検出した各候補について、能力宣言（マニフェスト `metadata.collab` または
ルートの `collab.json`）を読み、契約準拠か確認する:
- `collab.agent` があること
- `delegate` 相当の役割スキル（または能力宣言）を持つこと（review / status は任意）

非準拠なら理由を報告し、登録しない。

準拠した候補については、`metadata.collab.skills`（役割 → 実スキル名の宣言）を読み、
役割ごとの実スキル名を確定する。`skills` が省略されている場合は
**役割名と同名のスキルを持つ**とみなす（v1 固定名アダプタの前提と同じ）。

### 2. レジストリへの登録（ユーザー確認付き）

`.claude/bitz-env.local.md` を作成・更新する（YAML frontmatter + 本文）。
各アダプタのエントリには**役割→実スキル名のルーティングテーブル**（`routes`）を持たせる:

```markdown
---
adapters:
  - name: bitz-collab-example
    agent: agy
    routes:
      delegate: agy-delegate
      review: agy-review
      status: agy-status
    strengths: [量産, 長文読解, web検索]
    break-even: 3ファイル以上の一括生成
    registered: "YYYY-MM-DD"
    priority: 1
---

# bitz-env レジストリ

このファイルは env-register が管理する。手動編集も可（次回実行時に検証される）。
```

既登録のアダプタは重複追加せず、能力宣言に変化があれば `routes` を含めて更新する。
アンインストールされたアダプタは、ユーザー確認のうえエントリを削除する。

#### 名前衝突の検出と解決

新規登録するアダプタの `routes` の実スキル名が、既登録アダプタの `routes` に
既に存在する実スキル名と重複する場合、これを**名前衝突**として検出する。
検出したら:

1. 衝突している役割・スキル名・双方のアダプタ名をユーザーに報告する
2. 後発アダプタの当該スキル名に**アダプタ名プレフィックスによる名前空間化**を適用し
   （例: `delegate` が衝突するなら `bitz-collab-example-delegate` として `routes` に記録）、
   レジストリ上のエントリを衝突しない実体で確定させる
3. 名前空間化後もどちらを優先して解決するかは `priority`（数値が小さいほど優先）で
   レジストリに明示する。既登録アダプタの `priority` は変更しない

#### v1 固定名アダプタの移行

`metadata.collab.skills` を持たない（= v1 相当の）アダプタは、
「役割名 = 実スキル名」の**自明なルーティング**として登録する
（例: `routes: {delegate: delegate, review: review, status: status}`）。
契約 v2 が求める形式的な要件（能力宣言・ルーティングテーブルの記録）は
この自明ルーティングで満たされるため、アダプタ側の変更は不要。

### 3. 委譲マトリクスの更新

レジストリの登録・更新・削除が完了するたびに、ユーザーの明示依頼の有無にかかわらず
このステップを実行する（レジストリと委譲マトリクスを常に同期させる）。

CLAUDE.md の `<!-- bitz-env:begin -->` 〜 `<!-- bitz-env:end -->` 区間内にある
「今回の割り当て」表の**協調アダプタ行だけ**をレジストリの `routes` から再生成する。
マーカー区間が無い場合は env-init の実行を先に案内する。

### 4. 報告

登録・更新・削除したアダプタと、委譲マトリクスの変更内容を報告する。
名前衝突を解決した場合はその内容（衝突したスキル名・採用した名前空間・優先順）も報告する。
動作確認として、アダプタの `status`（あれば、レジストリの `routes.status` が指す
実スキル名）を1回呼んで疎通を見るのが望ましい。

## してはいけないこと

- 非準拠プラグインの黙認登録（契約チェックを省略しない）
- 名前衝突を検出せずに `routes` を上書き登録すること
- マーカー区間外の CLAUDE.md 変更
- レジストリ以外の場所への状態の記録
