# マイグレーション実行ランブック（env-update 用）

CORE-CON-009 / DSN-002 の累積マイグレーション機構を env-update が実行するための操作手順。
**規約の正**は plugin-creator の `plugin-structure/references/migration-steps.md`。本書はスキル
自己完結の原則（他スキルの references を相対参照しない）に従い、実行に必要な手順を bitz-env 側に
再掲したものである。書式が食い違った場合は plugin-creator 側を正とする。

## バージョン軸

- **D** = 配置先の記録バージョン。bitz-env では レジストリ `.claude/bitz-env.local.md` の
  frontmatter `bitz-env-version`（env-init / env-update が stamp したプラグイン version）。
- **T** = ライブラリ側 bitz-env の `plugin.json` の `version`（3マニフェスト共通の semver）。
- どちらか欠落・不正形式なら比較不能 → 安全側停止。semver 比較は 3 整数分解の数値比較。

## ステップのファイル配置（1ファイル1ステップ）

`references/migrations/<from>-to-<to>.md`（例: `0.6.0-to-0.7.0.md`）。ステップは**状態形式が
実際に変わった版にのみ**作る（形式無変更の version bump にステップは不要。飛び番は正常）。
連続チェーンとして書く（あるステップの `to` は次に形式が変わるステップの `from` と一致）。
最初のステップの `from` はプラグイン初版（baseline）。
**状態形式を変える PR は対応するステップを同じ PR に同梱する**（書き忘れは機械検出できない規律）。

## ステップの必須セクション

| セクション | 内容 |
|---|---|
| `from` / `to` | 入力形式と適用後の版（プラグイン version の semver、`from < to`）。ファイル名と一致 |
| `targets` | 変換対象（例: `config:.claude/bitz-env.local.md` / `deployed:.claude/agents/*.md#frontmatter` / `marker:AGENTS.md#bitz-env`） |
| `transform` | 変換内容（何を→何に）。before/after の例を必ず含める |
| `guard` | 適用済み判定＝冪等性の根拠。「この条件が成立していれば適用済みとして no-op」を機械確認可能に書く |
| `verify` | 適用成功の機械確認（YAML パース可・新キー存在・旧キー不在等） |
| `rollback` | 失敗時の復元手順（原則: 適用前バックアップから復元。破壊的置換を避け加算的変換を推奨） |

ステップ本文は必ず「guard 判定 → 未適用なら transform → verify」の順で記述する。

## チェーン解決と適用

1. D と T を読む。読めなければ安全側停止。
2. `D >= T` なら移行不要（更新なしか降格）→ 終了。
3. `migrations/` の全ステップをファイル名から from/to 抽出して `to` 昇順にソートし、
   **適用候補** = `D < to <= T` を取る。
4. 適用候補が空 → D〜T 間に形式変更なし → 移行不要で終了（正常。`migrations/` 空も同じ）。
5. **連続性検査（dry-run。書き込み前）**:
   - D が最古ステップの `from` より古く接続ステップが無い → **GAP** → 安全側停止。
   - 隣接候補で `step[i].to != step[i+1].from` → **GAP**（チェーン断裂）→ 安全側停止。
6. PASS なら `from` 昇順に**逐次適用**。各ステップで guard 判定 → 未適用なら transform → verify。
   verify 失敗は当該ステップの rollback を実行して停止・報告。
7. **全ステップ成功後に**、レジストリ `bitz-env-version` を D→T へ stamp（コミットマーカー。
   必ず最後）。途中失敗時は D が旧値のまま残るため再実行可能で、guard により適用済みステップは
   no-op で通過し収束する。

## 安全側停止（変換せず停止し状態と不足を提示）

- (a) D または T が semver として解釈不能。
- (b) D が未記録かつレジストリ自体が不在（env-init 未実行）、または D 未記録の
  救済フローでユーザーが続行を承認しない。レジストリが存在する D 未記録は即停止ではなく、
  SKILL.md 手順 1b の救済フロー（保守的推定 D̂ = 最古ステップの `from`、ステップ不在時は
  stamp 機構導入版 `0.7.0`。承認後の適用の最初に stamp）で後付けを試みる（ENV-FR-012）。
- (c) D が最古ステップの `from`（baseline）より古く接続ステップが無い。
- (d) 適用候補チェーンに断裂がある。
- (e) ステップの verify が不成立、かつ rollback 後も回復不能。
- (f) 変換対象ファイルが配置先に存在しない／パース不能で入力形式を判定できない。

(a)〜(d) は読み取りのみで判定でき、書き込み前に停止する。

## 書き込み前の承認フロー

配置先はリポジトリ外になりうる（対象プロジェクトの `.claude/bitz-env.local.md` 等）。
書き込み前に必ず: (1) dry-run で変換対象一覧と before/after 差分を提示、(2) 書き込み先の絶対パスと
リポジトリ内/外を明示、(3) ユーザー承認取得まで一切書き込まない（バックアップ取得含む実書き込みは
承認後）、(4) rollback 用バックアップ先を提示。書き込み先が作業中リポジトリ内の場合も dry-run 差分を
提示し、デフォルトブランチ直コミット禁止のブランチ規律に従う。

## 合成フィクスチャによる動作確認（migrations 同梱時に実施）

スクラッチ領域（`/tmp` 配下や worktree 内一時ディレクトリ）に旧形式を模したレジストリ／frontmatter を
用意し、少なくとも次を確認して結果を `evals/env-update/` に記録する:

- **冪等性**: 同一ステップの二重適用で状態が壊れない（2回目は guard で no-op）。
- **安全側停止**: 中間ステップを欠落させたフィクスチャで、書き込みが起きずに停止・提示される。
