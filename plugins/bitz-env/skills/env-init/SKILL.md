# env-init

## 目的

bitz-env プラグインが定義する開発環境 — ガードレール（機械強制 + ナラティブ）と
モデル非依存の協調運用（中心・advisor・worker） — を、対象プロジェクトへ
**ユーザー確認付きで**書き出す。プラグイン同梱フックは導入時点で既に有効なので、
本スキルはそれを補完する「生成層」を担当する。

## 原則

- **書き出しは必ずユーザー確認付き**。既存ファイルがある場合は上書きせず、
  diff を提示してマージ案を出す
- **生成物は自己完結**。プラグインの更新に自動追従しない（差分検出は env-doctor が行う）
- **再実行可能**。CLAUDE.md / AGENTS.md への挿入はマーカーコメント
  `<!-- bitz-env:begin -->` 〜 `<!-- bitz-env:end -->` で囲み、再実行時はその区間だけ再生成する
- **区間外へ新規バイトを追加しない**。マーカーで管理する既存ファイルへ追記するとき、
  既存内容との区切りに必要な空白・改行は開始タグの内側（`<!-- bitz-env:begin -->` の直後）に
  含める。区間外は既存記述のバイト単位保持だけでなく「何も足さない」ことまで含めて不変とする

## ワークフロー

### 1. 環境の確認

対象プロジェクトのルートで以下を確認する:

- プラットフォーム: Claude Code / Antigravity / 両方（ユーザーに質問）
- 既存ファイルの有無: `.claude/settings.json`、`AGENTS.md`、`CLAUDE.md`、`.claude/agents/`
- Git リポジトリか（そうでなければ先に `git init` を提案）

### 2. 書き込み前チェック（復旧可能性の確保）

既存ファイル（`.claude/settings.json` / `AGENTS.md` / `CLAUDE.md`）を書き換える前に、
展開先の Git 管理状態で分岐する:

- **git 管理下**（`.git/` が存在し対象ファイルが追跡対象）: git 自体を復旧手段とみなし、
  `.bak` の作成は省略してよい
- **git 管理外**: 書き込み前に対象ファイルを `<ファイル名>.bak`（例: `AGENTS.md.bak`）として
  同じディレクトリにコピーし、あわせて `git init` の実行をユーザーに案内する
  （案内するのみで、ユーザー確認なしに実行はしない）

新規作成（既存ファイルなし）の場合、書き換え対象がないため `.bak` は不要。

### 3. 中心モデルと役割の割り当て

ユーザーに「普段このプロジェクトで使うモデル（中心）」を聞き、割り当てを提案する:

| 中心モデル | advisor（相談先） | worker（委譲先） | 主なパターン |
| --- | --- | --- | --- |
| 最上位（Opus 等） | なし（不要） | Sonnet / Haiku | 委譲型のみ |
| 中位（Sonnet 等） | Opus | Haiku | 委譲型 + 相談型 |
| 軽量（Haiku 等） | Opus / Sonnet | なし | 相談型が主体 |

提案はあくまで既定値。ユーザーの予算・プラン制約に応じて変更してよい。
Antigravity のみの環境ではモデルエイリアスが解釈されないため `inherit` を用いる。

### 4. 生成（すべて確認付き）

| 生成物 | テンプレート | 内容 |
| --- | --- | --- |
| `.claude/settings.json` | `references/permissions.md` | deny/ask の permissions（既存があればマージ案を提示） |
| `AGENTS.md` | `references/templates/AGENTS-template.md` | ガードレール節 + 検証義務（`{{project}}` を置換） |
| `CLAUDE.md`（断片挿入） | `references/templates/CLAUDE-fragment.md` | 役割ベースの委譲マトリクス + 今回の割り当て表 |
| `.claude/agents/advisor.md` | `references/templates/advisor.md` | 相談役（`{{model}}` を手順3の割り当てで置換）。中心が最上位なら生成しない |
| `.claude/agents/worker.md` | `references/templates/worker.md` | 作業役（同上）。中心が軽量なら生成しない |

書き込み対象が手順2で「git 管理外」と判定されたファイルの場合、生成直前に `.bak` を作成する。

**プレースホルダの決定規則**: テンプレート中の `{{project}}` 等のプレースホルダは、
次の優先順位で値を決める — (1) ユーザーが明示した名称、(2) git リポジトリ名
（リモート URL 等から取得できる場合）、(3) 対象プロジェクトのディレクトリ名。
プロンプトに明示がない場合も仮の値を発明せず、この規則で決めた値とその根拠を
生成確認時にユーザーへ提示する。

**permissions マージの重複判定規則**: 既存 `.claude/settings.json` とテンプレートを
マージする際、文字列を正規化（前後の空白除去）した上で完全一致するエントリのみを
重複とみなして1件に統合する。表記ゆれやパターンの包含関係の意味的な判定は行わず、
どちらも残す（安全側に倒す = 多重登録を許容し、既存エントリの削除は行わない）。

### 5. 生成物トラッキング

生成・変更のたびに、対象プロジェクト直下のレジストリ `.claude/bitz-env.local.md` へ
以下を追記する（存在しなければ新規作成）:

- **展開時の bitz-env プラグイン version**: レジストリ先頭の YAML frontmatter に
  `bitz-env-version: "<version>"` として記録する（値は bitz-env の `plugin.json` の `version`。
  3マニフェスト共通）。これは env-update がバージョン比較・マイグレーションの基準
  （配置先バージョン D）として読む唯一の記録であり、省略すると env-update が D 不明で
  安全側停止するため、初回生成時に必ず stamp する（既存レジストリに frontmatter が無ければ追加する）
- **新規生成したファイル**: パスと生成日時（例: `.claude/agents/advisor.md`）
- **既存ファイルへ書き込んだマーカー区間**: パスとマーカー区間の位置
  （`<!-- bitz-env:begin -->` 〜 `<!-- bitz-env:end -->` を書き込んだファイル。
  例: `AGENTS.md`（ガードレール節）、`CLAUDE.md`（委譲マトリクス節））
- **バックアップの有無**: 手順4で `.bak` を作成した場合はそのパス

この記録は将来の **env-uninstall** スキルが撤去対象を判断する唯一の根拠になる
（レジストリに無い変更は撤去対象として扱われない）。そのため生成・書き込みの都度、
省略せず追記すること。

### 6. 検証と報告

- 生成した JSON が正しくパースできるか `python3 -m json.tool <ファイル>` で確認する
- 生成・スキップ・マージした項目、レジストリへ記録した `bitz-env-version`、および
  `.claude/bitz-env.local.md` への記録内容をユーザーに報告する
- 次のステップを案内する: 協調アダプタ（外部エージェント連携）があれば env-register、
  運用ルールの詳細は env-orchestration、定期診断は env-doctor、
  プラグイン更新後のバージョン更新は env-update

## してはいけないこと

- ユーザー確認なしの既存ファイル上書き
- マーカー区間の外にある既存記述の変更、および区間外への追記（区切りの空行1行も含む）
- プロジェクト固有でない場所（ホームディレクトリ等）への書き込み
