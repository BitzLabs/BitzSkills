# env-orchestration

## 目的

中心（ユーザーが選択しているモデル）を司令塔として、作業の性質に応じた
協調パターンの選択と実行を案内する。特定モデルの可用性に依存しない。

## 前提の読み込み

1. `.claude/bitz-env.local.md`（レジストリ）があれば読む — 登録済み協調アダプタと能力宣言、
   および役割（delegate / review / status）→ 実スキル名のルーティングテーブル
2. `.claude/agents/` に advisor / worker があるか確認する
3. どちらも無ければ委譲先ゼロの単独運用。必要なら env-init / env-register を案内する

## 委譲先の解決（レジストリ経由）

固定スキル名（例: 特定アダプタの `delegate` 相当スキル）を直接呼ばない。
呼び出す前に必ず次の手順でレジストリのルーティングテーブルを引く:

1. 必要な役割（`delegate` / `review` / `status`）を決める
2. レジストリのルーティングテーブルで役割に対応する実スキル名を引く
   （複数アダプタが同役割を持つ場合は能力宣言の得意領域・損益分岐で選ぶ）
3. 該当エントリが無ければ、その役割は不在として劣化動作（worker か中心へフォールバック、
   advisor 不在なら相談スキップ）する
4. 解決した実スキル名で呼び出す。役割名やアダプタ名をスキル名と決め打ちしない

## パターン選択の決定木

```
作業の性質は？
├─ 手順が明確な定型作業（修正・テスト実行・整理・一括更新）
│   └─ 【委譲型】worker へ。worker 不在なら中心が直接実行
├─ 量産・長文読解・Web 検索など外部エージェントの得意領域
│   └─ 【委譲型】レジストリの能力宣言に合致するアダプタの delegate 役割へ
│      （実スキル名はレジストリのルーティングテーブルで解決）。不在なら worker か中心
├─ 深い判断が必要（設計選定・原因不明の不具合・リスクの高い変更）
│   ├─ 影響が大きい（破壊的変更・セキュリティ関連・ユーザー明示要求）
│   │   └─ 【合議型】advisor + アダプタの review 役割（ルーティングテーブルで解決）へ並列諮問
│   └─ それ以外
│       └─ 【相談型】advisor へ。advisor 不在なら中心が判断し、その旨を明記
└─ 1ファイルの軽微な作業
    └─ 委譲しない（往復コストが節約分を上回る）。中心が直接実行
```

**worker とアダプタ delegate の両方に該当する場合**（定型作業だが規模が大きい等）:
レジストリの能力宣言にある `break-even` 目安（collab-contract の能力宣言フィールド）と
作業規模を照合し、目安を上回るならアダプタの delegate 役割を優先する。
`break-even` の宣言が無い、またはアダプタ不在なら worker を既定とする。
どちらを選んだかと判断根拠を記録に残す。

## 共通の運用ルール

- **検収は中心**: 委譲先・相談先・合議メンバーの自己申告を信用せず、
  テスト・機械チェックを中心が自分で実行して確認する
- **DIGEST のみに依存しない客観的検収**: DIGEST（要約報告）は幻覚・他ファイル破壊・
  隠蔽を隠しうる。中心は DIGEST を読んだ後、`git diff` / `git status` 等の
  客観的な状態変化を自ら取得し、報告内容と実際の変更が一致するかを確認する。
  一致しない・範囲外の変更がある場合は差し戻す
- **git 管理外の対象の検収**: 委譲先の作業対象が git 管理外の場合は、`git diff` の代わりに
  変更前バックアップ（`.bak` 等）とのファイル比較、対象ファイルの実在・内容・
  タイムスタンプの直接確認を代替の客観的検収手段とする。継続的に委譲する対象なら、
  作業前に `git init` をユーザーへ案内する（案内のみで勝手に実行しない）
- **実出力を欠く DIGEST への対応**: DIGEST が検証の実出力を含まない場合は契約違反
  （collab-contract 3節）として受理せず、実出力の提示を要求する。提示されるまで
  検収を完了扱いにせず、次の工程へ進めない
- **ネスト委譲の禁止（深さ = 1）**: 委譲・相談を受けた側（worker / advisor、
  アダプタの delegate / review）が、さらに別のエージェントへ委譲することは禁止する。
  追加委譲が必要だと worker / advisor が判断した場合は、その旨を DIGEST の
  未解決事項として報告するに留め、実際に追加委譲するかどうかは**中心のみ**が判断する
- **返答形式を明示**: 委譲・相談時は期待する成果物・報告形式（DIGEST）・検証方法を
  指示に含める。成果物全文を読み戻さない
- **並列は 5〜6 体まで**
- **劣化動作**: 相手が使えなくても停止しない。相談スキップ・合議の格下げ
  （1名なら相談型、0名なら単独判断）を行い、その旨を成果物に明記する

## 合議型の進め方

**worker を合議に含める基準**: 争点に実装可否・工数見積もり・実行手順の現実性など
実行観点の意見が必要な場合は worker も招集する。純粋な設計選定・原因調査など
実行観点を伴わない議題では advisor とアダプタの review のみで足りる。
どちらと判断したかとその理由を成果物に明記する。

1. 議題・判断基準・回答形式を1つのプロンプトにまとめる
2. メンバー（advisor、アダプタの review 役割、必要なら worker）へ**並列に**同じ議題を諮問する
3. 1ラウンドで意見を集め、割れた争点だけ絞って第2ラウンド（最大2ラウンド）
4. **裁定は多数決ではなく中心が行う**。各意見の根拠を突き合わせ、採用・棄却理由を記録する。
   機械検証できる争点は投票でなくテストで決める
5. 議題・各意見の要旨・裁定理由を成果物（設計文書・PR 本文等）に残す

## 協調アダプタとの連携

アダプタ（外部エージェント連携プラグイン）の契約は `references/collab-contract.md`（v2）が正。
レジストリに登録されたアダプタは、能力宣言（得意領域・損益分岐）に基づいて
委譲型・合議型の相手として使う。呼び出しは常に「委譲先の解決（レジストリ経由）」の
手順に従い、アダプタ固有のスキル名を決め打ちしない。
