# 協調アダプタ契約 v2（正）

bitz-env と連携する外部エージェント連携プラグイン（協調アダプタ）が満たすべき取り決め。
プラグイン同士は互いのファイルを参照できないため、連携は本契約（役割と能力宣言）と
プロジェクト内レジストリ（`.claude/bitz-env.local.md`）による疎結合で行う。

bitz-env はアダプタが1つも無くても完全に機能する。アダプタは純粋な追加である。

## 1. 標準の役割とスキル名

アダプタは以下の**役割**のうち提供するものをスキルとして持つ:

| 役割 | 内容 | 必須度 |
| --- | --- | --- |
| `delegate` | well-scoped なサブタスクの委譲実行 | 必須 |
| `review` | 現在の diff・設計へのクロスモデル検証（合議型のメンバー） | 推奨 |
| `status` | 実行中・完了ジョブの確認 | 任意 |

スキル名は役割名そのままではなく、**アダプタ固有のプレフィックス付き固有名を推奨**する
（例: `agy-delegate`, `codex-review`）。複数アダプタが同名スキルをグローバル名前空間へ
登録すると衝突する（上書き・意図せぬ別モデルの呼び出し）ためである。

役割とスキル名の対応は、env-register が登録時にレジストリの
**ルーティングテーブル（役割 → 実スキル名）**として記録する。呼び出し側
（env-orchestration）は固定スキル名を直接呼ばず、必ずレジストリ経由で解決する。
登録時に名前衝突を検出した場合、env-register はアダプタ名による名前空間化と
優先順の明示で解決する。

## 2. 能力宣言

アダプタの plugin マニフェスト `metadata` に `collab` セクションを持たせる:

```json
{
  "name": "bitz-collab-example",
  "version": "0.1.0",
  "metadata": {
    "collab": {
      "agent": "対象エージェント名（例: agy, codex）",
      "strengths": ["量産", "長文読解", "web検索"],
      "break-even": "委譲が割に合う目安の一文（例: 3ファイル以上の一括生成）",
      "skills": { "delegate": "agy-delegate", "review": "agy-review" }
    }
  }
}
```

`skills` は役割→実スキル名の宣言（省略時は役割名と同名のスキルを持つとみなす）。
マニフェストに書けない場合は、アダプタのルートに `collab.json`（同じ形）を置いてもよい。
env-register はどちらも読む。

## 3. 報告形式（DIGEST）と検収

- `delegate` は成果物全文ではなく **DIGEST（要約報告）** を返す:
  何を作った/変えたか、対象ファイル一覧、実行した検証とその実出力、未解決事項
- 「成功しました」だけの報告は契約違反。検証の実出力を必ず含める
- 検収は呼び出し元（中心）の義務であり、アダプタは検収を代行しない。
  中心は **DIGEST のみに依存せず**、`git diff` / `git status` 等の客観的な状態変化を
  自ら取得して検証する（自己申告の要約は幻覚・他ファイル破壊・隠蔽を隠しうる）
- 作業対象が git 管理外の場合、中心は変更前バックアップとのファイル比較や
  対象ファイルの実在・内容の直接確認を代替の客観的検収手段とする
  （継続的に委譲する対象なら、作業前の `git init` をユーザーへ案内する）
- DIGEST が検証の実出力を含まない場合、中心はこれを契約違反として**受理せず**、
  実出力の提示を要求する。提示されるまで検収を完了させない

## 4. 委譲の再帰禁止

- 委譲・相談を受けた側（worker / advisor）が、さらに別のエージェントへ委譲する
  **ネスト委譲は禁止**（深さ = 1）。委譲のピンポンによるコンテキスト・コストの
  枯渇を防ぐため
- 追加委譲が必要だと判断した場合、worker はその旨を DIGEST の未解決事項として
  報告し、追加委譲の要否は**中心のみ**が判断する

## 5. 互換性

- 既存の antigravity プラグイン（delegate / review / status 構成）は
  v1 契約の事実上の先行実装である。v1 の固定名アダプタは
  「役割 = 実スキル名」の自明なルーティングとして登録すれば v2 に準拠する（移行パス）
- 契約の拡張は後方互換で行う（既存フィールドの意味を変えない）
