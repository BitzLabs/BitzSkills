- **矛盾/曖昧の内容**: 「マーカー区間が無い場合は env-init の実行を先に案内する」という
  規定は SKILL.md ステップ3（L93）にのみ存在し、ステップ2（レジストリへの登録）には
  マーカー区間の有無に関する言及が一切ない。evals/env-register/runs/09（TC-04）では、
  マーカー区間が無い fixture に対し、ステップ2（レジストリ登録）も含めて全体を保留する
  拡大解釈で遂行した。run09 のログ「気づき」節は、これが「SKILL.md の記述からの拡大解釈」
  であり、「別の実行では『レジストリだけ先に登録し、CLAUDE.md 更新のみ保留する』という
  別解釈も成立し得る」と明記している。TC-04 のアサーションはレジストリの状態を検証しない
  ため今回の合否判定には影響しなかったが、実装のブレそのものは実際に観測された曖昧さである。
- **提案する修正**: ステップ1またはステップ2の冒頭に、マーカー区間の有無とレジストリ登録の
  可否の関係を明示的なルールとして確定させる一文を追加する。二案のいずれかを採用:
  (a)「マーカー区間の有無に関わらず、契約チェック・レジストリ登録（ステップ1・2）は
  独立して実行してよい。マーカー区間に依存するのはステップ3のみ」、
  (b)「マーカー区間が無い環境では bitz-env の協調運用フレーム自体が未確立とみなし、
  ステップ1の検出後、ステップ2以降は env-init 実行後まで保留する」。
  SI-ENV-011（委譲マトリクス同期規則）と合わせて一箇所（例: ステップ1冒頭）に
  集約して記述すると、規則同士の整合が取りやすい。一般化した修正であり過学習リスクは低い。
- **影響推定**: SKILL.md 本文の記述追加。採用案によっては env-init 未実行環境での
  レジストリ登録可否という実質的な挙動変更を伴うため、SI-ENV-011 と合わせて Design Gate
  相当の確認（人間の裁定）を推奨。evals/env-register の TC-04 相当ケースの再テストを推奨。
