- **矛盾/曖昧の内容**: SKILL.md ステップ3は「CLAUDE.md のマーカー区間内『協調アダプタ』行を
  レジストリの routes から再生成する」手順のみを記述し、*レジストリ変更（登録・更新・削除）
  のたびに毎回 CLAUDE.md も同期すべきか、ユーザーが明示的に依頼した場合のみ行うべきか*を
  規定していない。この空白が実行のブレとして実際に観測された:
  evals/env-register/runs/08（TC-03、入力プロンプト「second-adapter も登録して」＝
  委譲マトリクス更新の明示なし）は CLAUDE.md まで自発的に更新した一方、
  evals/env-register/runs/10（TC-05、入力プロンプト「レジストリを最新の状態に更新して」＝
  同じく明示なし）はレジストリのみに限定し CLAUDE.md を更新しなかった。両 run のログ
  「気づき」節が、この不整合を実行者自身も認識・指摘している。テストケースの期待成果物が
  ケースごとに異なる（TC-03 は更新前提、TC-05 は不要）ため今回の合否判定には影響しなかったが、
  発動条件が明文化されない限り、将来別の入力パターンで一貫しない挙動が再発するおそれがある。
- **提案する修正**: SKILL.md ステップ3の冒頭に、レジストリ変更全般に対する同期規則を
  一箇所で確定させる一文を追加する。例:「レジストリの登録・更新・削除が完了するたびに、
  ユーザーの明示依頼の有無にかかわらず委譲マトリクスの再生成を行う（ただしマーカー区間が
  無い場合はステップ3自体をスキップし env-init 実行を案内する＝SI-ENV-012 と整合させる）」。
  一般化した修正（特定ケースの文言追加ではなく、レジストリ変更全般に適用される同期規則の
  明文化）であり、過学習リスクは低い。
- **影響推定**: SKILL.md 本文の記述追加のみ（手順自体の追加ロジックは伴わない）。
  version bump（patch〜minor）。evals/env-register の TC-03/TC-05 相当ケースの再テストを
  推奨（挙動が変わる可能性があるため）。
- **裁定記録**: 2026-07-12 人間裁定（チャット指示「highの3件について進めましょう」）により accepted。提案どおり反映する。
