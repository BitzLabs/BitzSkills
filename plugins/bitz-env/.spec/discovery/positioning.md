# ENV-DSC-005 ポジショニング

> 競合の詳細調査は本スキルの範囲外。簡易 intake で足りる分だけ確認する。
> 定量比較の裏づけが取れないものは `TBD`。目的は機能チェックではなく空白地帯の発見。

## 競合代替の簡易 intake

顧客が「bitz-env がいなければすること」:

| 代替 | 種別 | 効く次元 | 弱点（空白地帯） | 出典 |
|---|---|---|---|---|
| 手書きの permissions + AGENTS.md をコピペ | 現状維持 | 導入が自由 | 3層が同期ズレする・両プラットフォームで二重管理・撤去が手作業 | `TBD` |
| プラットフォーム標準の hooks / permissions 単体 | 直接 | ネイティブで確実 | **プラットフォーム固定**。Claude Code と Antigravity で別々に作る必要 | `TBD` |
| pre-commit / husky 等の Git フック | 間接 | コミット時の検査 | **実行時割り込みでない**。エージェントの Bash 実行そのものは止められない | `TBD` |
| shell alias / rm ラッパー・自作スクリプト | 現状維持 | 手軽 | 属人的・エージェント経路を網羅できない・共有/撤去できない | `TBD` |

現状維持（手書き + プラットフォーム個別対応）が最大の競合。

## PoP（同等性のポイント — 過剰投資しない）

- permissions を正しく書ける
- フックが実際に発火して危険操作を止める
- 既存ファイルを壊さない（確認・復旧手段がある）

これらは「買わない理由の除去」。満たすだけでよく、超過しても勝てない。

## PoD（差別化のポイント — 資源を集中する）

1. **両プラットフォームを自動判別する単一フック** — Claude Code / Antigravity を
   一つの env_guard.py で吸収（ENV-FR-002）。二重管理を消す
2. **ガードレール3層の同期診断（env-doctor）** — permissions ⇔ フック ⇔ AGENTS.md の
   ズレを検出（ENV-FR-007）。他の代替にこの整合概念がない
3. **モデル非依存の協調運用（advisor/worker + アダプタ契約）** — 守りだけでなく
   協調の型まで込み（ENV-FR-005/006）。ガードツール単体との明確な差
4. **マーカー区間による安全な再生成と追跡撤去** — 壊さず入れ、綺麗に消せる（ENV-FR-004/010）

## ポジショニング・キャンバス（April Dunford）

1. **競合代替**: 手書き permissions + プラットフォーム個別の hooks（+ 現状維持）
2. **独自の属性**: 両プラットフォーム自動判別の単一フック / 3層同期診断 / モデル非依存協調
3. **価値（成果）**: エージェントに権限を委ねても**事故らない**・ルールが**形骸化しない**・
   モデルを替えても**協調の型を作り直さない**
4. **ターゲットセグメント**: 複数 AI エージェントを併用する個人〜少人数の開発者
5. **市場カテゴリ**: AI エージェント開発環境の「ガードレール + 協調運用」展開プラグイン

## ポジショニング・ステートメント（Moore 短形式）

> **複数の AI エージェントを併用する個人〜少人数の開発者**のための **bitz-env** は、
> **エージェントの誤操作事故と、守り・協調ルールの同期ズレ**に対する
> **開発環境展開プラグイン**であり、**プラットフォームを問わない常時ガードと
> モデル非依存の協調運用の型**を提供する。**手書きの permissions やプラットフォーム固有の
> hooks 単体**と違い、**単一フックが両プラットフォームを自動判別し、env-doctor が3層の
> 同期まで診断する（＝主要 PoD）**。

## 規律

- 価値（顧客の成果 = 事故らない・形骸化しない）を主張し、機能リストで戦わない
- PoD 主張は反証可能な仮説。崩壊クリティカルなものは ENV-DSC-006 の仮説表へ転記する
  （特に「両プラットフォーム自動判別の単一フック」は Feasibility 仮説として起票済み・verified）
