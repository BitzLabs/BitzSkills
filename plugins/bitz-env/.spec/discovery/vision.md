# ENV-DSC-001 ビジョン

> 遡及（reverse-derived）discovery。bitz-env は製作プラン（人間承認済み）に基づき
> 先に実装され、現在 v0.4.1。本ファイルは実装済みプロダクトの「なぜ・誰のため・何をやらないか」を
> 事後的に言語化する。根拠のない数値・市場データは `TBD` / `[proto / 未検証]` と明示する。

## Product Vision Board（Roman Pichler）— 5要素

### 1. Vision（目的地）

**どのプロジェクトでも、AI エージェントの破壊的な誤操作を導入初日に無害化し、
複数モデルの協調を前提にした開発環境を数分で立ち上げられる世界。**

エージェントに強力な実行権限を渡すほど「事故（rm -rf・force push・reset --hard）」の
コストが跳ね上がる。bitz-env はその安全弁と、モデル非依存の協調運用の型を、
プラットフォーム（Claude Code / Antigravity 2.0）を問わず標準装備にする。

### 2. Target Group（セグメント — 「みんな」は禁止）

- **主要**: 個人〜少人数で **複数の AI エージェントを併用** して開発する人
  （例: Claude Code の Fable/Opus/Sonnet + Antigravity/Gemini）。モノレポや
  スキル開発でエージェントに広い実行権限を渡す運用者。作者 hide 自身がこの層。
- **副次**: BitzSkills を採用する **将来の OSS 利用者**。自分のリポに危険操作ガードと
  AGENTS.md/permissions の雛形を素早く入れたい開発者。
- **anti-persona**: **完全なセキュリティ境界**（回避不能な sandbox・悪意ある回避の防止）を
  求めるエンタープライズのセキュリティ担当。bitz-env のガードは「誤操作抑止」であり
  セキュリティ境界ではない（ENV-CON-004）。この層は対象外。

### 3. Needs（解決する問題 / 便益）

- エージェントの破壊的操作が怖い。事故ってからでは遅い → **常時ブロックの安全弁が欲しい**
- permissions（settings.json）・同梱フック・AGENTS.md のナラティブが **バラバラで同期がずれる**
- モデルごとに役割（誰が advisor で誰が worker か）を **毎回設計し直すのが面倒**
- 導入・撤去で **既存ファイルを壊したくない / アンインストール後に生成物が残る** のが嫌

### 4. Product（際立つ少数の差別化要素 — 全機能リストではない）

- **単一の同梱フック（env_guard.py）** が両プラットフォームの入力形状を自動判別し、
  普遍的な破壊的操作の最小集合（rm -rf / git push --force / git reset --hard /
  git clean -f / sudo）を deny。判別不能な入力は安全に素通し（ENV-FR-001/002）
- **env-init の対話的展開** — マーカー区間 `<!-- bitz-env:begin -->`〜`end` による
  安全な再生成、ユーザー確認、git/バックアップによる復旧可能性（ENV-FR-003/004/009）
- **モデル非依存の協調運用** — 委譲型・相談型・合議型の3パターンと advisor/worker 生成、
  相手が不在でも止まらない劣化動作（ENV-FR-005）
- **env-doctor の3層同期診断** — permissions ⇔ フック ⇔ AGENTS.md のズレを検出（ENV-FR-007）
- **env-destroy のトラッキング撤去** — レジストリに基づき生成物を確認付きで安全に撤去（ENV-FR-010）

### 5. Business Goals（作り手にとっての便益）

- BitzSkills マーケットプレイスの **基盤プラグイン**。自リポの安全性を
  ドッグフーディングで担保し、OSS 採用の入口にする
- 収益は目標にしない（個人 OSS）。成功は「作者と採用者が事故なく開発を続けられること」

## Mission / Vision / Values の区別

- **Vision（目的地）**: 上記1。エージェント時代の開発環境から誤操作事故をなくす
- **Mission（日々の追求）**: 破壊的操作の最小 deny 集合と協調運用の型を、
  両プラットフォームへ「壊さず・確認付きで・撤去可能に」展開し続ける
- **Values（トレードオフ時の判断原則）**:
  1. **安全側に倒す** — 判別不能なら介入せず素通し、確認なしに書き込まない
  2. **移植性 > 網羅性** — deny は普遍的最小集合に限定（プロジェクト固有は permissions 側）
  3. **可逆性** — 生成物は必ず追跡・撤去・復旧できる状態に保つ

## PR-FAQ 圧力試験（Working Backwards）

### プレスリリース（出荷済みの体）

- **見出し**: bitz-env — 任意のプロジェクトに、危険操作ガードとモデル非依存の協調運用を対話1回で
- **小見出し**: 複数 AI エージェントを併用する開発者へ。事故る前に安全弁を、迷う前に協調の型を。
- **要約**: bitz-env は、破壊的操作を常時ブロックする同梱フックと、permissions/AGENTS.md/
  advisor・worker サブエージェントの雛形を、Claude Code と Antigravity 2.0 の両方へ展開する
  プラグイン。既存ファイルはマーカー区間だけを安全に書き換え、撤去も追跡できる。
- **問題（顧客の言葉）**: 「エージェントに実行権限を渡すのは便利だが、いつ `rm -rf` や
  force push をやられるか分からなくて落ち着かない。permissions とルール文書とフックの
  整合を手で保つのも限界だ。」
- **解決 / 差別化**: 単一フックが両プラットフォームを自動判別してガードし、env-doctor が
  3層の同期ズレを検出する。モデルを固定しない協調運用の型まで込みで、この組み合わせは他にない。
- **引用**: （責任者）「守りと協調を、プラットフォームに縛られず標準装備にしたかった。」
  （利用者）`[proto / 未検証]`
- **CTA**: `/plugin install bitz-env@bitzskills` → env-init を実行。

### FAQ

- **外部 FAQ**: 価格 = 無料（OSS）。対応 = Claude Code / Antigravity 2.0。
  既存設定を壊すか = マーカー区間のみ更新し、確認と復旧手段を提示する（壊さない）。
- **内部 FAQ**:
  - 市場規模 TAM/SAM/SOM = `TBD`（個人 OSS ゆえ市場定量化は未実施）
  - 競合 = 現状維持（手書きの permissions + AGENTS.md）／プラットフォーム固有の hooks 単体
    （詳細は ENV-DSC-005）
  - ユニットエコノミクス = 収益モデルなし。コストは作者のメンテ工数のみ
  - 主要リスク = ガードをセキュリティ境界と誤解される（ENV-CON-004 で明示的に否定）／
    正規表現ベースの誤検知・回避可能性
  - **Go/No-Go 基準（反証可能・後段 ENV-DSC-006 が執行）**:
    1. 崩壊クリティカルな Feasibility 仮説（両プラットフォーム吸収・応答時間）が
       実装で verified であること
    2. 作者のドッグフーディングで危険操作ブロックが実際に発火し、開発体験を損なわないこと
    3. 未検証の広域 Desirability は「着手」ではなく「拡大」の仮説として open 追跡できること
