# ENV-DSC-004 ペルソナ

> proto ペルソナ。作者本人（主要ペルソナ）以外の感情・生の声は `[proto / 未検証]`。
> 人口統計や台詞は捏造しない。証拠が届いたら引用つき事実へ昇格させる。

## Jobs-to-be-Done（ジョブストーリー）

- **J1（機能的・守り）**: 「AI エージェントに実行権限を渡して作業させる**とき**、
  破壊的な誤操作（rm -rf・force push・reset --hard）を事故る前に止めたい。
  安心して権限を委ねられる**ように**。」
- **J2（機能的・整合）**: 「permissions・フック・AGENTS.md を運用している**とき**、
  三者の同期ズレを手で追わずに検出したい。ルールが形骸化しない**ように**。」
- **J3（機能的・協調）**: 「大きな設計判断や原因不明の不具合に当たった**とき**、
  モデルを固定せず別視点（advisor）へ相談・合議したい。単独判断の事故を避けられる**ように**。」
- **J4（感情的）**: 「新しいプロジェクトを立ち上げる**とき**、安全弁と協調の型を
  最初の数分で入れ終えたい。毎回ゼロから設定する徒労を感じない**ように**。」
- **J5（社会的）**: 「OSS として公開する**とき**、他人のリポを壊さず綺麗に導入・撤去できる
  ものを配りたい。無責任なツールと見られない**ように**。」

ジョブはソリューション非依存で安定。ペルソナをジョブに繋留してピボットに耐えさせる。

## ペルソナカード

### 主要ペルソナ（プロダクトが最適化する1人）: マルチモデル個人開発者「hide 型」

- **1行要約**: Claude Code（Fable/Opus/Sonnet）と Antigravity（Gemini）を併用し、
  モノレポでスキル・プラグインを開発する個人開発者（作者本人）
- **文脈と行動**: エージェントに広い実行権限を渡す。司令塔＝賢いモデル、
  実作業＝安いモデルへ委譲する運用。両プラットフォームを行き来する
- **Jobs**: J1〜J5 すべて
- **Pains**: エージェントの誤操作事故の恐怖 / 3層の同期を手で保つ限界 /
  モデルごとの役割再設計の徒労
- **Gains**: 事故ゼロで権限委譲できる安心 / 診断1回でズレが見える / 協調の型の再利用
- **生の声**: 作者自身のドッグフーディングに基づく（実利用者・n=1）。
  「守りと協調をプラットフォームに縛られず標準装備にしたかった」

### 副次ペルソナ: OSS 採用者

- **1行要約**: BitzSkills を試し、自分のリポに危険操作ガードと雛形を入れたい開発者
- **Jobs**: J1・J4・J5 中心
- **Pains**: 導入で既存ファイルを壊されたくない / 合わなければ綺麗に消したい `[proto]`
- **Gains**: 確認付き展開とマーカー区間・撤去で安心して試せる `[proto]`
- **生の声**: `[proto / 未検証]`（フィードバック未収集）

### served（将来）: 小規模チーム

- 共通ガードレールと協調の型をチームで共有したい層。現時点の最適化対象ではない `[proto]`

### anti-persona: エンタープライズのセキュリティ担当

- 回避不能なセキュリティ境界を求める層。bitz-env のガードは誤操作抑止であり
  セキュリティ境界ではない（ENV-CON-004）。**明示的に対象外**。過剰な安心を売らない

## 主要ペルソナのジャーニー（段階 × レイヤー・要点のみ）

| 段階 | 行動 | 思考・感情 | ペイン | 機会（MoT） |
|---|---|---|---|---|
| 認知 | マーケットプレイスで bitz-env を知る | 「事故が怖い、何か守りが欲しい」 | 何が守られるか不明瞭 | ZMOT: description で「両プラットフォーム・誤操作抑止」を明言 |
| 導入 | `/plugin install` → env-init 実行 | 「既存設定を壊さないか」 | 上書き不安（谷） | **FMOT**: 確認付き・diff 提示・復旧可能（ENV-FR-003/009）で不安を解消 |
| 利用 | エージェント作業中にフックが deny 発火 | 「本当に止まった、安心」（山） | 誤検知で正当な操作が止まると苛立ち | **SMOT**: 実ブロックが約束を裏づける。誤検知率が体験を左右 |
| 保守 | env-doctor で3層のズレを診断 | 「手で追わずに済む」 | ルール形骸化の放置 | 定期診断で drift 0 を維持 |
| 撤去 | 合わなければ env-destroy | 「綺麗に消せて安心」 | 生成物の残留 | UMOT: 完全撤去の体験が推奨につながる（ENV-FR-010） |

- 感情曲線の谷は **導入時の上書き不安**。ここが最大のレバレッジ点で、
  確認・diff・復旧（ENV-FR-003/004/009）が集中的に対処している
- 全 Pain/機会はジョブに紐づく。機会リストは下流の設計（sdd-design）と
  スコープ（ENV-DSC-003）の入力になる
