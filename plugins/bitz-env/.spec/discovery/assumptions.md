# ENV-DSC-006 仮説検証ゲート

> 遡及 discovery。bitz-env は v0.4.1 まで実装済みで、Feasibility の多くは
> 実装で verified。したがって本ゲートは「着手可否」ではなく「実装済み方針の
> 妥当性が事後的に成立しているか」を裁定する。**最終裁定は人間が PR レビューで行う。**

## 全仮説の3分類

### Desirability（欲しいか）

| ID | 仮説 | 崩壊影響 | 状態 / エビデンス |
|---|---|---|---|
| D1 | マルチモデル個人開発者は、エージェントの誤操作事故を実際の痛みと感じ、常時ガードを求める | 高 | **作者のドッグフーディングで検証（n=1）**。広域は `[proto / 未検証]` |
| D2 | 3層の同期ズレは、手動管理より env-doctor 診断が欲しくなるほど頻繁に起きる | 中 | 作者の運用で drift は実在。広域頻度は `[proto]` |
| D3 | モデル非依存の協調運用（advisor/worker）は、モデル固定設計より価値がある | 中 | 作者の委譲運用（司令塔/作業者分離）で有用。広域は `[proto]` |

### Viability（続けられるか）

| ID | 仮説 | 崩壊影響 | 状態 / エビデンス |
|---|---|---|---|
| V1 | 個人 OSS として収益目標を持たず、メンテ工数が作者の許容内に収まる | 中 | 収益モデルなし（ENV-DSC-002）。工数は現状許容内 |
| V2 | OSS 採用が広がればフィードバックで品質が上がる（弱いネットワーク効果） | 低 | `[proto / 未検証]`。採用データ未収集 |

### Feasibility（作れるか）

| ID | 仮説 | 崩壊影響 | 状態 / エビデンス |
|---|---|---|---|
| F1 | 正規表現ベースのフックで実用的な誤操作抑止ができる（回避可能性は許容） | 高 | **設計裁定済み**（ENV-CON-004）。ENV-FR-001 verified（example-test） |
| F2 | 両プラットフォームのフック契約を単一 env_guard.py で吸収できる（＝主要 PoD） | 高 | **verified**（ENV-FR-002, example-test） |
| F3 | rules/*.md を両プラットフォームでコンテキスト注入できる（Claude Code は SessionStart stdout） | 中 | **implementing（ENV-FR-008）。実環境確認が残る** ← 唯一の未クローズ |
| F4 | 全 Bash 割り込みでも 200ms 以内に応答し開発体験を損なわない | 中 | **verified**（ENV-NFR-001, benchmark） |

## 崩壊クリティカル判定（Riskiest Assumption Test）

「間違っていたら戦略のどれだけが崩れるか?」で評価:

- **F1 / F2（高・崩壊クリティカル）**: ガードと両プラットフォーム対応という中核が崩れる。
  → **いずれも実装で verified**。クリティカル仮説は閉じている
- **F3（中・部分崩壊）**: rules 注入が失敗しても、フックの deny 本体（F1/F2）と協調運用は
  機能する。全崩壊ではなく機能低下にとどまる。テスト（example-test）と実環境確認手順が
  **定義済み**。「未検証だがテスト+閾値定義済み」＝ Go を妨げない
- **D1〜D3（Desirability）**: 崩壊クリティカルだが、**主要ペルソナ＝作者本人**であり
  ドッグフーディング（n=1）で価値が実証されている。広域 OSS の Desirability は未検証だが、
  それは「着手」ではなく「拡大」の仮説。実装済み・自用で成立しているため着手を止めない
- **V1（中）**: 成立（収益目標なし・工数許容内）。V2 は低影響で open 追跡

### kill / pivot 閾値（open 仮説の追跡）

- **F3**: 「実環境（Claude Code SessionStart + Antigravity rules）で bitz-env の rules 本文が
  実際にコンテキストへ読み込まれること」を確認。未達なら ENV-FR-008 を pivot（注入方式の再設計）。
  ただしガード本体には波及しない
- **D2/D3（広域）**: OSS 採用後にフィードバックを収集し、需要が確認できなければ
  該当機能の投資縮小を検討（拡大側の判断であり撤退ではない）

## Go / No-Go 裁定案（人間裁定の材料）

**裁定案: Go（条件付き）。**

- **根拠1**: 崩壊クリティカルな Feasibility 仮説 F1・F2（誤操作抑止・両プラットフォーム吸収）は
  実装 v0.4.1 で verified。残る F3（rules 注入）は部分崩壊にとどまり、テスト+実環境確認手順が定義済み。
- **根拠2**: 崩壊クリティカルな Desirability（D1〜D3）は主要ペルソナ＝作者本人のドッグフーディング
  （n=1）で実証済み。広域 OSS 需要は `[proto / 未検証]` だが「拡大」の仮説として open 追跡でよい。
- **根拠3**: 遡及 discovery ゆえ方針は既に実装で妥当性を示しており、No-Go 相当の
  未検証クリティカル仮説（テストも閾値も無いもの）は存在しない。**最終裁定は人間が PR レビューで行う。**

### Go 後に残す open リスト

- F3（ENV-FR-008 実環境確認）— 唯一の未クローズ・実環境検証手順あり
- D2/D3/V2 の広域検証 — OSS 採用後のフィードバックで再訪（ゲートは一度きりにしない）
- SI-CORE-006（env-destroy → uninstall 改名・共通ライフサイクル標準準拠）は人間裁定待ち。
  スコープ（ENV-DSC-003）の Should に反映済み
