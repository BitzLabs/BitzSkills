# ENV-DSN-002 stamp 後付け救済フローの設計（D 保守的推定と guard 冪等性）

- **背景 / 課題**: `bitz-env-version` stamp は v0.7.0（PR #74）で env-init に追加された。
  それ以前に展開された全環境（本リポジトリ自身を含む）はレジストリに記録が無く、
  env-update（ENV-FR-011）の安全側停止条件 (b)「D 未記録」に必ず該当して実質使えない。
  安全原則（推測で書き込まない）を保ったまま、既展開環境を update 可能な状態へ導く
  正式な救済パスが必要（SI-ENV-024）。

- **設計判断**:
  1. **救済フロー内蔵（SI-ENV-024 案 (a)）を採用**。env-update 自身が「レジストリ存在・
     `bitz-env-version` 不在」を検出したら、即停止ではなく救済フローへ分岐する。
  2. **D は保守的推定（最古側）とする**。歴代テンプレートとの照合による正確な D 復元は
     ライブラリ側に過去版が無く原理的に不可能。代わりに「未記録 = stamp 機構導入
     （v0.7.0）以前の展開」という事実だけを使い、推定 D̂ = `references/migrations/` の
     最古ステップの `from`（ステップ不在時はマイグレーション適用なし）とする。
     推定を最古側に倒すと適用候補が最大になるが、CORE-CON-009 のステップは
     guard（状態判定）付きで**適用済み状態への再適用は no-op** と検証済み
     （ENV-FR-011 の二重適用 no-op）。よって過剰適用は安全であり、
     過少適用（D を新しく見積もって形式移行が漏れる）だけが危険 — ゆえに最古側一択。
  3. **stamp のタイミングは「承認後の適用の最初」**。dry-run・D̂ 提示・ユーザー承認までは
     一切書き込まない（ENV-FR-011 の承認フローを維持）。承認後、まず D̂ をレジストリへ
     stamp してから通常手順（差分更新→マイグレーション→完了時 T stamp）へ続行する。
     こうすると stamp 直後から環境は「正常に stamp された旧環境」と区別がつかなくなり、
     途中失敗しても D̂ が残るため再実行は救済フローを経ず ENV-FR-011 の正常系で収束する。
  4. **レジストリ不在は救済対象外**。レジストリ自体が無い場合は env-init 未実行であり、
     推定の根拠（展開実績）が無いため env-init の実行を案内して安全側停止する。
     env-init の二重初期化防止（既存ファイルの確認フロー）との整合もこれで保たれる。
  5. **env-doctor は検出と案内のみ（読み取り専用維持）**。診断項目に「レジストリ存在・
     `bitz-env-version` 不在」の検出を WARN として追加し、修正案として env-update の
     救済フローを提示する。stamp の書き込みは doctor では行わない。

- **代替案と却下理由**:
  - **案 (b) env-init 再実行（冪等展開）で stamp させる案内のみ**: 書き込みロジックの追加が
    不要で最小だが、(1) env-init は「展開」スキルであり既存環境の update 経路として使うと
    二重初期化防止の確認フローと衝突する、(2) ユーザーが env-update を起動したのに別スキルへ
    たらい回しになる、(3) stamp だけでなく差分更新も env-init の上書き確認に依存してしまい
    マイグレーション機構（CORE-CON-009）を通らない — ため却下。
  - **正確な D 復元（歴代テンプレート照合）**: ライブラリは最新版しか持たず原理的に不可能。
    過去版の同梱はサイズと保守コストに見合わない。保守的推定 + guard 冪等性で十分。
  - **doctor による stamp 書き込み**: doctor の読み取り専用原則（ENV-FR-005 系の設計）を
    破るため却下。書き込みは承認フローを持つ env-update に集約する。

- **影響範囲・ロールバック**: 変更は env-doctor SKILL.md（診断項目追加）・env-update
  SKILL.md / migration-runbook.md（救済フロー追加と安全側停止文言の改訂）に閉じる。
  レジストリ書式に新フィールドは追加しない（既存 `bitz-env-version` の後付け書き込みのみ）。
  単独 revert 可能。
