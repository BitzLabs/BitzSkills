# EARS 要件からのテスト仕様導出

本ドキュメントは、EARS（Easy Approach to Requirements Syntax）形式で記述された要件仕様を解析し、具体的かつ実行可能で網羅的なテスト仕様を導出するための規則とパターンを定義します。テストは要件が正しく実装されたことを証明する唯一の手段です。

## 1. テスト導出と検証体系の基本原則

すべてのテストケースは、要件定義ドキュメントに含まれる EARS 節と明確かつ 1対1 以上の対応づけを持たなければなりません。テストなき要件は存在せず、要件なきテストも存在してはなりません。
なお、テストの検証メソッド（`verification_method` 統制語彙）の定義や、テストが成功（green）したとみなすための具体的な基準、またプロセスの検証階層の全体像については、既存の `verification.md` の「L2: コードの検証」および「テスト作成規則」に完全に従うものとします。本ドキュメントは、これらと重複しない範囲で「特定の EARS 構文から、具体的にどのようなテストパターンを導出・設計すべきか」の変換・翻訳規則に特化します。

## 2. EARS 節種別ごとのテスト導出パターン

EARS の各テンプレート構文は、システムの異なる側面（正常系、異常系、状態依存、普遍的ルールなど）を表現するため、それぞれ特有のテストアプローチと検証技法を要求します。要件の種別に応じて以下のようにテスト仕様を導出します。

### 2.1 Ubiquitous（普遍的要件）

- **構文**: `The <system> SHALL <system response>`
- **特性**: 特定のトリガや前提条件に依存せず、システム内で常時成立すべき不変条件（Invariant）や基本ビジネスルールを定義します。
- **導出パターン**:
  - **Property-Based Testing (プロパティベーステスト)**: 入力値の全ドメインにわたって成立すべき性質（Property）として定式化します。境界値だけでなく、ランダムに生成された多数の入力データに対して、不変条件が破られないかを検証するテストケースを設計します。
  - **Unit Test (単体テスト)**: ドメインエンティティの基本属性の制約や、インスタンス生成（コンストラクタ）時の不変条件バリデーションを検証するテストケースを網羅的に作成します。

### 2.2 Event-Driven（イベント主導要件）

- **構文**: `WHEN <trigger> THEN the <system> SHALL <system response>`
- **特性**: 特定のユーザー操作、外部システムからの入力、または内部で発生したイベントに対する、システムの直接的かつ決定論的な応答を定義します。
- **導出パターン**:
  - **BDD Scenarios (BDD シナリオ)**: `Given-When-Then` 形式のフィーチャファイル（Gherkin フォーマット等）に直接翻訳し、受け入れテストとして導出します。
    - `WHEN` 節のトリガ内容を、テストシナリオの `When`（アクション）にそのまま対応させます。
    - `SHALL` 節の応答内容を、テストシナリオの `Then`（アサーション）に対応させます。
  - **境界値テスト（Boundary Value Testing）**: トリガ条件が数値や特定の日付条件等を持つ場合、その境界付近での振る舞いを検証するパラメータ化テストを追加導出します。

### 2.3 State-Driven（状態主導要件）

- **構文**: `WHILE <system state>, the <system> SHALL <system response>`
- **特性**: システムが特定の状態（例: メンテナンスモード中、ユーザーがログイン済み、在庫がゼロ）にある間だけ継続して適用されるルールを定義します。
- **導出パターン**:
  - **Integration Test (統合テスト)**: 特定の状態をセットアップするための事前条件が複雑になるため、データベースの状態初期化スクリプトや、外部コンポーネントのモックを含む統合的なテストケースを作成します。
  - **状態遷移テスト**: `WHILE` 条件が満たされている状態（ON）と外れた状態（OFF）の両方でテストを実行し、意図した通りに振る舞いが分岐することをアサートします。

### 2.4 Unwanted Behavior（望ましくない振る舞いの要件）

- **構文**: `IF <trigger>, THEN the <system> SHALL <system response>`
- **特性**: 不正な入力、システムエラー発生時、例外的な状況での回復処理、フォールバック、エラー通知、リトライ挙動を定義します。
- **導出パターン**:
  - **Error Path Testing (異常系テスト)**: 意図した通りに特定の例外クラスがスローされるか、ユーザーへのエラーメッセージが適切か、監査ログ出力が行われるかを検証します。
  - **モックによる障害注入**: インフラ障害やトランザクション競合（データベースの OCC 競合など）をモックで意図的に発生させ、システムのフェイルセーフ機構やリトライロジックが正常に機能することを確認するテストを導出します。

### 2.5 Optional Feature（オプション機能要件）

- **構文**: `WHERE <feature is included>, the <system> SHALL <system response>`
- **特性**: 機能トグル（Feature Toggle）や特定のライセンス構成が有効な場合のみ発動する挙動を定義します。
- **導出パターン**:
  - 機能トグルの ON/OFF のそれぞれの設定値でテスト環境をパラメータ化し、実行環境の構成に応じた振る舞いの変化や、機能制限の有効性を検証するテストを導出します。

## 3. verification_method との対応関係

EARS から導出されたテストは、要件の Frontmatter で指定された統制語彙 `verification_method` にマッピングして実装されなければなりません。なお、各メソッドの正確な定義やグリーン（合格）の基準は、すべて `verification.md` を参照してください。

本ドキュメントでは、EARS の種別からどの `verification_method` を選択すべきかの推奨対応を示します。

| EARS 節種別 | 推奨する verification_method | 具体的な導出テストパターン |
|-------------|----------------------------|--------------------------|
| Ubiquitous | `pbt` または `example-test` | 不変条件のプロパティベーステスト、単体テスト |
| Event-Driven | `example-test` | BDDシナリオ、入出力例を用いた単体テスト |
| State-Driven | `example-test` | 状態をセットアップした結合・統合テスト |
| Unwanted Behavior | `example-test` | 例外発生、異常系の単体・統合テスト |
| (性能・容量要件) | `benchmark` / `load-test` | SLA・閾値をアサートするパフォーマンステスト |
| (セキュリティ要件) | `sast` / `dep-audit` | 静的解析ルール、依存関係スキャン |

## 4. 検証結果の .spec/ への記録書式

テスト仕様の導出が完了した後、そのテスト群がどの要件をカバーしているかのトレース情報は `.spec/` 内の所定のドキュメントに正確に記録されなければなりません。

### 4.1 テストコード側への要件 ID リンクの付与

導出されたすべての実テストケース（テスト関数名、シナリオタグ、テストファイル名など）には、対象となる要件 ID を明示的に含める規則を徹底します（例: `@Tag("FR-012")` や `def test_FR012_...`）。
これにより、テストランナーの実行結果（JSON や XML レポート）から要件のステータスを直接紐付け、Traceability Matrix の証跡とすることが可能になります。

### 4.2 トレース情報と結果の記録フォーマット

テスト仕様を記録するドキュメント（`.spec/specs/<feature>/` 配下のテスト仕様書ファイル）には、必ず以下の定型 Markdown フォーマットを使用して、導出元要件とテストの枠組みを記述します。

```markdown
### テスト仕様: FR-012 トークン有効期限の処理

- **対象要件**: FR-012
- **EARS 節**: IF token is expired, THEN the system SHALL return 401 Unauthorized.
- **導出元種別**: Unwanted Behavior
- **Verification Method**: example-test
- **テストケース一覧**:
  - `test_FR012_expired_token_returns_401`
  - `test_FR012_invalid_signature_returns_401`
- **検証ステータス**: (CI/CD などの実行パイプラインによってここに結果が自動収集され、 verified 判定の根拠として利用されます)
```

このフォーマットを遵守し維持することで、実装規律に基づく要件紐付けと連動し、機械的な `verified` 判定（`lifecycle.md` に規定される）に必要不可欠な証跡をプロジェクト全体で完全に揃えることができます。
