# 論理データモデル設計

データ構造の表現と制約の定義を行うための指針です。リレーショナルデータベース（RDB）に限らず、NoSQL、あるいはファイルベースのストレージを用いる場合でも、データの論理的な関係性と一貫性を担保するための設計が含まれます。本プロセスにおける最終的な成果物は `.spec/design/data-model.md` として出力することを前提としています。

## 目的と前提

システムのデータをどのように保存し、どのように参照するかは、アーキテクチャ全体のパフォーマンスと保守性に直結します。
物理的なストレージの特性（RDB のテーブル設計や NoSQL のコレクション設計）に踏み込む前に、まずはビジネスの要求に基づいた論理的なデータモデルを構築することが不可欠です。

- 対象読者: アーキテクト、データエンジニア、バックエンド開発者
- 入力となる情報: 要件定義書、API 仕様のドラフト、ユースケース記述
- 制約事項: 特定の製品（MySQL、MongoDB など）の機能に依存した論理設計は行わないこと。

## ドメインモデルとの対応付け

システムが扱うデータ構造は、ビジネスロジックの根幹であるドメインモデルと密接に連動して設計する必要があります。
Domain-Driven Design（DDD）の概念を適用し、データの永続化単位を設計します。

### Aggregate（集約）の設計
- データの変更におけるトランザクションの境界となる単位です。
- データストアへの読み書きは、原則としてこの Aggregate 単位で行われます。
- 異なる Aggregate 間の整合性は、即時の一貫性ではなく、Eventual Consistency（結果整合性）によって担保することを検討します。
- 一つの Aggregate が大きくなりすぎると、更新時のロックの競合が発生しやすくなるため、必要最小限の境界を見極めることが重要です。

### Entity（エンティティ）の抽出
- 一意の識別子（ID）を持ち、ライフサイクルを通じて同一性が保たれるオブジェクトです。
- データモデルにおける主要なテーブルや、ドキュメント型ストレージのルート要素に相当します。
- Bounded Context（境界づけられたコンテキスト）ごとに、同じ概念（例: 「ユーザー」）でも、コンテキストが異なれば（「決済コンテキスト」と「配送コンテキスト」）、異なる Entity として定義すべきかを見極めます。

### Value Object（値オブジェクト）の構造化
- 識別子を持たず、属性の集合によってのみ定義される不変のオブジェクトです。
- データモデルにおいては、Entity に内包されるカラム群、あるいはネストされた JSON オブジェクトとして表現されます。
- 例: 「住所」や「金額と通貨のペア」などは個別の Entity として扱わず、Value Object として埋め込む設計を優先し、意図せぬ副作用を防ぎます。

## エンティティ整合性とデータのルール

データの品質を維持するためには、アプリケーション全体で Entity Integrity（エンティティ整合性）が厳格に担保される必要があります。
特定のデータベース製品の制約機能（外部キー制約やトリガーなど）に依存せず、論理的なルールとして以下を明確に文書化します。

### 識別子の設計
- 各 Entity がどのように一意に識別されるかを決定します。
- ビジネス上の意味を持つ Natural Key（自然キー）と、システムが生成する Surrogate Key（サロゲートキー）の使い分け基準を定義します。
- 分散システムにおいては、単一障害点となるシーケンスの採番方式を避け、UUID v4 や v7、あるいは Snowflake ID などの分散採番方式を採用するかどうかもここで論理的に決定します。

### 関連と多重度の定義
- Entity 間の関係性（1対1、1対多、多対多）を明確にします。
- RDB の場合、多対多は中間テーブルとして解決されますが、論理モデルの段階ではビジネス上の関連性を正確に捉えることに注力し、中間テーブルの存在自体は物理モデルまで遅延させても構いません。
- NoSQL やファイルベースの設計では、関連を「参照（IDの保持）」として持つか、「埋め込み（ネスト）」として持つかを決定する重要な判断材料となります。

### ドメインルールと制約の担保
- 取りうる値の範囲、文字列のフォーマット、必須項目、特定の状態遷移に伴うデータ制約をリストアップします。
- これらの Business Rule（ビジネスルール）がどこで検証されるかの責任分界点を設計します。
  - データベース層（CHECK 制約や型定義）
  - アプリケーション層（ドメインモデルの生成時バリデーション）
  - API エンドポイント層（リクエストスキーマの検証）

## Normalization（正規化）と非正規化の判断

データを設計する際は、まず論理的な冗長性を排除する Normalization（正規化）のプロセスを経ます。
しかし、要件やワークロード特性によっては、あえて非正規化を採用する合理的な理由が存在するため、その判断基準を明確にします。

### 正規化を基本とするケース
- データの更新異常（Update Anomaly）や削除異常を防ぐため、原則として第3正規形（3NF）までを検討のベースラインとします。
- データの整合性が最優先される基幹系システムや、金融トランザクションなど、更新頻度（Write）が極めて高いワークロードに適しています。
- 「単一の事実が単一の場所にのみ存在する（Single Source of Truth）」ことを保証し、運用時のデータ不整合リスクを最小化します。

### 非正規化を許容・推奨するケース
- 参照系の Read（読み取り）レイテンシが厳格な非機能要件として設定されており、複数エンティティの結合（JOIN）コストが許容できない場合。
- Document Database（ドキュメントデータベース）や Key-Value Store（KVS）などの NoSQL を採用し、データのアクセスパターンが「常に特定の情報群をまとめて読み書きされる」と判明している場合。
- 履歴データや監査ログなど、一度書き込まれた後に更新されることがない（Immutable な）時系列データである場合。

非正規化を採用する場合はその判断を意図的なものとし、データの重複によって生じる結果整合性の遅延を許容できるか、あるいはデータ同期のメカニズム（イベント駆動アーキテクチャによるバックグラウンド更新など）をどう担保するかをセットで設計・明記する必要があります。

## ER図（Mermaid erDiagram）の作成基準

設計した論理モデルは、関係者間で視覚的に共有し、合意形成を図るために Mermaid の `erDiagram` 構文を用いて記述します。
作図の際は、以下の規律に必ず従ってください。

### 表記規則と構文ルール
- エンティティ名や属性名に日本語などの非 ASCII 文字を用いる場合は、必ず引用符（`" "`）で囲むこと（Mermaid のパースエラーを防ぐため）。
- 属性（カラムやフィールド）のデータ型を明記すること（例: `string`, `int`, `boolean`, `datetime`）。
- 主キーには `PK`、外部キーや他のエンティティへの論理的な参照には `FK` のマーカーを付与すること。

### 関係性の明示とラベル付け
- カーディナリティ（`||--o{` : 1対0以上の多、`||--||` : 1対1、など）を正確に使い分け、ビジネス上のドメインルールを視覚的に表現すること。
- 関連を示す線（エッジ）のラベルには、簡潔な動詞や役割（例: `contains`, `managed_by`, `assigned_to`）を記述し、単なる線以上の意味を持たせること。

### 図の分割と認知負荷の管理
- 図のノード数が増え、複雑になりすぎる場合は、Ubiquitous Language（ユビキタス言語）の境界となる Bounded Context ごとに図を論理的に分割し、読み手の認知負荷を下げること。
- 1つの図に含めるエンティティは最大でも 15〜20 程度を目安とします。
- システム全体の巨大な一枚絵（Big Picture）を作成することは避け、必要に応じて主要なクリティカルパスに関わるコアエンティティのみを抽出して描画します。

この論理データモデルの設計プロセスを経ることで、特定のデータベース製品や物理的なストレージの制約に縛られない、堅牢でビジネス要件に合致したデータ基盤の確固たる基礎を確立します。
