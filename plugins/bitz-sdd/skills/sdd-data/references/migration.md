# スキーマ変更・データ移行計画

システムの継続的進化において、データベースのスキーマ構造やストレージ内のファイル構造の変更は避けられません。
本ドキュメントは、システムのダウンタイムを最小限に抑えつつ、安全にデータ構造を移行・変更するための非破壊的な戦略と具体的な手順を定義します。
最終的な移行計画のアウトプットは `.spec/design/data-storage.md` に記載されます。

## 目的と前提

稼働中の本番環境において、数時間のダウンタイムを伴う一括更新（Big Bang Migration）は、ビジネス要件上許容されないケースが大半です。
特定の製品の移行ツールに依存する前に、データ移行の全体像を設計し、無停止あるいは最小の停止時間で安全にスキーマを進化させるアプローチを計画します。
この計画は RDB のテーブル変更だけでなく、NoSQL のドキュメント構造変更や、オブジェクトストレージ間のデータ移動にも広く適用可能です。

## スキーマ変更パターン

スキーマの変更は、アプリケーションコードのデプロイメント・ライフサイクルと密接に絡み合っています。
安全性を担保するため、原則として段階的な移行アプローチを採用します。

### Expand-Contract Pattern（拡張・縮小パターン）

破壊的なスキーマ変更（例：カラムの意味の変更、テーブルの分割、データフォーマットの移行）を行う場合、以下のフェーズを厳格に分けて実行します。
この手法は Parallel Change（並行変更）とも呼ばれ、各フェーズ間でいつでも後戻り可能な状態を維持します。

1. **Expand（拡張）フェーズ**:
   - データストアに対し、新しい構造（新しいカラム、新しいテーブル、あるいは新しいバケット）を追加します。この時点では既存の構造は一切変更・削除しません。
   - スキーマの拡張が完了しても、デプロイされているアプリケーションはまだ古い構造のみを参照・更新しています。
2. **Dual Write（二重書き込み）とデータ同期フェーズ**:
   - アプリケーションコードを更新し、古い構造と新しい構造の**両方に**データを書き込むようにします。
   - 同時に、バックグラウンドで独立した移行バッチ処理（Migration Job）を実行し、過去に作成された既存データを新しい構造に合わせて変換・コピーします。
3. **Read 切替フェーズ**:
   - 過去データの移行が 100% 完了し、新旧両者のデータの整合性が完全に取れていることが確認できた後、アプリケーションの Read（読み取り）処理を新しい構造へと切り替えます。
   - 万が一読み取り側で問題が発生した場合でも、旧構造への書き込みは継続されているため、即座に Read の向き先を元に戻す（ロールバックする）ことが可能です。
4. **Contract（縮小）フェーズ**:
   - 新しい構造のみでシステムが正常稼働していることを一定期間監視します。
   - 安定稼働が確認された後、不要になった古い構造への書き込み処理をコードから削除し、最終的にデータストアから古いスキーマ定義・データを物理的に削除して容量を解放します。

## 移行とロールバック計画

データ移行は、ネットワークエラーや想定外のデータフォーマットによって途中で失敗するリスクを常に内包しています。
そのため、前進する実行計画と同じ粒度でリカバリプランを策定する必要があります。

### 冪等性（Idempotency）の確保
- データ移行スクリプトやバッチ処理は、何度実行しても最終的なデータ状態が同じになるよう、Idempotent（冪等）に設計します。
- 処理が途中で中断・失敗した場合でも、途中から安全に再実行（リトライ）できるように、処理済みのマーカー（タイムスタンプやステータスフラグ）による進捗管理を導入します。

### ロールバック手順の定義
- 移行処理中に致命的なエラーが発生し、データが不整合な状態になった場合、システムを以前の健全な状態に巻き戻す（ロールバックする）ための具体的な手順を事前に用意します。
- データの物理的な逆変換スクリプト（Down Migration）の用意や、移行開始前に取得したスナップショット・バックアップからのリストア計画を策定し、復旧目標時間（RTO）を見積もります。

## 移行の検証手順

移行が「成功した」と判断するための客観的かつ定量的な検証プロセスを設計します。
移行完了後の本番稼働でデータの欠損やパフォーマンス劣化が発覚する事態を未然に防ぎます。

### バックグラウンド検証とシャドウリード
- Dual Write フェーズにおいて、新しい構造に書き込まれたデータと古い構造のデータを非同期で比較し、不一致がないかを継続的に監視します。
- 実稼働の Read リクエストの裏側で新しいデータ構造へのクエリも同時に実行し、結果を比較するシャドウリード（Canary Testing）を実施することで、メイン処理に影響を与えることなく移行ロジックの正確性をテストします。

### パフォーマンスの再評価
- データ構造の変更に伴い、クエリの実行計画やインデックスの効き方が変化する可能性があります。
- 移行後のスキーマに対して、本番相当のデータボリュームを用いた負荷テストや、実行計画（Explain 等）の解析を実施し、レイテンシ要件やスループット要件を満たしているか確認します。
- パフォーマンスの劣化が確認された場合は、Contract フェーズに入る前にインデックスの再設計やチューニングを完了させます。
