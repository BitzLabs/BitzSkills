# データ格納方式の選定

本ドキュメントは、システムが扱うデータの特性に合わせて、最適なストレージ技術を選定するためのガイドラインです。
特定のデータベース製品ありきで設計を始めるのではなく、要件とワークロードの証拠に基づいて技術を決定します。
この選定プロセスにおける最終的な結果は `.spec/design/data-storage.md` に記載されることを前提としています。

## 目的と前提

システムの信頼性やパフォーマンスは、採用するデータ格納方式に大きく依存します。
Relational Database（RDB）が常に最適解とは限らず、用途によっては NoSQL、あるいはシンプルなファイルシステムが適している場合もあります。
本設計では、プロジェクトの成果物が示唆する非機能要件（NFR）や要件定義を根拠とし、論理的な証拠駆動（Evidence-Driven）のアプローチで選定を行います。

## ワークロード特性の把握

データ格納方式を決定する前に、システムが対象とするデータの振る舞いやアクセスパターンを正確に把握する必要があります。
以下の観点から、対象システムのワークロード特性を詳細に分析します。

### アクセス傾向と比率
- **Read/Write 比率**: システムは読み取り中心（Read-heavy）か、書き込み中心（Write-heavy）か。
- **レイテンシ要件**: ユーザー体験において、数ミリ秒単位の極めてシビアな応答が求められるか、あるいはバッチ処理のように全体のスループットが重視されるか。
- **同時接続数**: ピーク時にどの程度のクライアントからの同時アクセスが発生するか。

### データの構造と一貫性
- **データ構造の複雑さ**: データは高度に構造化されリレーションが複雑か、それともフラットなログ形式か、不定形な JSON ドキュメントか。
- **トランザクション境界**: 複数の Entity（エンティティ）にまたがる厳密な ACID トランザクションが必要か、あるいは Aggregate（集約）単位での結果整合性で十分か。
- **検索要件**: 単純なキーによるルックアップ検索か、複数のカラムにまたがる複雑な集計クエリや全文検索が必要か。

### スケーラビリティとデータ量
- **データボリューム**: 初期想定のデータ量（GB/TB）と、年間の増加予測はどの程度か。
- **ライフサイクル**: データは永続的に保持されるべきか、一定期間後に自動的に揮発してよい（TTLが存在する）か。

## 候補となる格納方式の列挙

特定された要件を満たしうる格納方式のカテゴリを列挙します。ここでは特定の製品名（MySQL, MongoDB 等）ではなく、技術のアーキテクチャ特性に注目して分類します。

1. **Relational Database (RDB)**
   - 複雑なクエリ要件、厳密なスキーマ定義、複数テーブルにまたがる強力な ACID トランザクションが要求される場合。
2. **NoSQL (Document / Key-Value / Column-family / Graph)**
   - スキーマレスなデータ構造、極めて高い書き込みスループット、単一キーによる高速な読み取り、あるいはデータ間の複雑なネットワーク構造の探索が必要な場合。
3. **File System（ファイルシステム） / Object Storage（オブジェクトストレージ）**
   - 画像、動画、巨大なログファイル、あるいはエクスポートされたレポートなど、バイナリデータや非構造化データの保存先として機能します。
4. **Cache（キャッシュ）**
   - 一過性のデータ、セッション情報の保持、あるいは背後にあるプライマリストレージへの Read 負荷を劇的に下げるための中間インメモリ層として用いられます。

## 証拠駆動の技術適合性評価

候補技術の採用可否は、プロジェクト内の既存の要件や設計成果物に基づく**証拠（シグナル）**から判断しなければなりません。
雰囲気や一般的な流行で選定してはなりません。以下のフォーマットを用いて評価プロセスをドキュメント化します。

### 評価フォーマット

各ストレージ候補について、以下の項目を埋める形で評価表を作成します。

- **対象カテゴリ**: （例: ドキュメント型 NoSQL、オブジェクトストレージ など）
- **適合性**: `High` / `Medium` / `Low` / `None` のいずれか。
- **根拠シグナル**: 
  - この評価を下した具体的な設計要件や非機能要件（NFR）を挙げます。
  - **`High` をつける場合、プロジェクトのドキュメントから具体的な要件（例: 「Read 10,000 QPSが必要」「スキーマがユーザー定義で動的に変わる要件がある」）の引用が1つ以上必須です**。引用・提示できなければ自動的に `Low` または `None` と評価します。
- **判断**: `Adopt`（採用） / `Conditional`（条件付き採用） / `Reject`（却下）
- **理由**: 
  - なぜその判断に至ったのかを、技術的背景とともに記述します。
  - `Adopt` の場合、システムのどのコンテキストやアーキテクチャのレイヤーで使用されるかを明示します。
  - `Reject` の場合、代わりに既存のどの成果物や技術がその役割を代替・不要にするのかを論理的に説明します。
- **条件/リスク**: 
  - `Conditional` の場合、最終的に `Adopt` に昇格させるために真であるべき事柄や、検証が必要な項目（Open Question や TBD）を明記します。

### 評価の規律

- **熱意より証拠**: 「新しい技術だから使いたい」という開発者の意向は選定理由から排除します。「`Reject` + 理由の明文化」は、後任の設計者にとって非常に価値のある立派な成果物です。
- **制約要件との照合**: 「マネージドサービスのみ許可される」「特定のクラウドベンダーにロックインされないこと」といったシステム上の制約（CON要件）が存在する場合、それに違反する技術は直ちに `Conditional` または `Reject` とし、要件衝突の事実を名指しで記載します。
- **決定の可逆性**: 技術選定の判断は、将来の要件変更に伴って覆る可能性があります。そのため、単なる結論だけでなく、可逆な入力として「当時のコンテキストと理由」をセットで記録し、再評価を容易にします。
