# ファイル・データ交換形式の設計

システムがデータを保存・交換する際のフォーマットやスキーマの定義手法に関するガイドラインです。
データベース内の論理設計とは異なり、システム間連携 API のペイロード、エクスポートされるファイル、データパイプラインを流れる非同期メッセージの形式などを主な対象とします。
本設計の結果は `.spec/design/data-storage.md` に集約されます。

## 目的と前提

現代の分散システムにおいては、複数の Bounded Context（境界づけられたコンテキスト）間で絶えずデータがやり取りされます。
適切なデータ交換形式を選定することは、システム間の結合度を下げ、パフォーマンスを最適化し、将来の拡張性を担保する上で極めて重要です。
本ドキュメントでは、目的に応じたフォーマットの選定基準と、スキーマの安全な進化戦略を定義します。

## スキーマ定義と形式の選定

データの特性、アクセスパターン、人間による可読性の要否、そしてシステム間の結合度に応じて、最適なデータフォーマットを選定する必要があります。
以下の基準に従って候補を検討します。

### データ形式の候補と適用基準

- **JSON (JavaScript Object Notation)**
  - 最も一般的で、Web API (REST API, GraphQL) や設定ファイル、ドキュメント指向DBでの格納形式として幅広く使われます。
  - 人間が読みやすくデバッグが容易である反面、パースコストが相対的に高く、バイナリ形式に比べてデータサイズが肥大化しやすいという欠点があります。
- **CSV / TSV**
  - 表形式のデータ出力、レガシーシステムとのフラットファイル連携、エンドユーザーによる表計算ソフトでの閲覧が要求される場合に採用します。
  - スキーマの自己記述性がないため、ヘッダー行の取り扱いや文字コード（UTF-8等）、改行コード、エスケープ処理の厳密な取り決めが必須となります。
- **Parquet / ORC (Columnar Formats)**
  - データ分析基盤やデータレイクにおいて、大量のデータを効率的に圧縮・スキャンする必要がある場合に選択します。
  - 複雑なネスト構造を維持しつつ列指向で保存されるため、集計クエリ（特定のカラムのみを読み込む操作）のI/Oパフォーマンスが劇的に向上します。
- **Protobuf / Avro / Thrift**
  - マイクロサービス間の gRPC 通信や、メッセージブローカーを経由する大容量のストリーミング処理において採用します。
  - バイナリ形式によるシリアライズの高速性と、定義ファイルによる厳密なスキーマによる型安全性が担保されます。

## Versioning（バージョニング）と後方互換性

データ形式はシステムの進化、あるいはビジネス要件の追加に伴って必ず変化します。
システム間の疎結合を維持するためには、破壊的変更を防ぐスキーマの進化（Schema Evolution）戦略が不可欠です。

### 後方互換性（Backward Compatibility）の担保
- 既存のコンシューマー（読み取り側）の処理が壊れないよう、**既存のフィールドを削除したり、データ型を変更したりすることは原則禁止**とします。
- 新しい要件が発生した場合は、新しいフィールドを「オプション項目（省略可能）」として追加することを基本戦略とします。
- 必須フィールドを追加する場合や、破壊的な変更がどうしても避けられない場合は、スキーマ自体のバージョンを明示的に上げ（例: API の `/v1/` から `/v2/` への変更）、旧バージョンと並行稼働させます。

### スキーマのライフサイクルと廃止
- 古い形式のデータやエンドポイントは、明確な廃止（Deprecation）のタイムラインを設けて管理します。
- コンシューマーの開発チームに十分な移行期間を提供し、ログ監視によって古いバージョンの利用率がゼロ（あるいは許容範囲内）になったことを客観的な指標で確認した上で、安全に古い形式を削除します。

## データの品質とスキーマ検証

データフォーマットの構造を静的に定義するだけでなく、実際に流れてくるデータがその定義を満たしているかを動的に検証するメカニズムを設計します。

### バリデーションの導入
- JSON の場合、**JSON Schema** 等を用いて構造、必須項目、値の制約（文字列長や正規表現パターン）を宣言的に定義し、システム境界（APIゲートウェイやアプリケーションの入力層）で機械的に検証（Fail-fast）します。
- CSV ファイルをバッチ処理で取り込む場合は、行数、カラム数、データ型の検証を最初に行い、エラー行のみをスキップして処理を続行するか、ファイル全体を不正としてリジェクトするかのエラーハンドリング方針を明確に決定します。

### スキーマレジストリの活用
- 非同期メッセージング基盤を用いたシステム間連携において、プロデューサー（送信側）とコンシューマー（受信側）の間でスキーマ定義の認識ズレを防ぐため、共有の Schema Registry（スキーマレジストリ）の導入を検討します。
- これにより、送信前のシリアライズ段階でスキーマの妥当性が検証され、フォーマットの不整合なポイズンメッセージがシステム内に混入することを水際で防ぐことが可能になります。
