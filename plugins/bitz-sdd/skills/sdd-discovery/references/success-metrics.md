# 成功指標（North Star Metric + 入力指標 + ガードレール）

ビジョンの「勝利の定義」を測定可能な形に変える。下流の優先順位付け（RICE の Impact・MoSCoW・ポジショニング）の錨になる。結論は docs/01-context/success-metrics.md（proposed、テンプレは assets/docs-draft-success-metrics.md）へ。

## North Star Metric（NSM）— 1つだけ

顧客に届いた中核価値を捉える単一指標。条件:

1. **収益に先行する（遅行しない）** — NSM の改善が将来の収益/維持を確実に引き上げる先行指標であること。収益そのものは遅行アウトプットであり NSM にしない
2. **顧客が受け取った価値を顧客の言葉で測る**（例: 「週次でワークフローを完了したアクティブチーム数」。「ログイン数」ではない）
3. **チームがプロダクトの仕事で動かせて**、定期的に追跡できること

アンチパターン: バニティ指標（生サインアップ数・PV）、純粋な収益、チームが影響できない指標。

## 入力指標（3〜5個）

NSM を駆動する、チームが直接働きかける少数の指標。定番の分解: **広さ × 深さ × 頻度 × 効率**（アクティブユーザー数 × 1人あたり操作数 × 利用頻度 × 成功率）。各入力指標はプロダクトの仕事で引けるレバーに対応させる。

## マッピング枠組み（レンズとして1つ選ぶ）

- **AARRR** — Acquisition → Activation → Retention → Referral → Revenue。NSM と入力指標をこのファネルに置き、戦略がどの段階に賭けているかを露出させる
- **HEART** — Happiness / Engagement / Adoption / Retention / Task success。選んだ次元ごとに Goals → Signals → Metrics。UX 中心のプロダクト向き

プロダクトに合う方を選ぶ。両方を機械的に埋めない。

## 目標値とガードレール

各指標に記録する: **定義 / 測定方法・ソース / 目標値 / ガードレール**（NSM を最適化する間に劣化させてはならない対抗指標 — 例: エンゲージメントを伸ばすために維持率やサポート負荷を犠牲にしない）。

**根拠のない目標値は `TBD`** として Open Questions へ — 発明しない。

## 下流への接続

ガードレール指標と目標値は、Design Gate 後に NFR（verification_method: benchmark / load-test、数値必須）へ派生する第一候補。数値を書くときは検証可能な形（p95 / 期間 / 母集団の明示）を意識する。
