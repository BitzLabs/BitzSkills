# ペルソナとジャーニー（JTBD → ペルソナカード → 段階×レイヤー）

人口統計ではなく **Jobs-to-be-Done（ジョブ理論）** を軸にペルソナを作り、主要ペルソナごとにジャーニーをマップする。結論は docs/00_はじめに/ペルソナ・ジャーニー.md（proposed、テンプレは assets/docs-draft-personas-journeys.md）へ。

## Jobs-to-be-Done（JTBD）

人は状況の中で前進するためにプロダクトを「雇う」。ユーザー属性ではなくジョブを捉える。

- **ジョブストーリー形式**: 「〜のとき（状況）、〜したい（動機）、〜できるように（期待成果）」
- ジョブの3次元をカバーする: **機能的**（タスク）・**感情的**（どう感じたいか）・**社会的**（どう見られたいか）
- ジョブはソリューション非依存で安定。機能はそうではない。ペルソナをジョブに繋留すればピボットに耐える

## ペルソナカード（1人ずつ）

- **名前/アーキタイプ** と1行要約（役割。「みんな」は禁止）
- **文脈と行動** — 関係する状況。バニティな人口統計は書かない
- **Jobs** — このペルソナが追うジョブストーリー
- **Pains** — ジョブ達成の障害・摩擦・リスク
- **Gains** — 望む成果と便益
- **生の声** — ユーザー自身の言葉（リサーチ由来。なければ **[proto / 未検証]** と明記）

## proto と research-based（確度に正直に）

- **proto ペルソナ** — 仮定と机上調査の組み立て。未検証の主張すべてに **[proto]** を付ける。AI は骨組みは得意だが感情の妥当性は弱い — 発明した感情や台詞を事実として提示しない
- **research-based** — インタビュー/データに裏づけあり。証拠が届いたら proto の仮定を引用つき事実に置換して昇格させる
- **人口統計や台詞を決して捏造しない**。不明は `TBD`

**主要ペルソナ**（プロダクトが最適化する1人）を特定し、secondary / served / anti-persona と区別する。下流はまず主要ペルソナに集中する。

## ジャーニーマップ（主要ペルソナごとに1枚以上）

**段階（列）** — 製品に合わせて調整する既定の背骨: 認知 → 検討 → 購入/登録 → オンボーディング → 利用 → 継続/更新 → 推奨

**レイヤー（行）** — 全段階について:

- **タッチポイント/チャネル** — どこで相互作用するか（web / アプリ / メール / 営業 / サポート）
- **行動** — ジョブを前進させるために何をするか
- **思考と感情** — 可能な限り**生の声**で + 感情曲線（高/中/低）で谷を可視化
- **ペイン** — 摩擦・離脱リスク・期待とのずれ
- **機会** — ペインを除去し gain を増幅する方法

## Moments of Truth（決定的瞬間）

- **ZMOT** — 接点到達前の下調べ / **FMOT** — 最初の直接接触・第一印象 / **SMOT** — 約束を裏づけるか壊すかの実利用体験 /（任意）**UMOT** — 体験の共有 → 推奨へ

感情曲線が MoT で沈む箇所 = 最もレバレッジの高い改善点。

## 規律

- すべての Pain / 機会をペルソナ/ジョブに紐づける（根拠なしのマップにしない）
- マッピングの成果物は**優先順位付きの機会リスト**であって、きれいな図ではない
- 機会リストは後段のドメインストーリー（`sdd-design`）とスコープ（RICE の Reach/Impact）の入力になる
