# 失敗プロトコルとエスカレーション

検証 red を検知したら、**行動する前に失敗を三分類する**。

## 三分類

| 分類 | 判定基準 | 許された行動 |
|------|----------|--------------|
| **code-bug** | 仕様は明確で、実装が満たしていない | 自己修復リトライ（上限2回） |
| **spec-bug** | 仕様が矛盾・曖昧・実現不能 | **リトライ禁止**。spec-issue 起票 → 当該タスクを blocked にし、依存しない他タスクへ移る |
| **env-failure** | flaky・環境起因 | 1回だけ再実行。再発なら STATE.md に記録して人間へ |

## リトライ規則

- code-bug の自己修復は **1タスクあたり2回まで**。3回目の red で自動エスカレーション
- リトライごとに STATE.md へ「前回何を変え、なぜ失敗したか」を追記（同一修正の反復ループ防止）

## 権限分離の要（絶対規則）

spec-bug と判定しても requirements/ を**1文字も変更しない**。できるのは spec-issues/ への提案のみ。提案には次を含める:

- どのEARS文が、何と矛盾するか
- 提案する修正内容
- bump / supersede の自己判定（「緑を赤にし得るか」基準）
- `spec_inspect.py --impact` の結果貼付

これを緩めると検証中心主義は「実装に合わせて採点基準を書き換える自己採点」に退化する。本フレームワークで最も譲れない規則。

## エスカレーション報告の書式

停止時は1つの報告にまとめる: ①失敗分類 ②試行履歴（STATE.mdから要約） ③spec-issue（あれば） ④推奨アクション。

## 知見還流

人間の裁定後、**一般化できる教訓がある場合のみ** `docs/05_リリース・運用/教訓.md` へ1エントリ追記する（書くのは人間。エージェントはドラフト提示まで）。「一般化できる」の基準: プロジェクト固有の状況説明を消しても意味が通るか。これは Gate を待たない即時還流を許す唯一のケース。
