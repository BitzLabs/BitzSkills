# 並列実行と git 運用・権限マトリクス

## ブランチ規約

- feature ブランチ = `.spec/specs/<feature>/` と1対1対応
- **STATE.md はブランチローカル**（`.gitattributes` で `merge=ours`）。living memory のブランチ間共有は文脈汚染を招く
- requirements/ への変更（採番・状態遷移）は main 直コミットまたは最優先マージ。契約は全ブランチから同一に見えること

## マージ競合の構造的回避

- 1要件1ファイルなので異なる要件への変更は物理的に衝突しない
- `_index.md` / `_counter.md` は自動生成物。競合したら**再生成で解決**（手マージ禁止）
- 並列PR・worktreeの検証は `spec_inspect.py --check-only` を使い、
  `inspection-report.md` を生成・更新しない。全PR統合後、締め工程の専用ブランチで通常検査を
  1回実行してレポートを更新する。各作業ブランチでレポート差分をコミットしない

## 並列サブエージェントのスコープ

- tasks/ の依存グラフで `depends_on` が空のタスク群のみ並列投入
- 各サブエージェントの書き込み範囲 = 担当タスクの `boundary:` 列挙パス + テストディレクトリ + STATE.md 追記
- feature 単位のコスト予算（トークン/時間）を tasks/ メタデータに宣言。超過で全停止→人間へ

| 要件本文の変更 | ✓ | ✗ | ✗ | ✗ |

| tasks/ 生成 | ✓ | ✓ | ✗ | ✗ |
| 実装コード書き込み | ✓ | ✗ | ✓（boundary内） | ✗ |
| verified 判定 | ✗ | ✗ | ✗ | ✓（機械） |
