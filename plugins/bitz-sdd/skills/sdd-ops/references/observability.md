# 可観測性・SLO 設計（Google SRE）

サービスレベル目標を顧客期待から導き、監視・アラートに落とす。結論は docs/05_リリース・運用/運用・リリース.md の proposed ドラフトへ。

## SLI / SLO / SLA — この順で導出する

1. **SLI（指標）** — サービス健全性の測定シグナル（リクエスト成功率・p99 レイテンシ・鮮度など）。**何を・どう測るか**を定義する
2. **SLO（目標）** — SLI に対する内部目標（例: 30日間で成功率 99.9%）。チームは SLO に向けて運用する。目標は**顧客期待**（ポジショニング/便益）から導き、丸い数字に合わせない
3. **SLA（合意）** — 対外的に約束するレベル（違反時の帰結つき）。**SLO = SLA − バッファ**: 内部目標は常に対外約束より厳しく

サービスをクリティカリティで階層化する: critical / standard / best-effort。SLO は critical から定義する。

## エラーバジェット

`エラーバジェット = 1 − SLO`（99.9% → 0.1% の許容不能時間）。バジェットは**リスクと速度のトレードオフを統治する**: 残っている間はリリースに使い、使い切ったらリスクの高い変更を凍結する。critical サービスごとにバジェットと凍結ポリシーを明文化する。

## NFR カテゴリ（数値 + 由来 SLO を必ず併記）

| カテゴリ | 例 |
|---|---|
| 可用性 | 期間あたり uptime %（可用性 SLO 由来） |
| レイテンシ | 主要操作ごとの p95 / p99 |
| スループット | 定常/ピークの処理数 |
| エラー率 | 許容失敗率の上限 |
| 耐久性 | データ損失許容度 |
| RPO / RTO | 復旧目標（references/disaster-recovery.md と整合させる） |

**すべての NFR 候補は SLO に遡れること**。SLO の根拠がない数値は疑わしい。根拠がなければ `TBD` + 前提を記録（発明しない）。

## 監視・トレーシング・ログ・アラート

- **メトリクス設計**: リクエスト駆動サービスは RED（Rate / Errors / Duration）、リソースは USE（Utilization / Saturation / Errors）
- **分散トレーシング**: OpenTelemetry 等での相関 ID 伝播・サンプリング戦略（分散構成の場合）
- **ログ集約**: 構造化ログ・一元管理・相関 ID の埋め込み
- **アラート**: SLO/エラーバジェットに基づく閾値、エスカレーション経路、**各アラートに対応ランブックへの参照**を必須にする。SLO に紐づかないアラートは作らない（アラート疲れの温床）
- **ヘルスチェック**: 全サービスに liveness / readiness の定義

## 下流への接続

SLO・エラーバジェット・レイテンシ目標は Design Gate 後に NFR（verification_method: benchmark / load-test、数値必須）へ派生する第一候補。検証条件（負荷・期間・環境）まで書けると派生が速い。
