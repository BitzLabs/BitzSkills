# セキュリティ設計

docs/03_設計仕様/セキュリティモデル.md の proposed ドラフト（信頼境界・脅威・統制）を運用面から埋めるためのチェックリスト。

## 設計項目

### 認証（AuthN）

- ユーザー認証: OAuth2 / OIDC、IdP の選定（自前実装しない）、セッション/トークンの寿命
- サービス間認証: mTLS またはワークロード ID
- 機械アクセス（CI・バッチ・外部連携）の認証経路

### 認可（AuthZ）

- モデル選定: RBAC / ABAC とその根拠、ポリシーの置き場所（コード内 / ポリシーエンジン）
- **最小権限**を既定にする: ロール一覧と各ロールが「できないこと」の明記
- テナント分離（マルチテナントの場合）: データ層とアプリ層のどちらで守るか

### シークレット管理

- 保管: Vault / クラウド KMS / シークレットマネージャ（**ハードコード資格情報ゼロ**が lint 対象になる前提で書く）
- ローテーション戦略: 何を・どの周期で・自動か手動か
- 漏えい時の失効手順

### ネットワーク/ゼロトラスト

- セグメンテーション: 信頼境界ごとの分離、境界を越える通信の明示的許可
- 「内側だから信頼する」を作らない: 内部通信にも認証・暗号化

### 監査ログ

- 記録内容: **誰が・何を・いつ**（対象操作の列挙）
- 改ざん検知可能性: append-only / 署名 / 外部転送のいずれで担保するか
- 保持期間と閲覧権限

### コンプライアンス

- 該当規制（個人情報・業界規制）の列挙と、各統制への対応表
- データ分類（機密/内部/公開）と分類ごとの取り扱い規則

## 規律

- 統制の数値・閾値（ローテーション周期・保持期間）は根拠つきで。なければ `TBD`
- 統制は後段で NFR（sast / dep-audit / manual-check）に派生する前提で、検証可能に書く
- 恒久判断（IdP 選定・シークレット基盤）は ADR ドラフトへ
