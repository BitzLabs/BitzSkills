# 災害復旧（DR）設計

サービス階層ごとに復旧目標を定め、検証可能な復旧手段に落とす。結論は docs/05-operations/OPERATIONS.md の proposed ドラフトへ。

## RTO / RPO（サービス階層別に定義する）

- **RTO**（Recovery Time Objective）— 復旧までに許容できる停止時間
- **RPO**（Recovery Point Objective）— 許容できるデータ損失の時間幅

critical / standard / best-effort の階層ごとに目標を置き、**提案中のアーキテクチャで達成可能か**を確認する（単一リージョン構成で RTO 5分は書けない）。数値は顧客期待・SLA と整合させ、根拠がなければ `TBD`。

## バックアップ戦略

- 対象・頻度・保持期間（RPO から逆算する）
- **リストア試験の計画**: 「バックアップがある」ではなく「リストアできることを定期的に確認している」を要件にする
- バックアップ自体の保護（別アカウント/リージョン・不変ストレージ）

## フェイルオーバー設計

- 単位: クロス AZ / クロスリージョンのどこまでやるか（コストと RTO のトレードオフを明示）
- 自動か手動か・障害検知から切り替えまでの時間
- フェイルバック（戻し）の手順も設計する — 切り替えっぱなしにしない

## ランブック

- 障害シナリオごとの復旧手順（**責任者・判断基準つき**）
- アラートからランブックへの参照を張る（observability 設計と整合）
- 手順は実行可能な粒度で（「確認する」ではなくコマンド/画面まで）

## 復旧テスト計画

- 定期的な DR 訓練（テーブルトップ → 実切り替え）
- 可能ならカオスエンジニアリング（障害注入）で検証を自動化する
- テスト結果は LESSONS_LEARNED / metrics に還流する

## 下流への接続

RTO/RPO は Design Gate 後に NFR（verification_method: load-test / manual-check）へ派生する。manual-check にする場合は手順（リストア試験・DR 訓練）を要件本文に列挙できる形で書く。
