# インフラ構成設計

プロダクショングレードのインフラ構成を設計するためのチェックリスト。規模に応じて縮退させる（単一 VM / PaaS で足りるならコンテナオーケストレーションを持ち込まない — 過剰設計はレビューの operations 観点で減点対象）。

## 設計項目

### コンテナ/オーケストレーション（採用する場合）

- クラスタ構成: ノードプール分割・リソースクォータ・namespace 戦略
- デプロイ戦略: rolling / blue-green / canary とトラフィック移行計画
- 自動スケール（HPA 等）と可用性維持（PodDisruptionBudget 等）
- ロールバック手順: 自動トリガ条件と所要時間の見積もり

### ネットワーク

- 境界設計: Ingress/Gateway・ロードバランサ・TLS 終端の位置
- サービス間通信: mTLS の要否・NetworkPolicy によるセグメンテーション
- 外部公開面の最小化（公開が必要な経路の列挙と、それ以外の遮断）

### IaC 方針（コードは書かない — 方針だけ決める）

- ツール選定と根拠（Terraform / Pulumi / CloudFormation 等）→ ADR ドラフト
- モジュール分割方針・state 管理（リモート state・ロック）
- 手作業変更の禁止範囲と例外手順

### マルチ環境

- 環境の段（dev / staging / prod）と昇格経路
- 環境差分の管理方法（overlay / 変数、環境ごとの分岐を最小に）
- 本番との等価性: staging が何を保証し、何を保証しないかを明記

## 規律

- 各設計判断をアーキテクチャの根拠（どのコンテキスト/NFR がそれを要求するか）に紐づける。根拠を引用できない構成要素は入れない
- 恒久判断（オーケストレーション採否・IaC ツール・環境構成）は ADR ドラフトへ
- 結論は docs/05-operations/OPERATIONS.md の proposed ドラフトに、作業表は `.spec/design/infra/` に置く
