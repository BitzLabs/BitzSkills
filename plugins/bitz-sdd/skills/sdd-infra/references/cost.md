# コスト見積もり

多次元のコスト見積もりを作る。結論と前提は `.spec/design/infra/cost-estimate.md`（assets/cost-estimate.md をコピー）へ。**単価・数量の根拠がないコストは書かない** — 前提を `TBD` として人間に確認する。

## 最初に確認する前提（人間へのヒアリング項目）

- 環境の種別と数（dev / staging / prod）
- 想定負荷: TPS・データ量・成長率
- 可用性目標（SLO 階層 — 冗長化コストに直結）
- 通貨と期間（月次/年次）

## コスト構造（3分類）

### 1. クラウドインフラ

- **Compute** — インスタンス/ノード数 × 単価。サイジングの根拠（TPS → 必要リソース）を明記
- **Storage** — データ量 × 成長率 × 保持期間（バックアップ含む）
- **Network** — エグレス・リージョン間転送（DR 構成は転送費が効く）

### 2. ライセンス/サブスクリプション

- 採用技術（technology fitness で Adopt にしたもの）の課金モデル: OSS / 従量 / 席数 / 契約
- 監視・ログ基盤の SaaS 費用（ログ量課金は見落としがち）

### 3. 運用

- サポート契約・オンコール体制の人的コスト
- 環境ごとの倍率（staging は prod の何割か、を明示的に決める）

## 規律

- 見積もりは**幅で出す**（楽観/悲観、または ±%）。単一の数字は前提が崩れた瞬間に嘘になる
- 各項目に前提（単価の出典・サイジング根拠）を併記する。前提が変わったら再計算できる形に
- 予算制約（constraints.md）と突き合わせ、超過するなら設計へフィードバックする（スコープ/冗長化の再検討）
- 確定した前提と採否判断は ADR ドラフトに要約する（見積もり詳細は短命のまま）
