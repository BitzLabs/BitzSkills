# 実装の規律

本ドキュメントは、要件仕様を実際のコードベースへ落とし込む際にエージェントと人間の双方が守るべき実装の規律と、それを担保するための自動検証プロセスを定義します。規律の維持はシステムの品質と保守性を確保するための生命線です。

## 1. 要件と実装の紐付け

すべての実装コードは、承認された要件仕様（FR/NFR等）に対する明確な追跡性（Traceability）を持たなければなりません。
コードベースに対するいかなる変更も、それが「どの要件を満たすために行われたか」という根拠を示すことが求められます。

### 1.1 implements 宣言による紐付けの確立

タスク定義ファイル（`.spec/tasks/` 配下）においては、必ず `implements:` メタデータを使用して対象となる要件 ID を指定します。
- **状態遷移のトリガ**: この宣言により、指定された要件のステータスは `approved` から実装中を示す `implementing` へと状態遷移します（遷移のトリガと不可逆性の詳細については `lifecycle.md` を参照してください）。
- **複数要件の実装**: 1つのタスクが複数の要件を包括的に実装する場合、YAML のリスト形式で複数の要件 ID を列挙することが許可されます。
- **マッピングの基盤**: これにより、後続の Verify フェーズにおいて「どのコードの集合が、どの仕様を満たそうとしているか」というマッピングが確立され、網羅性の検証が可能になります。

### 1.2 コード側からの逆引き参照

タスク定義による順方向の紐付けに加え、実装コード側でも対応するクラスやメソッドの Docstring（ドキュメント文字列）、または特定のコメントタグ等に要件 ID を記載し、双方向の追跡性を確保することが強く推奨されます。これにより、コードレビュー時に「なぜこの処理が存在するのか」の文脈が即座に共有されます。

## 2. 権限分離と契約（Contract）の保護

システムの安定性と一貫性を保つため、モジュール間の取り決めである「契約（Contract）」の変更には、厳格な権限分離（Segregation of Duties）を適用します。

### 2.1 契約として保護される要素の定義

以下の要素はシステム全体の「契約」として扱われ、一部の実行エージェントによる暗黙的・独断的な書き換えがシステムレベルで禁止されます。
- API エンドポイントのシグネチャ、リクエスト/レスポンスの Schema（スキーマ）
- データベースのスキーマ定義、およびデータベースマイグレーションスクリプト
- Domain Service（ドメインサービス）や Repository Interface（リポジトリインターフェース）の公開シグネチャ
- 外部システムとの連携に関するプロトコル定義

### 2.2 実装エージェントの権限制限

実装を担当する並列サブエージェントは、タスク定義の `boundary` で明示的に許可された範囲内のファイルに対してのみ変更を加えることができます。
- **契約の書き換え禁止**: 実装エージェントは、既存の「契約」定義ファイルを自己の判断で書き換える権限を持ちません。例えば、インターフェースの引数変更が必要と判断した場合でも、勝手にファイルを書き換えることは違反となります。
- **権限マトリクスの遵守**: 実装サブエージェントに許可される行為と禁止される行為の完全なリストは、`parallel-git.md` の「権限マトリクス」の表に準拠します。これに反する動作は即座にブロックされます。

### 2.3 契約変更のための正規プロセス

実装を進める中で、既存の契約に不備があることや、要件変更に伴う契約の変更が必要であることが発覚した場合、以下のエスカレーション手順を踏みます。
1. **実装の中断**: エージェントは即座に該当コードの修正作業を中断します。
2. **Issue 起票**: エージェントは `spec-issues/` ディレクトリに新しい Issue（仮番号、例えば `SI-<branch>-<n>`）を起票し、契約変更の必要性、具体的な影響範囲、および提案内容を記録します。
3. **人間による裁定**: 人間の開発者（アーキテクト）がその Issue をレビューし、契約の変更を裁定（Approve または Reject）します。
4. **仕様の更新と再開**: 承認された場合、要件と契約ドキュメントが正式に更新・採番され、新しい契約に基づく実装タスクが再発行されます。

## 3. spec_inspect.py による追跡性の自動検証

実装と要件の間に乖離が生じることを防ぐため、CI/CD パイプラインの Verify フェーズにおいて専用スクリプト `spec_inspect.py` を用いた自動突合検証を実施します。

### 3.1 implements マップの突合と網羅性検証

`spec_inspect.py` はリポジトリ全体をスキャンし、実装状況の詳細なレポート（`inspection-report.md`）を生成します。その際、以下の点に注目して検証を行います。

- **要件の充足性**: `approved` 状態以上のすべての要件（FR/NFR等）が、`implements` 宣言を持つ少なくとも1つの有効なタスクによってカバーされているかを確認します。漏れがある場合はエラーとなります。
- **孤児タスクと幽霊参照の検出**: 存在しない要件 ID や、既に `deprecated` 状態となった要件 ID を `implements` リストに指定している無効なタスクを検出します。
- **変更境界の厳守の検証**: タスクメタデータに定義された `boundary` のパスリストと、該当タスクで行われた実際の Git コミットの差分を突合し、宣言外のファイルが不正に変更されていないかを検証します。

### 3.2 判定結果によるプロセスの厳格な制御

この機械的な突合において不整合（カバー漏れ、幽霊参照、境界違反など）が1つでも検出された場合、システムは決して `verified` ステータスへ進むことはできません。
- 検証エラーを解消するためには、タスク定義の修正や新たな実装タスクの追加が必要です。
- 人間がこの機械判定の結果を手動で上書きし、強制的にパスさせることは禁止されています（プロセスの検証に対するこの不変条件の詳細は `verification.md` の L3 を参照）。

## 4. コミット粒度と変更履歴の規律

実装における変更履歴（Git History）は、コードレビューを円滑にし、将来の変更伝播分析（Impact Analysis）を支える重要な証跡となります。

### 4.1 タスク単位での独立コミットの原則

1つの実装タスクは、原則として1つの独立した Git コミット（またはプルリクエストのマージ時に Squash された単一の論理的コミット）として記録されなければなりません。
コミットメッセージには必ずタスク ID と関連する要件 ID を含め、変更の意図を機械的に検索可能にします。
- **推奨フォーマット例**: `feat(domain): [TASK-042] implement user validation (implements FR-012)`

### 4.2 文脈汚染（Context Pollution）の徹底防止

各コミットの内容は、タスク定義の `boundary` で宣言されたスコープに厳密に収まらなければなりません。
- 指定された要件とは無関係なリファクタリング、フォーマット修正、または他のタスクが管轄するファイルへの「ついで」の変更を含めることは「文脈汚染」とみなされ、固く禁じられます。
- この規律により、`lifecycle.md` に規定される「最小再実行」の原則が正確に機能し、仕様変更時の不要で過剰な変更伝播を確実に抑止できます。

## 5. 耐障害性と例外マッピングの実装規律

実装エージェントは、ハッピーパスだけでなく、予期せぬ状態やインフラストラクチャ層の障害に対する防御的なコードを書くことを義務付けられます。

### 5.1 規律ある例外処理の義務

- **例外の階層化**: 汎用的な例外のキャッチ（例: Javaの `catch (Exception e)` や Pythonの `except Exception:`）を避け、ドメイン層やインフラ層で定義された具体的な例外クラスを特定して処理します。
- **リトライロジックの実装**: データベースのトランザクション競合（OCC 競合など）のような一時的な障害（Transient Errors）に対しては、Exponential Backoff（指数的バックオフ）を伴うリトライロジックを実装に組み込まなければなりません。
- **追跡可能なログ記録**: 障害発生時には、トラブルシューティングに必要な識別子（トランザクション ID、ユーザー ID など）を必ずログに出力する規律を遵守します。
