# タスク分解とスケジューリング規約

本ドキュメントは、承認済みの要件仕様から実装作業へ移行する際のタスク分解の手法と、並列実行可能なタスクのスケジュール判断基準を定義します。本規約は BitzSkills における `.spec/` ワークフローの根幹をなすものであり、エージェントが効率的かつ衝突なく自律的に実装を進めるための基盤となります。

## 1. 1タスク1ファイル方式

BitzSkills の `.spec/` ワークフローにおいて、実装フェーズの作業は完全に細分化された「タスク」単位で管理されます。
すべてのタスクは、`.spec/tasks/` ディレクトリ配下に、1タスクにつき1つの専用 Markdown ファイルとして作成されなければなりません（例: `.spec/tasks/TASK-001.md`）。

### 1.1 この方式の目的と利点

タスクを個別のファイルとして管理する方式には、複数の重要な利点があります。

1. **物理的競合の排除**: タスクファイルが物理的に分離されているため、複数のエージェントが並列してタスクの着手や状態更新を行っても、バージョン管理システム上でのマージ競合が構造的に回避されます。
2. **状態の可視化**: ディレクトリ内のファイルリストとそのステータスを機械的に走査するだけで、プロジェクト全体の実装進捗やボトルネックを即座に把握できます。これは `spec_inspect.py` などの自動化スクリプトがシステムの状態を監視する基盤となります。
3. **追跡性の担保**: 後述するメタデータにより、どの作業がどの要件に結びついているかが明確になります。コードと仕様の間に完全な Traceability（追跡可能性）を確立することが可能です。
4. **文脈の局所化**: エージェントが1つのタスクに集中できるため、Context Pollution（文脈汚染）を防ぎ、高品質なコード生成が期待できます。

## 2. タスクの定義属性（メタデータ宣言）

各タスクファイルは、その実行とスケジューリングに必要な属性情報を Frontmatter または構造化データとして保持します。これらのメタデータはタスクのライフサイクル管理に不可欠です。

### 2.1 必須の宣言項目

タスク定義ファイルには、最低限以下の情報が含まれている必要があります。

- **ID**: タスクの一意な識別子。ファイル名のベースと完全に一致させます（例: `TASK-042`）。この ID を使用して、コミットメッセージやテストコードから逆引きを行います。
- **implements**: このタスクが実装を担当する要件 ID（例: `FR-012`, `NFR-005`）のリストです。
  - ここに要件 ID が列挙されることで、該当する要件のライフサイクルは `approved` から実装中であることを示す `implementing` へと自動的に状態遷移します（遷移トリガと不変条件の詳細は `lifecycle.md` を参照）。
- **depends_on**: このタスクの実行を開始する前に、完全に終了していなければならない（Done 状態であるべき）先行タスク ID のリストです。依存がない独立したタスクの場合は、空のリストを明記します。
- **boundary**: このタスクを実行するエージェントが、ファイルの書き込みや変更を許可される対象パス（ファイルまたはディレクトリ）の厳密なリストです。絶対パスまたはプロジェクトルートからの相対パスで記述されます。

### 2.2 オプションの宣言項目

必要に応じて、以下の情報を含めることができます。

- **cost_budget**: タスク実行に割り当てられたトークン予算や想定時間。
- **status**: タスクの現在の状態（`todo`, `in-progress`, `blocked`, `done` など）。
- **description**: タスクの具体的な実装指示や留意事項。

## 3. タスク粒度の基準

タスクの粒度は、実装エージェントが文脈を見失わず、効率的かつ正確に処理できるサイズに最適化されなければなりません。以下の基準に従ってタスクを分割します。

### 3.1 単一責任とドメインに基づく分割

1つのタスクは、単一の論理的なコンポーネントや変更単位に限定します。
- **例1**: 1つの Domain Service（ドメインサービス）クラスの新規作成および対応する単体テストの実装。
- **例2**: 1つの Value Object（値オブジェクト）とその不変条件（Invariant）ルールの実装。
- 広範なリファクタリングや、複数のモジュールを同時に触る必要がある作業は、影響範囲ごとにタスクを細かく分割し、段階的に適用するように設計します。

### 3.2 境界（Boundary）の明瞭さとサイズ

`boundary` として宣言されるファイル数は、可能な限り最小限に抑える必要があります。
- 原則として、プロダクトコードの1〜3ファイル程度と、それに対応するテストコードの変更を1タスクの目安とします。
- 境界が広範になるタスクは、エージェントの処理能力を超えたり、他タスクとの競合リスクを高めたりするため、インターフェースの修正とロジックの実装といった形で分割してください。

### 3.3 コスト予算の制約

既存の `parallel-git.md` に規定されている通り、feature 単位のコスト予算（トークン数や実行時間）は事前にタスクメタデータへ宣言されます。
タスクが予算を超過して暴走することを防ぐため、複雑度の高い機能実装は単一のタスクに詰め込まず、設計タスクと実装タスクのようにフェーズを分けて分割します。

## 4. 並列投入可能なタスク群の判定方法

タスクの並列実行はリードタイムの短縮に不可欠ですが、リポジトリの不整合を防ぐための厳格な制御メカニズムが必要です。

### 4.1 依存関係に基づく順序性の判定

並列に実行可能なタスクの第一条件は、先行作業への依存が完全に解決されていることです。
実行キューに並列投入できるタスクは以下のいずれかを満たすものに限られます。
- `depends_on` 宣言が空（初期着手可能な独立タスク）。
- `depends_on` で指定されたすべての依存先タスクが、既に完了状態（Done）に達している。
このルールにより、DAG（有向非巡回グラフ）に基づく順序正しい実行が保証されます。

### 4.2 書き込み境界（Boundary）の空間的独立性

順序性の条件を満たしたタスク群であっても、同一のファイルへ同時に書き込みを行うことはGitの競合を招きます。
- 同時に実行エージェントへ割り当てるタスク群は、その `boundary` 宣言が互いに一切重複してはなりません。
- もし `boundary` が重複するタスクが実行可能キューに複数存在する場合、優先度に応じて一方が完全に完了するまで他方の実行を保留し、直列化します。
- 並列サブエージェントの基本スコープ定義や、人間とエージェント間の詳細な権限マトリクスについては、既存の `parallel-git.md` の規定に完全に従い、本基準と矛盾させずに運用してください。

## 5. スキャフォールディングと実装のタスク導出パターン

アプリケーションの初期構築時（Scaffolding）や大規模な新機能の実装時には、以下のようなパターンでタスクを抽出し、依存関係を構築します。このパターン化により、漏れのない計画が立案可能です。

### 5.1 フェーズ1: 契約・設計の固定タスク

- **内容**: API のインターフェース定義やデータベースの Schema（スキーマ）定義を作成します。
- **特性**: すべての実装タスクの基礎（depends_on の対象）となります。このフェーズが完了しない限り、後続の並列実装は開始できません。

### 5.2 フェーズ2: ドメインの独立実装タスク

- **内容**: 外部依存を持たない Value Object（値オブジェクト）や中核となる Entity（エンティティ）の実装。
- **特性**: フェーズ1に依存しますが、エンティティ間での依存関係が薄いため、複数エンティティ間で大規模な並列実行が可能です。

### 5.3 フェーズ3: インフラストラクチャ連携タスク

- **内容**: データベースの Repository（リポジトリ）実装や、外部 API クライアント、構成ファイルの生成タスク。
- **特性**: フェーズ2で作成されたドメインモデルに依存します。データベースの種類ごとにタスクが分割されることもあります。

### 5.4 フェーズ4: 統合と結線タスク

- **内容**: DI コンテナの設定、コントローラーやエンドポイントへのドメインサービスの結線作業。
- **特性**: 先行するすべてのフェーズの完了に依存します。システム全体を通した動作確認が可能になる最終段階です。

このようにタスクをフェーズごとに構造化することで、安全かつ最大効率の並列スケジュールが機械的に導出されます。タスクプランナーはこのパターンに沿って `depends_on` グラフを構築する義務があります。
