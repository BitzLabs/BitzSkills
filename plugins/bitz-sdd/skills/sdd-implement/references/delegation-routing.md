# 委譲ルーティング — Execute フェーズの委譲ゲート

タスク着手前に、各タスクを**役割**で分類し、司令塔（＝いま起動しているモデル）の
ティア上の**相対位置**に基づいて「委譲 / 相談・上申 / 自己実行」を判定する。

具体モデルの束縛はプロジェクトの**委譲レジストリ**（環境ごとに持つ。例: 消費側プロジェクトの
オーケストレーター設定＝CLAUDE.md の委譲レジストリ表）が正。本ファイルは**モデル名を持たず**、
役割と相対ティアだけで手順を定義する（モデル世代交代に本ファイルは非依存）。

## 前提: 委譲レジストリ

プロジェクトは次を宣言したレジストリを持つ:

- **ティアはしご**: 上位→下位のモデルティア順序。
- **役割→委譲先→ティア** の束縛表。委譲先は下位モデルのサブエージェント（機械的作業担当・
  深い推論担当）や別プラットフォーム委譲（量産・クロス検証）。
- **司令塔ティア** = ユーザが起動したモデルのティア（はしご上の自分の位置）。

## 判定手順

### 1. 役割分類

各タスクを次のいずれかに分類する:

- **機械的**: 手順が明確な定型修正・テスト実行・ファイル整理・frontmatter 一括更新。
- **難調査・設計**: 設計案作成・原因不明の不具合調査・トレードオフ分析。
- **量産**: スキャフォールド・網羅テスト生成・一括移行・長文要約・Web検索。
- **検証**: セカンドオピニオン・クロスモデル検証。

### 2. 損益分岐（過剰委譲の抑止）

1ファイルの軽微な編集など、委譲の往復コストが節約を上回る**小さな単発タスク**は
委譲せず司令塔が直接実行する。委譲は無条件に善ではない。

### 3. 下位への委託（機械的・量産）

役割の委譲先ティアが司令塔より**下位**なら、その委譲先へ委譲する。

- 下位の委譲先が存在しない（司令塔が最下位）なら司令塔が自己実行する。
- 役割の委譲先ティア == 司令塔ティア（**同一ティア**）なら省トークンにならないため委譲しない（無効化）。
- 量産・検証の**別プラットフォーム委譲**は相対ティアと直交し、常に選択可能。

### 4. 上位への相談・上申（難調査・設計）

司令塔より**上位**のティアが存在するなら、そこへ相談・上申する。

- 上位が存在しない（**司令塔が最上位**）なら相談相手はいない＝司令塔が自己判断する。

### 5. 検収は司令塔

委譲先の「成功しました」を鵜呑みにせず、機械チェック（release_check / spec_inspect / テスト）で
司令塔が自分で検証する。

## 相対選択の要点

- 委譲は絶対的な役割固定ではなく、司令塔ティアからの**相対**で決まる。
- **下位が無ければ委譲しない・上位が無ければ相談しない・同一ティアは無効**。
- どのモデルがどのティアかは本ファイルではなく**レジストリ**を見る。
