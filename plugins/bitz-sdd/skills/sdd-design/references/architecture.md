# アーキテクチャ統合（3ビュー + 技術適合性評価）

docs/03_設計仕様/アーキテクチャ.md の proposed ドラフトを作るための手法。**統合であって発明ではない** — 図のすべてのノードは既存の設計要素（コンテキスト・API・エンティティ・要件）に対応させる。どの成果物にも根拠のない構成要素が必要になったら、それはギャップとして明示する（能力を発明しない）。

## Part 1 — 3つの相補的ビュー（mermaid）

プロジェクトの成果物が支える分だけ描く（省略したビューはその旨を記す）:

1. **ランタイム/コンテナビュー**（`flowchart`）— 主成果物。外部アクター/システム → エッジ → Experience → Process → System → データストア。コンテキストでグルーピングし、各ノードに Core / Supporting / Generic のタグを付ける。主要データフローとクリティカルパスを示す
2. **クリティカルパスビュー**（`sequenceDiagram`）— 最も差別化的または最もリスクの高い1リクエストフロー（決済 Saga 等）。レイテンシと障害モードを明示する
3. **デプロイ/スケールビュー**（表または `flowchart`）— コンテキストからデプロイ可能なサービスへの対応（モノリス/マイクロサービス制約を尊重）と、サービスごとに異なるスケール特性（スループット型 / 低レイテンシ型 / 耐久型）。各サービスを根拠の NFR/SLO に紐づける

図の規律:

- ラベルは簡潔に。改行は `<br/>`。差別化要素（★）と障害時デフォルト（fail-open 等）はインラインで記す
- すべての構成要素の対応表（ノード → 設計要素/要件）を図の下に置く
- 上流が未決の境界・数値は `TBD` と書く（塗り隠さない）
- mermaid: ノード ID は ASCII、日本語ラベルは引用符で囲む。矢印は `-->`（`->` は構文エラー）。1図 20 ノード以内、超えたら subgraph か分割

## Part 2 — 技術適合性評価（証拠駆動）

候補技術（API ゲートウェイ・メッセージブローカー・分散トランザクション基盤・台帳/監査ストア・キャッシュ・ベクトルストア等、**プロジェクトの成果物が示唆するカテゴリ**）を、プロジェクト自身の成果物に照らして評価する。適合性は設計に既に現れているシグナルから読み取る — 雰囲気で選ばない。各候補について:

- **適合性**: High / Medium / Low / None
- **根拠シグナル**: 引用元の設計要素・要件を具体的に挙げる（**High は具体的な引用が1つ以上必須**。引用できなければ Low/None）
- **判断**: Adopt / Conditional / Reject
- **理由**: なぜか、採用ならアーキテクチャのどの層/コンテキストに入るか。Reject なら既存のどの成果物がそれを不要にするか
- **条件/リスク**: Conditional が Adopt になるために真であるべきこと（Open Question・TBD）

規律:

- **熱意より証拠**。「Reject + 理由」は立派な成果
- 制約（CON 要件、例: マネージドサービス限定）に反する技術は Conditional/Reject とし、衝突を名指しする
- 判断は可逆な入力として理由ごと記録する（変更再伝播で再検討できるように）
- どのシグナルも支持しない技術でリストを水増ししない

## Part 3 — レイヤリングとデータ層の対応

MVC / レイヤードアーキテクチャを採るシステムでは、レイヤ構造とデータ層の対応を明示する。

| レイヤ | 責務 | データ層との関係 |
|---|---|---|
| Presentation（View / Controller） | 入出力・画面遷移・リクエスト整形 | 永続化の詳細を知らない。DTO / ViewModel 経由 |
| Application（ユースケース） | ユースケースの調整・トランザクション開始 | **トランザクション境界をここで宣言**（境界の実体は集約/エンティティ整合性に従う） |
| Domain（Model） | ビジネスルール・不変条件 | ストレージ非依存。domain-model.md が正 |
| Infrastructure（データ層） | 永続化・外部 I/O の実装 | Repository 等で抽象化し、依存は常に内向き（Domain がデータ層に依存しない） |

規律:

- **ドメインモデルが先、スキーマは後**。データ層の設計は `sdd-data` スキルが担当し、成果物は `.spec/design/data-model.md` / `data-storage.md`
- ランタイムビューのデータストアノードは、`sdd-data` の格納方式選定（証拠駆動）と一致させる
- レイヤ間の依存方向違反（例: Domain から SQL 直叩き）はギャップとして明示する
