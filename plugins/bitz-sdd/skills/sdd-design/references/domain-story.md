# ドメインストーリー（Domain Storytelling）

特定の**ペルソナ**が**重要ジョブ**を達成する具体的な一本道（ハッピーパス）を物語として記述する手法。ドメインモデルとその後の要件派生（1 Activity ≒ 1 EARS 節の粒度）の根拠になる。

## 3要素

- **Actors** — 主役のペルソナ（docs/01-context/personas-journeys.md 由来）+ 途中で相互作用する人/システム
- **Work Items** — 道中で扱うモノ/情報。ジャーニーとジョブの語彙を**そのまま**使う（同義語を発明しない）。ドメインモデルがあればエンティティに対応づける
- **Activities** — 番号付きの行動列。ジョブストーリー（「〜のとき、〜したい、〜できるように」）とジャーニーの段階から導出し、一貫したハッピーパスに並べる

## 規律

- **1ストーリー = 1ペルソナ × 1重要ジョブ**。一本道に保ち、分岐は本流に入れず「例外シナリオ」（最大3件、ジャーニーのペインから選ぶ）として別記する
- 粒度は**ユーザータスク単位**（1 Activity がひとつの画面操作/対話に対応する程度）。UI を持たないシステム内部ステップは書かない
- ジャーニー/ジョブが根拠にならないステップを**でっち上げない**。`TBD` + Open Question として残す
- ペルソナが未確立なら黙って一般フローに劣化させず、先に `sdd-discovery` を提案する

## 書式

assets/domain-story.md をコピーして `.spec/design/stories/story-<ペルソナ>-<ジョブ>.md` として書く。構成: 概要（2〜3文）→ Actors 表 → Work Items 表 → Main Flow（番号付き、各行に根拠のジョブ/ジャーニー段階を付記）→ mermaid `sequenceDiagram`（番号を Main Flow と一致させる。ハッピーパスのみ）→ 例外シナリオ → Open Questions。

## 停止条件

対象の各ペルソナ×ジョブについて: Actors 表（ペルソナ≥1）・Work Items 表・根拠つき番号フロー・sequence 図が揃ったら完了。ストーリーの結論（中核概念・操作の流れ）はドメインモデルのドラフトに反映する。
