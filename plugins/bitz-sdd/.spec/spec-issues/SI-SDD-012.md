- **目的**: BasicPlan の `000.governance/05.ドキュメント標準.md` では、人間向けナラティブを
  日本語の6章構成（`00_はじめに`〜`05_リリース・運用`）で管理し、コードリポジトリ作成時に
  `.spec/` と `docs/` の二層へ移管する契約を定めている。一方、現行 `sdd-docs` は英語の
  8章構成（`01-context`〜`08-knowledge`）と英語ファイル名を公開テンプレート・同期先・検査契約に
  採用している。BitzLabs の標準環境として両者を一致させ、日本語を基本言語とする文書構成を
  `sdd-docs` の正規契約として確立する。
- **提案する修正**:
  1. `docs/` の正規構成を BasicPlan と同じ
     `00_はじめに/`、`01_システム仕様/`、`02_ユースケース/`、`03_設計仕様/`、
     `04_テスト仕様/`、`05_リリース・運用/` の6章へ改訂し、テンプレートの表示名・本文・
     ファイル名を日本語化する。外部API・CLI・SDK・移行ガイドなど参照資料が多いプロジェクトだけ、
     `MASTER.md` で宣言した任意章 `06_リファレンス/` を追加できるようにする。型名・API名・
     機械ID・frontmatterのキーと機械値は英語のまま維持する。
  2. 現行8章の各成果物を6章へ写像する移行表を設計する。少なくとも discovery は
     `00_はじめに`、要件は `01_システム仕様`、ユースケースは `02_ユースケース`、設計・ADRは
     `03_設計仕様`、検証は `04_テスト仕様`、運用・リリースは `05_リリース・運用` に収容する。
     glossary と governance は `00_はじめに`、実装パターンと ADR は `03_設計仕様`、恒久的な
     教訓は ADR または `05_リリース・運用` の振り返り、postmortem は
     `05_リリース・運用` に配置する。参照資料は内容に応じて `03_設計仕様` / `05_リリース・運用`
     または宣言済みの任意 `06_リファレンス` に配置する。
  3. `MASTER.md`、文書ID、frontmatter、`derived_from` の機械契約は失わず、日本語パスを
     `docs_inspect.py` と `sdd_sync.py` が決定的に扱えるよう改訂する。台帳名を日本語化する場合は
     旧 `MASTER.md` の互換読取期間と移行診断を設ける。
  4. 既存利用者向けに dry-run 既定の移行スクリプトまたは明示的な移行手順を提供し、英語8章と
     日本語6章の混在、既存文書の衝突、手動本文の上書きを検出して停止する。
  5. テンプレート、同期マッピング、構造検査、スキル本文、関連要件を同一PRで更新し、
     日本語6章を初期化した直後と `.spec` から pull した直後の双方で strict 検査が通る
     回帰テストを先行追加する。
  6. `docs/調査報告/` のような調査メモ・アーカイブは6章へ強制移動せず、`MASTER.md` で
     sdd-docs の管理対象外パスとして宣言可能にする。管理対象外の境界は SI-CORE-030 の
     リポジトリ文書分類と整合させる。
- **対象ファイル**: `skills/sdd-docs/assets/docs-templates/`、
  `skills/sdd-docs/scripts/docs_inspect.py`、`sdd_sync.py`、移行用スクリプト（新規候補）、
  `skills/sdd-docs/SKILL.md`、各上流・設計・検証・運用スキルの同期先表、関連テスト、
  SDD-FR-020 / 030 / 040 / 100 / 101 と後継または改訂要件、bitz-sdd の3マニフェスト。
- **確認観点**:
  - 既存要件との関係: 公開パスを規定する SDD-FR-020 / 030 / 040 / 100 / 101 の改訂または
    supersedes が必要。BasicPlan の日本語6章構成をSSOTとして参照しつつ、BitzSDD固有の
    frontmatter・台帳・同期契約は維持する。
  - ガードレール: 既存 `docs/` を一括上書き・削除しない。移行は dry-run と衝突検出を備え、
    大量変更は人間がdiffを確認できる粒度に分割する。
  - 検証: 日本語6章の初期化、旧8章からの移行、再実行の冪等性、衝突時停止、pull/push/diff、
    `docs_inspect.py --strict` をテストする。`release_check.py` とモノリポ一括 spec inspect も維持する。
  - 軽量レーン適否: **不適**。公開される文書パス、テンプレート、同期API、検査契約を変更するため、
    通常フロー + Design Gate とする。
- **影響推定・ロールバック**: `--impact SDD-FR-020`、`SDD-FR-030`、`SDD-FR-101` はいずれも
  bitz-sdd の既存タスク `SDD-TSK-002` 1件を依存成果物として示した。移行を独立段階に分け、
  問題時は日本語6章の既定化・同期マッピング・検査契約を同一PRでrevertし、旧8章の読取互換を残す。
- **依存**: なし。本issueで同期先構成を先に裁定し、その後 SI-SDD-010（frontmatterと本文同期）、
  SI-SDD-011（Discovery同期マッピング）を進める。BasicPlan 側の文書移動は本issueの対象外。
- **合意済み設計方針**: 2026-07-18 の人間レビューにより、必須6章で不足はなく、拡張性は
  宣言式の任意 `06_リファレンス` で確保する方針とした。`MASTER.md`、frontmatterキー、文書ID、
  statusなどの機械契約は英語のまま維持し、調査・アーカイブは明示的な管理対象外として扱う。
- **予備判定（推薦）**: **accept 推奨**。BasicPlan がBitzLabs全体の文書標準として日本語6章を
  定義しているのに、配布ツールが別の英語8章を生成すると、コードリポジトリ移管時に二重標準と
  パス変換が残る。既存文書保護を移行契約として先に設計する条件で採用する。
- **実施**: 2026-07-18 に SDD-FR-125〜SDD-FR-129 として要件化し、SDD-DSN-002 の
  Design Gate（SDD-REV-002: PASS）を経て実装・検証を完了した。日本語必須6章、宣言式の
  任意 `06_リファレンス` と管理対象外パス、日本語同期先、dry-run既定かつrollback可能な
  移行CLIを bitz-sdd 2.0.0 に反映した。
