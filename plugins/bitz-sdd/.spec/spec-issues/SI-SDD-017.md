- **目的**: 「要件化 PR を先に land → 実装 PR を並列で後続」という推奨フロー
  （sdd-implement の並列投入・flow-pr の未マージ依存の原則に沿った運用）を取ると、
  要件化 PR のマージから実装 PR のマージまでの間、デフォルトブランチが
  spec_inspect の孤児要件 FAIL（approved だが参照タスクなし）になる。
  SI-CORE-031/032 では PR #70〜#74 の間この状態が発生した。CI に spec_inspect を
  組み込むと要件化先行フローが構造的に落ちるため、過渡状態の扱いを規定する。
- **提案する修正**（いずれかを要件化時に選択）:
  1. **猶予案**: `approved` ステータスの要件はタスク未紐付けでも WARN 扱いにする
     （`implementing` 以降でタスクなしなら従来どおり FAIL。「approved = 実装待ち
     キュー」という意味論に合わせる）
  2. **閾値案**: `--strict` フラグで現行挙動、デフォルトは approved のみ WARN
  3. **現状維持 + 運用規定案**: ツールは変えず、「要件化と最初のタスク起票は
     同一 PR で行う」を lifecycle.md に規定する（要件化先行フローを禁止する側に倒す）
- **対象ファイル**: `plugins/bitz-sdd/skills/sdd-core/scripts/spec_inspect.py`、
  `references/lifecycle.md`、対応するテスト（テスト先行）。
- **確認観点**:
  - approved + タスクなしの合成フィクスチャで期待どおり WARN / FAIL になること
  - 既存ワークスペース（root / 全プラグイン）の判定が意図せず変わらないこと
  - 案3の場合はツール変更ゼロで lifecycle.md の規定のみ
- **影響推定・ロールバック**: 検証ツールの判定変更は全利用プロジェクトに波及するため
  テスト先行必須。単独 revert 可能。
- **依存**: なし。関連: SI-SDD-018（同キャンペーン発の inspect 改善。独立に実装可能）。
- **予備判定（推薦・裁定は人間）**: **accept を推薦（案1を推奨）**。approved を
  「人間が承認した実装待ち」と定義する現行ライフサイクルと、タスク紐付けを要求する
  検査の間に意味論の不整合がある。案1はライフサイクル定義に検査を合わせる最小修正。
  検査ツールの判定変更＝契約変更のため軽量レーン不可、通常フロー + テスト先行を推奨。
