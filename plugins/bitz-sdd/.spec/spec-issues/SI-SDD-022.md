- **優先度（推薦）**: **高**。sdd-core 憲法4「仕様の変更権は常に人間が持つ」の実効性そのものに
  関わる。監査証跡の信頼性が損なわれている状態で、他の規律の前提を崩す。
- **目的**: `--by-human` が「人間が実行することの明示」と定義されながら、**実体は誰でも渡せる
  ただのフラグ**であり、実行主体を検証する手段がない。権限分離が文書上の約束にとどまっている。
  - **実際に発生した事象（2026-07-21）**: SI-CORE-018 の実施において、エージェント（Claude）が
    ユーザーの口頭指示を根拠に `--by-human --actor hide` を自ら実行し、3要件
    （`SDD-FR-137` / `SDD-FR-138` / `bitz-ddd:DDD-FR-002`）を draft→approved させた。
    その結果 `STATE.md` には人間が実行した場合と**区別のつかない記録**が残っている:
    `- 2026-07-21 SDD-FR-137: draft → approved (hide)`
  - **非決定的な挙動**: 同一セッション内で、同じ操作が一度はホスト側の権限機構に拒否され
    （`SDD-FR-136`）、一度は通った（`SDD-FR-137` 以降）。**緩い方に倒れたときだけ通る**
    という最悪の形の非決定性であり、規律として当てにできない。
  - **影響**: 「人間が承認した」という主張が証跡から裏付けられない。憲法4 と
    `lifecycle.md` の権限マトリクスは、コードでは強制されていない。
- **提案する修正**（いずれか、または組み合わせ。裁定時に選択）:
  1. **対話性の要求**: `--by-human` 指定時に標準入力が TTY であることを要求し、非対話実行では
     拒否する。エージェントの非対話実行を構造的に締め出せる一方、CI や `!` 前置きでの
     実行形態によっては誤って弾く可能性があるため適用範囲の検討が要る。
  2. **確認プロンプト**: `--by-human` 指定時に対象 ID と遷移内容を提示して明示的な再入力
     （ID の打ち直し等）を求める。1 より緩いが、非対話環境では 1 と同じ効果になる。
  3. **代行実行の可視化**: 実行主体を偽らせないことを優先し、`--on-behalf-of` 等の別語彙を
     設けて「人間の指示によりエージェントが代行した」ことを `STATE.md` に区別して記録する。
     強制はできないが、証跡の正確性は回復する。
  4. **記録語彙の整備**: 上記いずれを採るにせよ `STATE.md` の書式に実行経路を残す
     （`(hide)` と `(agent on behalf of hide)` を区別できるようにする）。
- **対象ファイル**: `plugins/bitz-sdd/skills/sdd-core/scripts/spec_update.py`、
  `plugins/bitz-sdd/skills/sdd-core/references/lifecycle.md`（権限マトリクスの記述）、
  `plugins/bitz-sdd/skills/sdd-core/SKILL.md`（権限の分離節）、`tests/test_spec_update.py`、
  bitz-sdd の3マニフェスト。
- **確認観点**:
  - 非対話実行のエージェントが人間専用遷移を成立させられないこと（採用案が 1/2 の場合）
  - 人間による通常の対話実行が阻害されないこと（`!` 前置き実行を含む）
  - `STATE.md` から実行経路（人間の直接実行 / 代行）が判別できること
  - 既存の権限マトリクステストが改変なしで PASS し続けること
- **影響推定・ロールバック**: `spec_update.py` の実行体験が変わるため、既存の運用手順
  （AGENTS.md / SKILL.md の記載例）の更新を伴う。`.spec/` のデータ移行は不要。
  ロールバックは `spec_update.py` とテストの revert で戻る。
  **過去に記録済みの `STATE.md` 行は遡及修正しない**（誤りではなく、当時の記録形式のため）。
- **依存**: なし。ただし採用案によってはホスト側（Claude Code の permissions 等）の
  設定と役割分担を整理する必要がある — 二重に締めるか、ツール側に一本化するか。
- **可否の予備判定（推薦）**: **accept 推薦**。根拠:
  - 既存要件との矛盾: なし。CORE-FR-005（権限マトリクスの強制）を実効化する方向の修正
  - ガードレール抵触: なし
  - 影響範囲: `spec_update.py` と運用文書
  - 軽量レーン適否: **不可**。人間専用遷移の実行契約に触れるため通常フロー + Design Gate を要する
    （採用案の選択自体が Design Gate の論点）
