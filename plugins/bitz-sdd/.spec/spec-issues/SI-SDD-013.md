- **目的**: BasicPlan は各パッケージに `02.ユースケース/UCNNN_名称.md` を置き、アクター、
  事前・事後条件、基本フロー、代替フロー、関連機能要件を記述して、要件妥当性とテスト設計の
  入力にしている。BitzSDD にはペルソナ／ジャーニー、任意のDDDドメインストーリー、
  「主要ユースケースからFRを派生する」という断片的規定はあるが、ユースケースを作成・採番・
  検証・要件／テストへトレースし、`docs/02_ユースケース/` へ同期する独立フローがない。
  DDDプラグインの有無にかかわらず利用できる標準ユースケース工程を追加する。
- **提案する修正**:
  1. Discovery Gate 後、要件承認および詳細設計の前にユースケースを定義する工程を設ける。
     `.spec` 側のマスター配置、`UCNNN` 採番、frontmatter、状態遷移をsdd-core契約として定め、
     手書き採番を避けるスキャフォールドを提供する。
  2. 日本語テンプレートにユースケースID・名称・目的／概要・アクター・事前条件・成功時事後条件・
     基本フロー・代替／例外フロー・関連要件ID・受入条件／テスト入力・備考を持たせる。
  3. `bitz-ddd` 導入時は `ddd-story` のハッピーパスを入力として再利用し、ユースケース側で
     観測可能な成功条件・代替フロー・要件トレースを補う。未導入時は discovery のペルソナ／JTBDと
     承認済みスコープから直接作成し、DDDの有無で成果物契約が変わらないようにする。
  4. 1ユースケースから複数のEARS要件を導出できる対応表を定義し、要件側から元UCへ、
     テスト仕様側からUCと要件へ追跡できるよう `spec_inspect.py` で参照切れを検出する。
  5. `sdd-docs` の pull で個別UCを `docs/02_ユースケース/UCNNN_名称.md` とREADME／索引へ展開し、
     push/diffの対応範囲、frontmatter保持、手動本文保護を定める。
  6. 専用 `sdd-usecase` スキルを追加するか `sdd-design` の独立工程とするかをDesign Gateで比較し、
     採番・雛形生成・トレース検証・同期の回帰テストを先行して実装する。
- **対象ファイル**: `skills/sdd-usecase/`（新規候補）または `skills/sdd-design/`、
  `skills/sdd-core/scripts/spec_scaffold.py` と `spec_inspect.py`、artifact/lifecycle/gates契約、
  `skills/sdd-docs/scripts/sdd_sync.py` と日本語テンプレート、`skills/sdd-test/` の導出規定、
  関連テスト、SDD-FR-030 / 033 / 101 と後継または改訂要件、bitz-sdd の3マニフェスト。
- **確認観点**:
  - 重複: `ddd-story` は1ペルソナ×1重要ジョブのハッピーパスとドメインモデル入力を担当する。
    本issueは代替フロー、要件・テストのトレース、標準docs配置を含むユースケース契約であり、
    両者の責務と変換規則を明示して二重管理を避ける。
  - 既存要件との関係: SDD-FR-030 / 101 のドメインストーリー集約を壊さず、SDD-FR-033 の
    graceful degradationを維持する。既存ストーリーをUCへ強制移行しない。
  - ガードレール: 根拠のないアクター・代替フローを生成しない。UCから生成したEARS要件はdraftとし、
    人間のapproveを迂回しない。
  - 検証: DDDあり／なし、複数要件への導出、参照切れ、採番衝突、同期の冪等性、索引更新、
    strict docs検査、EARSからのテスト導出を回帰テストする。
  - 軽量レーン適否: **不適**。新しい成果物スキーマ、工程、同期先、トレース契約を追加するため、
    通常フロー + Design Gate とする。
- **影響推定・ロールバック**: `--impact SDD-FR-030` と `SDD-FR-101` は bitz-sdd の既存タスク
  `SDD-TSK-002` 1件を依存成果物として示した。新規UC契約は既存ドメインストーリーを変更せず
  additiveに導入し、問題時は専用工程・同期マッピング・検査規則を一括revertできるようにする。
- **依存**: SI-SDD-012（日本語6章と `02_ユースケース` の正規パス確定）、SI-SDD-010
  （docs frontmatterと本文同期の契約統一）。推奨順序は SI-SDD-012 → SI-SDD-010 → 本issue。
- **予備判定（推薦）**: **accept 推奨**。現状のハッピーパス中心のドメインストーリーだけでは、
  BasicPlanが実運用している代替フロー、関連要件、テスト入力を一貫して保持できない。DDDと競合しない
  変換境界を設ける条件で、BitzSDDの標準工程として追加する。
