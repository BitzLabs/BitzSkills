# ペルソナとジョブ — bitz-sdd

> 遡及的 discovery。生の声・感情・台詞はユーザーリサーチ未実施のため、開発者本人
> （hide）の実体験に基づくもの以外はすべて `[proto / 未検証]` と明示する。捏造しない。

## Jobs-to-be-Done（ジョブストーリー形式）

- **J1（機能的）**: *個人で AI エージェントに機能を作らせるとき、要件・設計・実装・検証が
  トレースされた状態で進めたい。あとで仕様と実装が乖離しないように。*
- **J2（機能的）**: *エージェントが「テストして動きました」と報告してきたとき、その主張を
  機械的に検収したい。自己申告を鵜呑みにせず品質を担保できるように。*
- **J3（機能的）**: *仕様を書くとき、機械が検証する正と人間が読む文書を二重管理したくない。
  一方を直せば他方が追従するように。*
- **J4（感情的）**: *規律を課すことで「勝手に暴走される不安」から解放されたい。エージェントを
  安心して走らせられるように。*
- **J5（社会的）**: *OSS として、自分の開発規律を再現可能な形で他者に示したい。「ちゃんとした
  やり方でやっている」と見られるように。*
- **J6（機能的・OSS 利用者）**: *SDD/EARS を試したいとき、重厚な手法を学ばずに導入して
  最初の要件を起票したい。学習コストで挫折しないように。* `[proto / 未検証]`

## ペルソナカード

### P1: 個人開発者「hide」（主要ペルソナ・アーキタイプ = ドッグフーダー）
- **1行**: Claude Code / Antigravity を日常使いする個人開発者。BitzSkills のメンテナ本人。
- **文脈と行動**: モノレポで複数プラグインを開発。uv で Python ツールを回す。司令塔
  （オーケストレーター）＋サブエージェント運用でトークンを節約しながら開発する。
- **Jobs**: J1・J2・J3・J4・J5。
- **Pains**: エージェントの暴走・自己申告の非信頼性・仕様と実装の乖離・正の二重管理・
  重い手法を入れると自分の維持コストが跳ねる。
- **Gains**: 検証で裏づいた完了・単一の正・軽量な規律・再利用可能なスキル資産。
- **生の声**: 「自己申告ではなくスクリプトが検収する」（Values に反映済み。本人の運用方針）。
  それ以外の感情表現は `[proto]`。

### P2: OSS 利用者（副次ペルソナ・アーキタイプ = 導入検討者）

- **文脈と行動**: 素の CLAUDE.md 運用や spec-kit 系を試したが定着せず、軽量な代替を探す。
- **Jobs**: J6・J1・J2。
- **Pains**: 学習コスト・導入手順の不明瞭さ・「本当に効くのか」の不確実性。`[proto / 未検証]`
- **Gains**: 低摩擦な導入・最初の `.spec/` 到達・検証の自動化。`[proto / 未検証]`
- **生の声**: `[proto / 未検証]`（リサーチ未実施のため台詞は捏造しない）。

### P3: served — AI コーディングエージェント（アーキタイプ = 規律の受け手）
- **1行**: Claude Code / Antigravity 2.0 のエージェント。SKILL.md の description/本文で行動が規定される。
- **Jobs**: 「曖昧な指示でも `.spec/` を正として正しい工程・権限・検証に従いたい」。
- **Pains**: 発動条件が曖昧だとスキルが起動しない・本文が長いとトークンを浪費する。
- **Gains**: 明確な description による確実な発動・薄い本文・スクリプトへの委譲。

## 主要ペルソナ

**P1（個人開発者 hide）を主要ペルソナとする。** 理由: 現時点で唯一の検証済み実利用者
（ドッグフーディング）であり、下流の優先順位は P1 の J1〜J4 に集中する。P2（OSS 利用者）は
需要が `[proto / 未検証]` のため secondary とし、Desirability 検証の主対象（assumptions.md）に置く。
