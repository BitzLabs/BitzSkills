# 仮説検証ゲート — bitz-sdd

> 遡及的 discovery。bitz-sdd は v1.4.6 で**実装・リリース済みかつ本人がドッグフーディング中**
> という特殊事情がある。したがって Feasibility / 一部 Desirability は「実装が動いている」
> という**既存エビデンス**で裏づけられる一方、OSS 外部利用者の Desirability / Viability は
> 未検証（`[proto]`）である。この非対称性を明示して裁定材料を揃える。

## 仮説一覧（3分類 + 崩壊影響 + テスト+閾値）

### Desirability（欲しいか）

| ID | 仮説 | 崩壊影響 | 現状エビデンス | テスト + 事前閾値 |
|---|---|---|---|---|
| D1 | 個人開発者は「エージェントの暴走・自己申告・仕様乖離」を上位の痛みと感じ、規律で解決したい | **クリティカル** | 本人（P1）がドッグフーディングで継続利用＝実証済み（弱いが実在） | 継続利用の観測。閾値: 本人が3ヶ月継続利用しなくなったら棄却 |

## 崩壊クリティカル判定サマリ

| 崩壊クリティカル仮説 | 状態 |
|---|---|
| D1（痛みの実在） | **弱く検証済み**（本人ドッグフーディング。母数1） |
| D3（OSS 需要） | **未検証**（テスト定義済み・閾値未定義 = No-Go トリガ候補） |
| F1（機械検証の実現可能性） | **検証済み**（実装 + unit-test + CI） |

## Go / No-Go 裁定案（最終裁定は人間が PR レビューで行う）

**裁定案: 条件付き Go。**

- **根拠1（Feasibility 済）**: 崩壊クリティカルな F1 は実装・unit-test・CI で検証済み
  （`tests/test_spec_inspect.py`、SI-CORE-001〜003 で誤検知も是正）。F2/F3 も運用実績あり。
- **根拠2（Desirability の非対称性）**: D1 は本人ドッグフーディングで弱く検証済み。ただし D3
  （OSS 外部需要）は**未検証かつ事前閾値が未定義**（テスト自体は仮説表で定義済み）——
  本来なら No-Go トリガに該当する。
- **根拠3（遡及 discovery の特殊性）**: 本プロダクトは既にリリース済みで**主要ペルソナ（P1）
  向けには設計着手可否を問う段階を過ぎている**。よって「P1 主要スコープは Go、D3（OSS 拡大）は
  Go の前提にしない open 仮説として追跡」とするのが妥当。

**条件（Go の付帯）**:
1. D3（OSS 需要）は崩壊クリティカルな open 仮説として明示追跡し、OSS 拡大に賭ける施策
   （オンボーディング改善等）に着手する前に、fake-door/導入計測＋事前閾値を定義する。
2. V2（本文肥大化）は 3段階読み込み（SI-CORE-013）で早期に対処する。
3. 指標の実測基盤（metrics.md の `TBD` 群）整備を別 issue 化する。

> **最終裁定は人間が本ファイルを含む PR のレビューで行う**（sdd-core のゲート規約が正）。
> エビデンスが届いたらゲートは再訪し、一度きりの儀式にしない。docs/ 該当ドラフトが影響を
> 受けたら再伝播プロトコルに乗せる。
