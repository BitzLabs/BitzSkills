# 6. 内部処理とアーキテクチャ詳細 (Internal Processing)

本章は、Claude Code の「表からは見えない内部動作」を、2026年7月時点の公式ドキュメント（code.claude.com / platform.claude.com）で裏取りした内容に基づいて解説します。エージェントループ・権限評価・コンテキスト圧縮・CLAUDE.md 読み込み・サブエージェント分離・タスク管理の6点を扱います。

---

## 6.1 エージェントループと権限評価の内部順序

Claude Code のエージェントループは、モデルが `tool_use` を出力するたびに「マッチャー評価 → フック評価 → パーミッションルール評価 → 実行 → 結果のフィードバック」という決定論的な段階を踏みます。ここで最も重要なのが判断ソースが競合したときの優先順位で、**`deny`（拒否）が常に最優先**されます。

具体的には、`settings.json` の `permissions.deny` ルールおよび `ask` ルールは、`PreToolUse` フックが何を返したかに関わらず必ず評価され、フックが `"allow"` を返してもそれだけで `deny` ルールを覆すことはできません。逆に、`PreToolUse` フックが終了コード `2` でブロックした場合や `permissionDecision: "deny"` を返した場合は、`bypassPermissions` モードや `--dangerously-skip-permissions` を使っていても強制的にブロックされます。複数の `PreToolUse` フックが競合した場合の優先順位は `deny > defer（判断保留）> ask > allow` です。

この設計により、「フックで許可したつもりが上位の deny ルールで弾かれる」「bypass モードにしたのにフックの deny だけは効く」といった直感に反する挙動が起こり得ます。権限設計はこの評価順序を前提に組む必要があります。

## 6.2 コンテキストウィンドウ管理と自動圧縮の3層構造

Claude Code は単一の「`/compact` で丸ごと要約」だけでなく、コスト効率のために3段階の圧縮機構を持ちます。

第一段階の **Microcompact** はモデル呼び出しを伴いません。入力トークンが閾値（目安180K）を超えた際に、ファイル読み込み・シェル出力・grep/glob・Web取得結果などツール実行結果のうち古いものだけを自動的に間引き、直近分のみを目標トークン量（目安40K）まで残す軽量処理です。

第二段階の **フル Compact**（`/compact` または自動発火）は、専用のモデル呼び出しで会話全体を要約し直す重い処理です。自動発火のトリガー閾値は環境変数 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` でコンテキスト容量に対する割合として調整可能（既定は概ね95%前後）で、要約結果を書き込むため常に一定のトークン枠（目安2万トークン）を予約します。

第三段階の **セッションメモリ圧縮** は、あらかじめ抽出したノートを使うことでモデル呼び出し自体をスキップする最も軽量な圧縮です。加えて、圧縮が3回連続で失敗した場合は自動圧縮を停止するサーキットブレーカーが組み込まれており、暴走的な API 消費を防ぎます。

## 6.3 CLAUDE.md の階層読み込みと遅延ロード

`CLAUDE.md` は次の4階層で解決されます。

1. **エンタープライズ管理ポリシー**（macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`、Linux/WSL: `/etc/claude-code/CLAUDE.md`、Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`）
2. **ユーザーレベル**（`~/.claude/CLAUDE.md`）
3. **プロジェクトルート**
4. **サブディレクトリ**

上位（プロジェクト）の指示は下位（ユーザー個人設定）より優先され、`.claude/rules/*.md` のようにモジュール化されたルールファイルは、より広範なルートの記述より優先して適用されます。

特徴的なのは、**サブディレクトリの `CLAUDE.md` はセッション開始時には読み込まれず、エージェントがそのディレクトリ内のファイルに実際にアクセスした時点で遅延ロードされる**点です。これは新設された `InstructionsLoaded` フックイベントの発火条件（`session_start` / `nested_traversal` / `path_glob_match` / `include` / `compact` などのマッチャー値）からも裏付けられます。

## 6.4 サブエージェントのコンテキスト分離

サブエージェント（`Task` ツール経由で起動）は、親セッションとは完全に独立したコンテキストウィンドウ・システムプロンプト・ツールアクセス範囲・パーミッションモードを持ちます。親エージェントがサブタスクを委譲すると、探索的なファイル読み込み・検索結果・試行錯誤の過程はすべてサブエージェント側のコンテキストに閉じ込められ、**親に返るのは最終出力のみ**です。

これにより、大量のログ解析やテスト実行結果、ドキュメント取得のような「ノイズの多い」作業をサブエージェントに逃がし、親のコンテキスト消費を抑えられます。独立した `Task` 呼び出しを複数投げれば並列実行も可能で、これがサブエージェントパターンの主要な性能上の利点です。

---

## 6.5 Todo / Plan の内部管理とタスクツール群

非自明なタスクに対して、Claude Code はまずステップバイステップの実行計画をセッション内の短期記憶として保存し、状態（未着手 / 進行中 / 完了）をリアルタイムに更新しながら進めます。これはユーザーへの「意図確認」の役割も果たし、Todo がユーザーの意図と一致していれば指示が正しく伝わっている確認になります。

なお、TypeScript Agent SDK 0.3.142 / Claude Code v2.1.142 以降、内部実装は単一の `TodoWrite` ではなく **`TaskCreate` / `TaskUpdate` / `TaskGet` / `TaskList`** という粒度の細かいツール群に置き換えられています。新規タスクは `TaskCreate`、状態変更は `TaskUpdate` と呼び出しが分割されるため、SDK 側でツール呼び出しを監視するコードはタスク ID をキーにしたマップで管理する設計が必要です。

> [!NOTE]
> 実務上の注意として、Todo に登録するステップ数が7以上になると、モデルが効率化のためにステップをまとめて実行し、途中経過を可視化するという Todo 本来の意図が損なわれる傾向が報告されています。
