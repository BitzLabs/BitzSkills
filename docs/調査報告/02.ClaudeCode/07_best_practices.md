# 7. ベストプラクティス (Best Practices)

Claude Code を安全かつ効率的に運用するための実務指針を、公式ドキュメントの推奨に沿ってまとめます。

---

## 7.1 CLAUDE.md の書き方

`CLAUDE.md` は「エージェントへの事前学習資料」であると同時に、階層構造の中で **最も上位（エンタープライズ）が最も汎用的な安全側のルール、最も下位（サブディレクトリ）が最も具体的な例外ルール** という役割分担を意識すると、指示の衝突が起きにくくなります。

ルートの `CLAUDE.md` を肥大化させず、特定ディレクトリだけに適用したい規約は `.claude/rules/*.md` や当該ディレクトリ直下の `CLAUDE.md` に切り出すことで、遅延ロードの恩恵（無関係な作業でトークンを消費しない）を受けられます。依存バージョンやブランチ名のように頻繁に変わる情報は陳腐化するため避け、ビルド／テストコマンド・コードスタイル・危険操作の禁止事項など「安定して真であり続ける事実」に絞るのが定石です。

---

## 7.2 スキル設計と Progressive Disclosure

スキルは `name` と `description` の段階でのみ最初に読み込まれます。したがって `description` に「いつ使うべきか」を具体的なトリガー条件として書き切ることが最重要です。本文や `references/` は実行時に必要になって初めて読み込まれるため本文側は詳細でも問題ありませんが、`description` が曖昧だとそもそも活性化されません。

本リポジトリの規約どおり、`metadata`（version / author / created / updated）の運用を徹底し、変更のたびに semver でバージョンを上げる規律を守ることで、スキルの改善履歴を追跡可能にします。

---

## 7.3 フック活用

フックは「モデルの判断に頼らない決定論的な制御層」として位置づけるのが安全です。セキュリティ上重要な制約（`.env` へのアクセス禁止、`rm -rf` の禁止など）は、CLAUDE.md やプロンプトへの指示だけに頼らず、`PreToolUse` フックの `permissionDecision: "deny"` として実装することで、プロンプトインジェクションを受けても迂回されない防御になります。

`matcher` はツール名だけでなく `if` フィールドで Bash のサブコマンド単位まで絞り込めるため、「Bash は許可するが `git push` 系だけ確認する」といった粒度の設計が可能です。フックのタイムアウト（既定600秒、`UserPromptSubmit` は30秒など）を把握し、重い処理は非同期化することも実運用上重要です。

---

## 7.4 パーミッション設計

評価順序が `deny > ask > allow` であること、フックの `deny` だけは `bypassPermissions` でも効くことを前提に、「本当に自動許可してよいコマンド」だけを `allow` に、破壊的操作は明示的な `deny` またはフックでの二重ブロックに、という三層構造で設計します。

エンタープライズ環境では `managed-settings.json` に `allowManagedHooksOnly` を設定することで、ユーザー・プロジェクト・プラグイン由来のフックを無効化し、組織で承認済みのプラグイン経由のフックのみを強制適用できます。

---

## 7.5 MCP 運用

MCP サーバーは `.mcp.json` にタイムアウト（`"timeout": 600000` のように ms 単位）を明示しておくと、長時間かかるツール呼び出しでの意図しない切断を防げます。自前で Node 製 MCP サーバーを立てる場合、イベントループを維持する処理（`process.stdin.resume()` 等）がないとプロセスが早期終了し `-32000` エラーの原因になります。SSE ベースの HTTP MCP をリバースプロキシ（nginx 等）越しに置く場合はバッファリング無効化が必須で、怠ると `-32001` のタイムアウトエラーが頻発します。

---

## 7.6 コスト／トークン最適化

サブエージェントへの委譲でノイズの多い探索処理を親コンテキストから隔離すること、不要な巨大ファイルの全文読み込みを避けること、SDK 利用時は `maxTurns` と `maxBudgetUsd` の両方で暴走時の上限を明示することが基本線です。特に無人実行（CI/CD や監視システムからの自動修復エージェント起動など）では `maxTurns` を必ず設定し、OAuth トークンの有効期限切れによる長時間セッションの中断リスクを避けるため、対話的 OAuth より API キー認証を使う方が運用上安定します。

---

## 7.7 大規模リポジトリ・CI/CD 統合

SDK は既定でファイルシステム設定を読み込まないため、CLAUDE.md やプロジェクト固有のフックを CI 上でも効かせたい場合は `settingSources: ['project']` を明示指定する必要があります（第3章の GitHub Actions 例は `claude` CLI 経由なのでこの制約を受けませんが、SDK を直接利用する場合は要注意です）。認証は対話的 OAuth ではなく `ANTHROPIC_API_KEY` を使い、破壊的操作を防ぐため `--disallowed-tools` や managed-settings による許可リスト方式を徹底します。

---

## 7.8 セキュリティ運用

プロンプトインジェクション対策としては、外部コンテンツ（Web 取得結果、Issue 本文、PR 差分など）を信頼できない入力として扱い、Read/Edit/WebFetch の `deny` ルールで機密ファイル（`.env`、認証情報、`/etc` 配下等）への到達をツール層でブロックする多層防御が推奨されます。`/sandbox` によるサンドボックスモードの利用、ネットワークコマンドへの `deny` ルール、外部 SAST/DAST ツールとの併用、`--dangerously-skip-permissions` を本番運用で使わないこと、が繰り返し強調されています。

テレメトリは `env.DISABLE_TELEMETRY=1` で無効化できますが、これは同時にリモートのプレビュー機能やキルスイッチも無効化する副作用があるため、機能面への影響も理解した上で有効化する必要があります。
