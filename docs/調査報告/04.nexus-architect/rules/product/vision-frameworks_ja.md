# Rules: Vision Frameworks (define-vision)

`/product:define-vision`の参照用である。両方のフレームワークを適用する。Vision Boardはコアを設定し、PR-FAQはそれにプレッシャーテストを行う。

## Product Vision Board (Roman Pichler) — 5 elements

1. **Vision (ビジョン)** — 究極の目的。プロダクトが世界にもたらす変化。インスピレーションを与える、記憶に残る1文。大きく野心的であること。
2. **Target Group (ターゲットグループ)** — 誰がユーザーであり顧客であるか。**セグメント化すること — 決して「全員」としないこと。**
3. **Needs (ニーズ)** — 解決される問題または提供されるメリット。なぜユーザーが乗り換えるのか。
4. **Product (プロダクト)** — 際立ったいくつかの機能 / 主要な差別化要因（完全な機能リストではない）。
5. **Business Goals (ビジネス目標)** — プロダクトがどのように会社に利益をもたらすか（収益、成長、維持）。

良いビジョンとは、野心的であり、インスピレーションを与え、共有され、簡潔で記憶に残るものである。

**Mission vs Vision vs Values**: ビジョン＝目的地（世界における変化）。ミッション＝それがどのように追求されるか（プロダクトが誰のために、日々の何を行うか）。バリュー＝トレードオフが生じたときにチームが保持する意思決定の原則である。記入されたVision Boardからミッションとバリューを導き出す。

## Working Backwards — PR-FAQ (Amazon)

プロダクトがすでに出荷されたかのように書く。売り込みではなく、真実の探求である。

**プレスリリース**（約1ページ）:
- 見出し — プロダクト名
- 副見出し — ターゲット顧客 + コアとなるメリット（1行）
- 概要の段落 — それが何であるか、誰のためのものか、見出しとなるメリット
- 問題の段落 — 顧客のペインを、**顧客自身の言葉で**
- 解決策の段落 — プロダクトがそれをどのように解決するか。**差別化**
- 引用とCTA — 役員の引用 + 顧客の引用、そして開始方法

**FAQ**（2層構造）:
- **外部FAQ** — 価格、機能性、購入方法、サポート
- **内部FAQ** — 市場規模（TAM/SAM/SOM）、競合状況、ユニットエコノミクス、運用上の課題、リスク、および**Go/No-Go基準**

> Go/No-Go基準は耐荷重性を持つ。これらは後で`/product:validate-assumptions`によって強制される。これらを具体的かつ反証可能なものにすること（例：「インタビューを受けたターゲットユーザーのX%が、この問題を彼らのトップ3にランク付けした場合のみGoとする」）。

## ID convention

各ボード要素および各主要なPR-FAQの主張は、アップストリーム列（ビジョンのルートの場合は空）を持つ`VIS-xxx` IDを取得する。すべての`VIS-`ノードを`work/traceability.json`に追記する。
