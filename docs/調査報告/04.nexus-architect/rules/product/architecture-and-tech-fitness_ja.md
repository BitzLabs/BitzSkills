# Rules: Architecture Synthesis & Technology Fitness (design-architecture)

`/product:design-architecture`の参照。蓄積された設計（Bounded Context (境界づけられたコンテキスト)、APIレイヤー、データモデル、NFR）を、一貫性のある**ランタイムアーキテクチャ**に統合（synthesize）し、**プラットフォーム技術の標準セットの適合性**を評価し、それぞれの採用決定とその根拠を出力する。

アーキテクチャは*統合*であり、新たな発明ではない：すべての図のすべてのノードは、既存の`CTX-`/`API-`/`ENT-`/`NFR-` idにマッピングされなければならない。もし図に、どの成果物も正当化しないコンポーネントが必要な場合、それはフラグを立てるべきギャップである — 機能をでっち上げてはならない。

## Part 1 — Architecture diagrams (Mermaid, traceable)

**3つの補完的なビュー**を作成する（プロジェクトの成果物がサポートするものを使用する。スキップされたものはメモする）：

1. **ランタイム / コンテナビュー** (`flowchart`) — 外部アクターとシステム → エッジ/イングレス → Experience API → Process API → System API → データストア。Bounded Contextごとにグループ化し、各ノードにCore / Supporting / Generic（`domain-map.md`から）のタグを付ける。主要なデータフローと、インライン/クリティカルパスを示す。これが主要な成果物である。
2. **クリティカルパスビュー** (`sequence`) — 最も差別化要因となる、または最もリスクの高い単一のリクエストフロー（例：インラインの適用パス、チェックアウトサーガ）。レイテンシ/障害モードを明示的にする。
3. **デプロイメント / スケーリングビュー** (表または`flowchart`) — コンテキストがデプロイ可能なサービスにどのようにマッピングされるか（マイクロサービス/モノリスの制約を尊重する）、およびサービスごとに異なるスケール特性（スループット vs. 低レイテンシ vs. 永続性）。それぞれを、それを管理する`NFR-`/`SLO-`に紐付ける。

図の規律:
- サブドメインの階層によって色分けする。ラベルは簡潔に保ち、ノードラベルの改行には`<br/>`を使用する。
- 差別化要因（★）とデフォルトの障害モード（例：フェイルオープン）をインラインでマークする。
- すべてのコンポーネントはidに追跡される。図の下に追跡マッピングをリストアップする。
- アップストリームの成果物が境界や数値を未決定のままにしている場所には`TBD`を記録する（ごまかさないこと）。

## Part 2 — Technology fitness assessment (evidence-driven)

プラットフォーム技術の**標準チェックリスト**を、プロジェクト*自身*の成果物に対して評価する。適合性は、直感からではなく、すでに設計にあるシグナルから読み取られる。各技術について以下を出力する：

- **適合性**: High / Medium / Low / None
- **見つかったトリガーとなるシグナル**: （`NFR-`/`CTX-`/`ENT-`/`API-`/`CON-`/`SCP-` idを引用する）
- **決定**: Adopt / Conditional / Reject
- **根拠**: その理由と、採用された場合は*アーキテクチャのどこに組み込まれるか*（どのレイヤー/コンテキスト）。却下された場合は、どの既存の成果物がそれを不要にするか。
- **条件 / リスク**: ConditionalがAdoptになるために何が真でなければならないか（オープンな質問、`TBD`）。

標準チェックリスト（実行ごとに4つすべてを評価する）:

### Kong (API Gateway)
- **What**: クラウドネイティブなAPIゲートウェイ / イングレス。authN/Z（OIDC、JWT、キー認証、ACL）、レート制限、mTLS、リクエスト変換、トラフィック制御（カナリア）、およびエッジの可観測性のためのプラグイン。
- **Adoptとなるシグナル**: **複数のコンシューマー/チャネル**を持つExperience/Process APIレイヤー。各サービスに存在すべきではない横断的なエッジの関心事（認証、レート制限、クォータ、mTLS）。終端すべき外部IdP（OIDC）。パートナー/公開APIサーフェス。単一のイングレスを必要とするマイクロサービス群。**Experience APIの前段のエッジ**にマッピングされる。
- **Rejectとなるシグナル**: 単一のサービス / 外部コンシューマーなし。制約によってすでにマネージドゲートウェイが義務付けられている。APIサーフェスが内部専用であり、自明である。

### ScalarDB (universal transaction manager)
- **What**: **複数/異種データベース間およびマイクロサービス間でのACIDトランザクション**（Two-Phase Commit (二相コミット)を含む）を提供する汎用トランザクションマネージャーであり、ポリグロットなバックエンドを単一のトランザクションAPIの下に抽象化する。エディション：Core (OSS) / Enterprise (クラスター)。
- **Adoptとなるシグナル**: **複数の集約/コンテキスト**にまたがる不変条件を伴う**マイクロサービス制約**（`CON-` microservices）。**ポリグロット永続化**。そうでなければ壊れやすい**サーガ**を強要するような、強い一貫性の必要性。`api-design`で可視化されるサービス間のトランザクション書き込み（複数のSystem APIをアトミックに変更するProcess API）。**Systemレイヤー / データアクセス**にマッピングされる。
- **Rejectとなるシグナル**: サービス間トランザクションのない単一のデータストア。結果整合性がどこでも許容される、純粋な追記型 / イベントソーシングモデル。
- **Note**: 採用された場合、`select-scalardb-edition` / `design-scalardb`（architectプラグイン）を続けるべきかどうかフラグを立てる — これがnexus-architectのScalarDBパイプラインへのブリッジとなる。

### ScalarDB Analytics
- **What**: ScalarDBが管理するデータ上の分析（HTAP）クエリレイヤー — ウェアハウスへの個別のETLなしで、オペレーショナルストア全体にわたるフェデレーテッド分析/レポートを提供する。
- **Adoptとなるシグナル**: 別のウェアハウスを立ち上げるのが時期尚早であるような、**コンテキストにまたがる**分析/レポート/メータリング要件（例：ノーススター/課金指標のための使用量メータリング、オペレーショナルデータ全体のコンプライアンスレポート、クロスエンティティのダッシュボード）。ScalarDBの適合性がMedium以上であることを前提とする。トランザクショナルストアの横の**読み取り/分析プレーン**にマッピングされる。
- **Rejectとなるシグナル**: 分析ワークロードがない。専用のウェアハウス/レイクがすでにスコープ内にある。分析が単一コンテキストであり、オペレーショナルストアによって自明に提供される。

### ScalarDL (tamper-evident ledger)
- **What**: ビザンチン障害検知ミドルウェア — 実行の検証可能性/ファイナリティを備えた、**改ざん検知可能でデジタル署名された台帳**。許可されていないデータ変更を検知する（否認防止）。
- **Adoptとなるシグナル**: **改ざん検知 / 完全性 / 否認防止 / 検証可能な監査証跡**の要件（`NFR-`の完全性、追記型の`ENT-`監査台帳、`SCP-`の改ざん防止の証拠機能、規制データの保持）。手作りのハッシュチェーンを置き換えるか、またはそれを裏付けるものとして、**監査/台帳コンテキスト**にマッピングされる。
- **Rejectとなるシグナル**: 完全性/監査/否認防止の要件がない。通常のCRUDの耐久性で十分である。

### Extending the checklist
上記の4つは必須である。成果物がカバーされていないカテゴリー（例：メッセージブローカー、ベクトルストア）を明確に求めている場合は、プロジェクトに関連する候補を追加する — 同じ証拠駆動のフォーマットで。いかなるシグナルもサポートしない技術でリストを水増ししてはならない。

## Discipline
- **熱意よりも証拠を重視する。** 適合性がHighの場合は、1つ以上の具体的な成果物idを引用しなければならない。引用がない場合はLow/Noneとなる。
- **決定は可逆的なインプットである。** NFRやスコープが変化した際に`adapt-change`が再検討できるように、根拠を記録する。
- **制約を尊重する。** `CON-`（例：マネージド専用）に違反する技術は、競合を明記した上でConditional/Rejectとする。
- **採用をでっち上げてはならない。** 理由を伴う「Reject」は、有効で価値のある出力である。

## ID convention & handoff
アーキテクチャのビュー/コンポーネントには`ARCH-`、技術適合性の決定には`TECH-`。アップストリームの`CTX-`/`API-`/`ENT-`/`NFR-`/`CON-`/`SCP-`参照とともに`work/traceability.json`に追加する。ScalarDB/ScalarDLの**Adopt**は、**architect**プラグイン（`/architect:select-scalardb-edition`、`/architect:design-scalardb`、`/architect:design-scalardb-analytics`）へのブリッジとなる。
