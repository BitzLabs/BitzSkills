# Rules: UI → Feature → Data Derivation (generate-ui-mock, define-features, define-data-model)

ジャーニーを画面に、画面を機能に、そして機能をデータモデルに変換するPhase 3チェーンの参照用である。その中核をなすのは**Domain Storytelling (ドメインストーリーテリング)**だ。UIは具体的なストーリーであり、そこから機能とエンティティが読み取られるため、各レイヤーはその上のレイヤーをトレースすることになる。

## generate-ui-mock — solution exploration, then render

**「最初のスケッチがデザインを固定化してしまう」リスクを軽減する。**描画する前に、各*優先ジョブ*について以下のことを行う：

1. 1〜2行で**2〜3のソリューションアプローチ**をスケッチする。
2. それらを簡潔に比較する（ジョブ、ジャーニーの機会、ポジショニングと照らし合わせて）。
3. **明確な根拠とともに1つを選択する。**

次に、選択したアプローチに必要な画面を列挙し、**Domain Storytellingの番号付けされたアクティビティ順に並べ**（1つのアクティビティ≒1つの画面ステップ）、要素を配置し、各画面を**インラインCSSを含む自己完結型のHTMLファイル**として`ui-mocks/`以下にレンダリングする。各画面には、それが機能する`STORY-`アクティビティ / `JNY-`機会 / `JOB-`の注釈を付ける。

- **ストーリーをクリッカブルなフローに結びつける。**アクティビティの順序はクリックパスである。各画面のフローを進めるアクションは、次のアクティビティの画面への実際の`<a href>`であり、前の画面への戻るリンクと`step N of M`のインジケーターを備える。ファイル名はストーリーとステップをエンコードし（`{STORY}-NN-{slug}.html`）、リンクが決定論的になるようにする。ストーリーごとの`{STORY}-index.html`はアクティビティの順序をリストし、ステップ1にリンクする。欠落しているステップは無効化された`TBD`リンクであり、決して行き止まりではない。分岐は対象の画面にリンクする。読者は**ストーリー全体を最後までクリックして進めることができなければならない**。
- **忠実度のルール**：視覚的な洗練さではなく、*「読者が各画面にどの機能とどのデータが含まれているか分かること」*に最適化する。ローファイで構わないが、曖昧であってはならない。
- 各モックはランダムな最初のドラフトではなく、「選択されたソリューションのレンダリング」である。

## define-features — Action → Command

各画面のインタラクティブな要素を読み取る：

- 各**ユーザーアクション** → **Command (コマンド)**（機能の候補）。動詞を先頭にして命名する。
- **スコープとのすり合わせ** — アウトオブスコープ（`SCP-` Won't）にあるものはすべて削除し、Should/Couldは延期する。
- 画面間で**重複を統合**し、**MoSCoW**の優先度を割り当てる。
- 各機能（`FEAT-`）には、名前、説明、画面、および`JOB-`/`JNY-`/`NSM-`にトレースする根拠を記録する。

**空入力のガード**：`ui-mocks/`が空の場合、**停止して報告する** — 決して空の機能リストを出力してはならない（ダウンストリームへの空の伝播を防ぐため）。

## define-data-model — two passes

**Pass 1 — 明示的なエンティティ（UI + 機能から）：**
- 画面/概念名 → 概念の候補
- フォームフィールド → 属性
- アクションの結果 → 状態 / イベント
- **Commandを受信し、Event (イベント) を発行する**集約 → エンティティ

**Pass 2 — 暗黙的なエンティティ（機能 × エンティティのCRUDマトリックス）：**
- CRUDマトリックス（機能 × エンティティ）を構築する。ギャップや交差から暗黙的な概念が明らかになる：**多対多の結合、履歴/バージョニング、監査ログ、ステートマシン/ステータス**のエンティティ。
- それらを追加し、それぞれの追加に対する**機能的関係の根拠を記録する**。

**エンティティの基準**：概念がエンティティになるのは、それが**永続化**され、**識別子/キー**を持ち、**複数の操作を受信する**場合のみである。過度な正規化を避ける（すべての属性をエンティティに昇格させない）。最終的なモデル（明示的 + 暗黙的）の**Mermaid ER図**を作成する。

## ID convention

機能には`FEAT-`、エンティティには`ENT-`を使用する。ui-mocksは`JNY-`/`JOB-`を注釈として付ける。すべてを`work/traceability.json`に追記し、アップストリームへの参照（`JNY-`/`JOB-`/`SCP-`/`NSM-` → `FEAT-` → `ENT-`）を含める。
