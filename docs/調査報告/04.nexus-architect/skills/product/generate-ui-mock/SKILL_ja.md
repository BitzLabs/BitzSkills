# UI Mocks

## Desired Outcome

1つの成果物を作成します：

1. **ナビゲーション可能な UI モック（Navigable UI mocks）** — `reports/02_spec/ui-mocks/`（画面ごとに1つの自己完結型 HTML ファイル。インライン CSS。クリッカブルプロトタイプに接続されている）:
   - 優先ジョブごとの短い**ソリューション探索ノート（solution-exploration note）**: 2〜3 のアプローチを比較し、根拠とともに1つを選択します。
   - 選択されたアプローチが必要とする画面で、**そのペルソナ×ジョブのドメインストーリーから派生したもの（derived from the domain story for that persona×job）** — ストーリー内の番号が振られた各アクティビティは画面のインタラクションになります — それぞれに、それが奉仕する `STORY-`/`JNY-`/`JOB-` と、それが使用する `CMP-` コンポーネントがアノテーションとして付けられます。
   - **画面遷移のための配線（Wired for screen transitions）**: 各画面のフローを進めるアクションは、ストーリーの次のアクティビティの画面への実際の `<a href>` となり、読者は**ストーリー全体を最初から最後までクリックして進めることができます（click through the whole story end to end）**。ストーリーのシーケンスはファイル名（`{STORY}-NN-{slug}.html`）にエンコードされます。各画面には「戻る」リンクとステップインジケーターが備わります。ストーリーからの分岐/代替パスは、ターゲットの画面にリンクします。欠落しているステップは、行き止まりではなく無効化された `TBD` リンクになります。
   - ストーリーごとの**フローインデックス（flow index）**（`{STORY}-index.html`）。これは、アクティビティのシーケンスをリストし、ステップ 1 にリンクします — そのストーリーをクリックして進めるためのエントリポイントです。
   - **アクティブなデザインシステムによるスタイリング（Styled by the active design system）**: システムの `tokens.css` が各画面に注入されるため、すべてのモックが1つの視覚的言語（カラー/スペース/タイプ）を共有します。**mid** フィデリティでは、コンポーネントスタイルも適用されます。**lo** フィデリティでは、トークンのみです。
   - `--fidelity`（デフォルトは `lo`）ごとのフィデリティ — 視覚的な洗練さよりも、読み取り可能な機能とデータ。

## Invocation

```
/product:generate-ui-mock [--fidelity=lo|mid] [--auto] [--lang=ja|en]
```

| Argument/Flag | Required | Description |
|---------------|----------|-------------|
| `--fidelity=lo\|mid` | Optional | `lo`（デフォルト）= トークンのみ。`mid` = トークン + デザインシステムのコンポーネントスタイル。アクティブなシステムの `fidelity_support` によって上限が制限されます。 |
| `--auto` | Optional | ファシリテーションなしで生成します。最も正当化されたアプローチを自動的に選択します |
| `--lang` | Optional | 出力言語（UIコピー）を上書きします |

## Decision Criteria

- **ドメインストーリーが軸であり、画面はそれに配線されます。** ペルソナ×ジョブのドメインストーリーが存在する場合、画面フローはストーリーの番号付きアクティビティ（1アクティビティ ≈ 1画面インタラクション）に従い、アクティビティの*順序*が**クリックパス（click path）**になります: 各画面のフローを進めるアクションは次のアクティビティの画面にリンク（`<a href>`）するため、読者はストーリーをステップ実行できます。ストーリーと矛盾するフローを発明しないでください。ストーリーにステップが欠けている場合は、リンクを無効化された `TBD`（決して行き止まりではない）としてレンダリングし、それを書き留めます。
- **ナビゲーションはモックの一部であり、別の関心事ではありません。** すべての画面は、ストーリーのフローインデックスから最後までクリックして到達可能です。ファイル名はストーリーとステップ（`{STORY}-NN-{slug}.html`）をエンコードするため、リンクは決定的（deterministic）になります。すべての画面には「戻る」リンクとステップ N-of-M のインジケーターがあります。ストーリーの分岐はターゲットの画面にリンクします。ストーリー順にクリックして進められないモックは不完全です。
- **デザインシステムが視覚的言語です。** `work/pipeline-progress.json` → `options.design_system` からアクティブなシステムを解決します。その `tokens.css` をすべての画面に注入し、トークンを参照します（ハードコードされた値は決して使用しません）。設定されたシステムがない場合 → アドホックな lo-fi スタイリングにフォールバックし、その旨を注記します。競合するパレットを黙って発明することは決してしないでください。
- **描く前に探索する。** 優先ジョブごとに2〜3のアプローチを比較してスケッチし、明示的な根拠とともに1つを選択します — モックは、ストーリーのフローのために*選ばれた*ソリューションのレンダリングです。
- **フィデリティよりも可読性。** 読者は各画面にどのような機能とどのようなデータが含まれているかを判別できなければなりません。Lo-fi は問題ありませんが、曖昧なものは不可です。
- **すべての画面を**、`STORY-` アクティビティ（存在する場合）/ `JNY-` 機会 / `JOB-` に**トレースする**こと。
- **終了条件（Stop condition）**: 各優先ジョブには選択されたアプローチとその画面が自己完結型 HTML としてレンダリングされており、各画面に `STORY-`/`JNY-`/`JOB-` のアノテーションが付けられていること。**さらに、ストーリー全体がフローインデックスから最後のアクティビティまでクリック可能であること**（フローを進めるすべてのアクションが次の画面にリンクしており、ギャップは行き止まりではなく `TBD` リンクになっていること）。

## Process

1. **コンテキストの読み取り** — ドメインストーリー、ジャーニーの機会（journey opportunities）、ポジショニング、ペルソナ、アクティブなデザインシステム（`options.design_system` → `design-system/{name}/tokens.css` + `components.md`）、`work/traceability.json`。
2. **ソリューションの探索（Explore solutions）** — 優先ジョブごとに、2〜3 のアプローチをスケッチし、比較し、根拠とともに選択します。`@rules/product/ui-to-domain.md` を適用します。
3. **画面の列挙と順序付け（Enumerate screens & order them）** — ドメインストーリーの番号付きアクティビティをウォークし、それぞれを画面のインタラクション（1アクティビティ ≈ 1画面ステップ）にマッピングし、アクティビティの順序を画面のシーケンスとして保持します。ストーリーのないペルソナ×ジョブについては、ジャーニーの機会にフォールバックします。各画面に決定論的なファイル名 `{STORY}-NN-{slug}.html`（NN = ゼロ埋めされたステップ）を割り当てます。分岐/代替パスや、欠落しているステップ（→ `TBD`）があれば書き留めます。
4. **レイアウトとレンダリング（Lay out & render）** — 要素の配置 → 画面ごとに自己完結型 HTML を作成し、デザインシステムの `tokens.css` をインラインで注入し、トークンを参照します。`--fidelity=mid` では `CMP-` コンポーネントスタイルも適用します。`lo` ではトークンのみです。アクティブなシステムがない場合 → アドホックなインライン CSS（注記あり）。
5. **フローの配線（Wire the flow）** — 各画面で、フローを進めるアクションを次のアクティビティの画面への実際の `<a href>` にします。前の画面への「戻る」リンクと `step N of M` インジケーターを追加します。分岐をターゲット画面にリンクさせます。欠落しているステップは、無効化された `TBD` リンクとしてレンダリングします。アクティビティシーケンスをリストしステップ 1 にリンクするストーリーごとのフローインデックス `{STORY}-index.html` を記述します。
6. **アノテーション（Annotate）** — 各画面に `STORY-`（存在する場合）/ `JNY-` / `JOB-` と、使用されている `CMP-` のタグを付けます。
7. **ナビゲーション可能性の検証（Verify navigability）** — すべての画面がフローインデックスから最後のアクティビティまでクリックして到達可能であり、行き止まりがないことを確認します（ギャップは `TBD` リンクとしてのみ現れます）。
8. **トレーサビリティの追記** — 上流の `STORY-`/`JNY-`/`CMP-` 参照、`next`/`prev` の画面エッジ、および選択されたアプローチの根拠を持つ画面ノードを `work/traceability.json` に追加します。
9. **記録** — HTML ファイル + フローインデックス + 探索ノートを書き込みます。決定事項を `work/context.md` に追記します。

## Output

`reports/02_spec/ui-mocks/` — 画面ごとに1つの自己完結型 HTML ファイル（`{STORY}-NN-{slug}.html`）、アクティビティシーケンスをリンクするストーリーごとのフローインデックス `{STORY}-index.html`、およびソリューション探索ノート。各画面は `STORY-`/`JNY-`/`JOB-` でアノテーションされ、戻る/次へ のナビゲーションを持ち、フローインデックスからストーリーの順にクリックして到達可能です。
