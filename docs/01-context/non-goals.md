# スコープ

## 標準環境の定義

BitzSkills が定義する「標準環境」は、OSイメージや言語SDK一式ではなく、
**AIエージェントと開発する際の作業規律・再利用ワークフロー・検証・配布の層**を指す。
単一の巨大プラグインではなく、用途別プロファイルとして構成する。

## 標準プロファイル

| プロファイル | 構成 | 用途 |
|---|---|---|
| 標準開発 | `bitz-env` + `bitz-sdd` | ガードレールと仕様駆動開発を使うBitzLabsの既定 |
| DDD拡張 | 標準開発 + `bitz-ddd` | 複雑なドメイン設計、DDD成熟度評価 |
| 軽量Git | `bitz-env` + `bitz-flow` | `.spec` を採用せず、安全なGit運用だけを標準化 |
| スキル開発 | 標準開発 + `skill-creator` | Agent Skills の作成・検証・改善・配置 |
| プラグイン開発 | 標準開発 + `plugin-creator` | Claude Code / Antigravity 向けプラグイン開発。収録スキルはCodexでも利用 |

`bitz-sdd` は現時点で `sdd-git` を含むため、同じ目的で `bitz-flow` を重複導入しない。
責務移管（SI-CORE-010）後にプロファイルを再評価する。

3プラットフォームへの配布は、全コンポーネントの機能同等性を意味しない。特に `bitz-env` の
機械フックと `env-init` は現在 Claude Code / Antigravity を対象とし、CodexはスキルとAGENTS規律を
利用する段階である。Codex向けの完全対応または正式な劣化モードは SI-ENV-023 で追跡する。

## MoSCoW

### Must

- Agent Skills 準拠の自己完結スキルをプラグインとして配布する。
- Claude Code / Antigravity 2.0 / Codex CLI の3マニフェストを整合させる。
- 危険操作、権限境界、Git規律、検証義務を共通ガードレールとして提供する。
- SDDによる要件・設計・実装・検証のトレーサビリティを提供する。
- インストール、更新、検証、撤去の安全な手順を文書化する。
- `release_check.py`、spec inspect、pytestの実出力で変更を検証する。

### Should

- 用途別プロファイルで最小限の導入構成を案内する。
- プラットフォーム差を各プラグインの境界に隔離し、中核スキルを共有する。
- doctor / update / migration により導入後のドリフトを検出・解消する。
- ドッグフーディングで標準環境自体を同じ規律により開発する。

### Could

- 標準プロファイルの一括導入・健全性確認コマンド。
- 適用プロジェクトの匿名・ローカル集計。
- プロジェクト種別ごとの追加プロファイル。

### Won't（現段階）

- OS、コンテナ、IDE本体、言語ランタイム、SDK、パッケージマネージャーのプロビジョニング。
- アプリケーション固有のソース雛形、業務ライブラリ、データベース製品の標準化。
- クラウドアカウント、ネットワーク、実行基盤、デプロイ先の構築。
- 認証情報・APIキー・秘密情報の保存または集中管理。
- すべてのプロジェクトへの全プラグイン強制導入。
- 各AIプラットフォームの機能差を完全に隠蔽すること。

## 制約

- リポジトリ外への配置・上書き・削除はユーザー承認を必要とする。
- 各プラットフォームの一次資料またはCLI実測を仕様判断の根拠とする。
- 少人数保守を前提に、重複責務と同一情報の多重記載を増やさない。
- スキル変更時はスキルとプラグイン双方のversionを更新する。
