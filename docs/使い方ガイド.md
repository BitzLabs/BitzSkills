# BitzSkills 使い方ガイド

インストールから最初のプロジェクトまでを、目的別に案内する入門ガイド。
シナリオ別の具体例は [ユースケース集](ユースケース.md) を参照。

## 1. インストール

### Claude Code

```
/plugin marketplace add BitzLabs/BitzSkills
/plugin install bitz-sdd@bitzskills          # 仕様駆動開発
/plugin install bitz-ddd@bitzskills          # DDD 手法（bitz-sdd と併用）
/plugin install skill-creator@bitzskills     # スキル開発
/plugin install plugin-creator@bitzskills    # プラグイン開発
```

必要なものだけ入れてよい。開発中の動作確認は
`claude --plugin-dir <リポジトリ>/plugins/<plugin名>` でインストール不要。

### Antigravity 2.0

```bash
agy plugin install <リポジトリ>/plugins/<plugin名>
agy plugin list
```

## 2. どのプラグインを使うか

| やりたいこと | 使うプラグイン |
|---|---|
| アプリ・サービスを仕様駆動で開発したい | **bitz-sdd**（必要に応じて bitz-ddd） |
| 既存コードベースの設計品質を評価したい | **bitz-ddd**（ddd-evaluate） |
| エージェント用のスキルを作りたい・改善したい | **skill-creator** |
| Claude Code / Antigravity のプラグインを作りたい | **plugin-creator** |

## 3. bitz-sdd で開発を始める

### 3.1 最初のセットアップ（プロジェクトごとに1回）

エージェントに「docs を初期化して」と伝えると `sdd-docs` が docs/ ツリー
（MASTER.md・01-context〜08-knowledge）をテンプレートから立ち上げる。
以後、`.spec/`（仕様・設計・検証のマスター）と `docs/`（人間向けナラティブ）の
2層構造で進む。

### 3.2 フルワークフロー

「〜を仕様駆動で設計して」と伝えるとフローが始まる:

1. **上流探索**（`sdd-discovery`）: ビジョン・成功指標・スコープ・ペルソナ → 仮説検証ゲート（Go/No-Go は人間）
2. **設計**（`sdd-design`）: ドメイン概要 → API → アーキテクチャ。
   永続データがあれば `sdd-data`（ER 図・格納方式選定・移行計画）、
   インフラ・SLO・DR が要るなら `sdd-ops` を追加
3. **レビュー**（`sdd-review`）: 多観点並列レビュー → Design Gate（proposed → active は人間）
4. **実装**（`sdd-implement`）: 要件を `.spec/tasks/` へ分解（implements / depends_on / boundary）して実装。
   並列開発は `sdd-git` の worktree 運用
5. **検証**（`sdd-test`）: EARS 要件からテストを導出・実行し、結果を `.spec/specs/` に記録
6. **報告**（`sdd-report`）: 「進捗を教えて」で `.spec/reports/status-report.md` を自動生成

各ゲート（Discovery / Design / Promotion）の裁定は常に人間が行う。
エージェントは証跡を揃えて提示するだけで、自分で通過させない（権限分離）。

### 3.3 軽量レーン（小さな変更）

軽微な修正にフルワークフローは不要。**spec-issue 起票 → 要件化（approved は人間）→
タスク → 実装** だけで回せる。discovery / design はスキップしてよい
（公開 API や `.spec` スキーマなど契約に触れる変更を除く）。

### 3.4 機械検証

```bash
python3 <bitz-sdd>/skills/sdd-core/scripts/spec_inspect.py <repo-root>   # .spec の構造・traceability
python3 <bitz-sdd>/skills/sdd-docs/scripts/sdd_sync.py diff --root .     # .spec ⇄ docs の乖離確認
```

合否は機械が出し、人間はゲートと例外だけを見る、が BitzSDD の憲法。

## 4. bitz-ddd を重ねる（DDD 開発）

bitz-ddd を導入すると、設計工程が DDD 手法に置き換わる:

- 「ドメインストーリーを書こう」→ `ddd-story` がペルソナ×ジョブのハッピーパスを `.spec/design/stories/` に記述
- 「ドメインモデルを作って」→ `ddd-model` が Entity / Value Object / Aggregate と境界づけられたコンテキストを導出
- 「このコードの成熟度を測って」→ `ddd-evaluate` が DDD 12基準 + MMI 4軸で採点

未導入なら `sdd-design` が軽量ドメインスケッチにフォールバックし、フローは止まらない。

## 5. skill-creator でスキルを作る

1. 「新しいスキルを作りたい」→ `skill-creator` が対話的に設計（迷ったら「スキル開発の進め方が分からない」→ `skill-pipeline`）
2. 「検証して」→ `skill-validator` が仕様準拠をチェックリスト検査
3. 「テストして」→「評価して」→ `skill-tester` が スキルあり/なし比較を実行し、`skill-evaluator` が採点（成果物は `evals/<skill-name>/`）
4. 不合格なら「改善して」→ `skill-optimizer`
5. 「配置して」→ `skill-packager` が実環境（`~/.claude/skills/` 等）へ
6. 配置後は `skill-instrumenter`（計装）→ `skill-observer`（観察）→ `skill-improver`（提案）の自己改善ループ

## 6. plugin-creator でプラグインを作る

`/plugin-creator:create-plugin` で構想から完成まで 8 フェーズのガイド付きワークフローが走る。
個別の疑問はスキルが答える: 構成は `plugin-structure`、コマンドは `plugin-commands`、
エージェントは `plugin-agents`、フックは `plugin-hooks`、MCP は `plugin-mcp`、
設定は `plugin-settings`、スキル同梱は `plugin-skills`。

両対応（Claude Code + Antigravity）のポイント: マニフェストを2つ置き
（`.claude-plugin/plugin.json` とルート `plugin.json`）、中核はスキルで作り、
`model: inherit` を使う。仕様差の詳細は各スキルの references が持っている。

## 7. トラブルシューティング

| 症状 | 確認すること |
|---|---|
| スキルが発動しない | プラグインがインストール済みか（`/plugin` / `agy plugin list`）。発動フレーズは各 README の「発動の例」参照 |
| `spec_inspect.py` が FAIL | `.spec/inspection-report.md` の問題一覧（幽霊参照 / 孤児要件 / frontmatter）を見る。approved 以降の要件には implements するタスクが必要 |
| `sdd_sync.py` が上書きしそうで怖い | まず `diff` アクションで乖離を確認。同期は mtime 比較で新しい側だけが勝つ |
| プラグインの読み込みがおかしい | `claude --debug` / `agy plugin validate`。plugin-structure の references/troubleshooting.md |
