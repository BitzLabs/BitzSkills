# BitzSkills ユースケース集

「何ができるか」をシナリオ別に。手順の全体像は [使い方ガイド](使い方ガイド.md) を参照。

---

## UC-1: Web アプリ（MVC + Database）を仕様駆動でゼロから作る

**使うもの**: bitz-sdd

ToDo 管理アプリのような「画面・サーバー・DB」構成の定番開発。

1. 「個人用の ToDo 管理 Web アプリを仕様駆動で設計して」
2. `sdd-discovery` がビジョン・スコープ・ペルソナを `.spec/discovery/` に確立 → Go/No-Go を裁定
3. `sdd-design` がドメインスケッチ・API・アーキテクチャ（MVC レイヤリングとデータ層の対応を含む）を設計
4. `sdd-data` が ER 図（Mermaid）→ 格納方式の証拠駆動選定（SQLite? PostgreSQL? JSON ファイル?）→ 永続化戦略 → 移行計画
5. `sdd-review` の多観点レビュー判定を見て Design Gate を裁定
6. `sdd-implement` → `sdd-test` で要件 ID に紐づく実装と EARS 由来のテスト
7. 「進捗を教えて」でいつでも status-report

> このシナリオは実際に eval 済み（`evals/bitz-sdd/report.md`: ケース 3/3・アサーション 15/15 合格）。

## UC-2: 既存システムのリファクタリング判断（ブラウンフィールド）

**使うもの**: bitz-ddd（+ bitz-sdd）

「このモノリス、マイクロサービスに分けるべき？」に定量的な根拠を作る。

1. 「このコードベースの成熟度を評価して」
2. `ddd-evaluate` が DDD 12基準（ユビキタス言語〜Ports & Adapters）と
   MMI 4軸（凝集度・結合度・独立性・再利用性）で採点し、`.spec/reviews/` にレポート
3. スコアから移行準備度（Mature / Moderate / Needs Improvement / Immature）と
   改善優先度トップ3〜5 が出る
4. 統合改善計画 → `sdd-design` の再設計工程へ

## UC-3: 本格的な DDD でドメインを設計する

**使うもの**: bitz-ddd + bitz-sdd

1. 「決済ドメインのストーリーを書こう」→ `ddd-story` が
   1ペルソナ × 1重要ジョブのハッピーパスを物語化（mermaid sequenceDiagram 付き）
2. 「モデル化して」→ `ddd-model` が2パス導出
   （ストーリーからの明示的導出 + CRUD マトリクスでの暗黙概念の洗い出し）で
   Entity / Value Object / Aggregate と境界づけられたコンテキストを確立
3. トランザクション境界 = 集約境界が `sdd-data` の永続化戦略にそのまま引き継がれる

## UC-4: 複数エージェントで並列開発する

**使うもの**: bitz-sdd（sdd-implement + sdd-git）

1. `sdd-implement` が要件をタスクへ分解（1タスク1ファイル、implements / depends_on / boundary 宣言）
2. `depends_on` が空で `boundary` が互いに素なタスク群だけを並列投入
3. `sdd-git` の worktree 運用: **1エージェント = 1 worktree = 1ブランチ**。
   共有チェックアウトへの同時書き込みが物理的に起きない
4. 失敗したら worktree ごと破棄してタスク再投入（`git reset --hard` 不要、main は無傷）
5. squash merge で「1タスク = 1コミット」、フッター `Implements: FR-012` で
   `spec_inspect.py` の traceability と突合

## UC-5: チーム開発を GitHub Issue 駆動で回す

**使うもの**: bitz-sdd（sdd-git）

1. `.spec/spec-issues/` の提案を GitHub Issue 化（frontmatter `github_issue:` で双方向リンク）
2. `feat/<issue#>-<slug>` ブランチ → Draft PR → CI（テスト + spec_inspect）→ レビュー → squash merge → `Closes #N`
3. PR タイトルは Conventional Commits（CI で機械検査するのを推奨）

## UC-6: 自分専用のスキルを作って配布する

**使うもの**: skill-creator

1. 「社内 API の呼び出し手順をスキルにしたい」→ `skill-creator` が対話で設計
2. `skill-validator` → `skill-tester` → `skill-evaluator` で品質を確認
   （スキルあり / なしの比較で「本当に効いているか」を測る）
3. `skill-packager` で `~/.claude/skills/` へ配置、または zip / プラグイン化して配布
4. 運用開始後は `skill-instrumenter` で計装しておくと、失敗が
   `evals/observations/` に自動記録され、`skill-improver` が改善を提案する

## UC-7: 両対応プラグイン（Claude Code + Antigravity）を作る

**使うもの**: plugin-creator

1. `/plugin-creator:create-plugin 経費精算を自動化するプラグイン` で 8 フェーズのガイドが開始
2. 構成の決定（`plugin-structure`）: マニフェスト2つ・中核はスキル・`model: inherit`
3. コンポーネント実装: フックで危険操作をブロック（`plugin-hooks`）、
   外部 API は MCP 統合（`plugin-mcp`）、設定は `.local.md`（`plugin-settings`）
4. `plugin-validator` エージェント + `agy plugin validate` で両対応を検証

## UC-8: リポジトリの規約を「機械検証できる要件」にする

**使うもの**: bitz-sdd（ドッグフーディング）

規約文書は書いただけでは守られない。この リポジトリ自身がやっている方法:

1. ルートに `.spec/` を置き、AGENTS.md の規約を EARS 要件（`CORE-CON-*`）として起票
2. 各要件に verification_method と実在する検証手段（pytest / CI / チェックリスト）を紐づけ
3. `spec_inspect.py` が traceability を常時検証。CI が release_check + pytest を強制
4. エージェント発の改善提案は `.spec/spec-issues/` 起票 → 人間裁定 → 実装の一方通行
   （「AI が勝手に直した」を構造的に防ぐ）

実例はこのリポジトリの `.spec/` を参照（要件・タスク・spec-issue が2周分入っている）。
