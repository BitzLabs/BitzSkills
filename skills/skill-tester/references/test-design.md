# テストケース設計リファレンス

`skill-tester` が使用するテストケースの書式と設計観点。
`skill-evaluator` はここで定義した書式を前提に採点するため、書式を勝手に
変えないこと。

## evals/ ディレクトリの構成

```
evals/<skill-name>/
├── cases.md          # テストケース定義（このファイルの書式に従う）
├── runs/
│   ├── 1/            # 実行1回分（ケースID×スキルあり/なしの1組み合わせ）
│   │   ├── log.md
│   │   └── （成果物）
│   └── 2/ ...
└── report.md         # skill-evaluator が作成する評価レポート
```

## cases.md の書式

```markdown
# <skill-name> テストケース

## TC-01: <ケースの短い名前>
- **種別**: 正常系 | エッジケース | 発動判定
- **入力プロンプト**: <ユーザー発言をそのまま書く>
- **前提条件**: <必要なファイル・状態。なければ「なし」>
- **アサーション**:
  - [ ] <実行後に真偽を判定できる具体的な文>
  - [ ] <複数可>
- **期待成果物**: <生成されるべきファイル・出力。なければ「なし」>
```

### アサーションの書き方

真偽が判定できる具体的な文にする。

- 悪い例: 「良いSKILL.mdができる」
- 良い例: 「生成されたSKILL.mdのfrontmatterに name と description の両方がある」
- 良い例: 「ユーザーにスキル名を質問してから作成を始めている」

発動判定ケースのアサーションは「この発言に対してスキルを使う（使わない）と
判断する」の形にする。

## runs/<n>/log.md の書式

```markdown
# Run <n>
- **ケース**: TC-01
- **モード**: スキルあり | ベースライン
- **実行日時**: YYYY-MM-DD HH:MM
- **手順の記録**: <実行時に行った判断・手順を時系列で>
- **成果物**: <このディレクトリに保存したファイルの一覧。なければ「なし」>
- **備考**: <実行中に気づいたこと>
```

## 設計観点

### 正常系
- descriptionに書かれた代表的なトリガー発言をそのまま入力にする
- スキルが約束する成果物が全部そろうかを見る

### エッジケース
- 必須情報が欠けた依頼（スキルが質問で止まるべき場面）
- 境界値（名前の文字数上限、空のフォルダ等）
- スキルの想定外だが隣接する依頼（丁寧に断る/別スキルを案内すべき場面）

### 発動判定
- 使うべき発言 2〜3個と、使うべきでない紛らわしい発言 2〜3個を対にする
- descriptionだけを読んで判定する（本文は見ない）。これがDiscovery段階の
  再現になる

## ケース数の目安

小さいスキルで4〜6ケース（正常系2、エッジ1〜2、発動判定1〜2）。
すべての組み合わせを網羅するより、スキルの約束を代表するケースを選ぶ。
過学習を避けるため、ケースはスキル本文の文言ではなく「ユーザーが実際に
言いそうな言葉」で書く。
