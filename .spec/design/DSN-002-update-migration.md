## 1. 目的と適用範囲

`<plugin名>:update` の最小契約は現状「バージョン更新と依存再確認」のみで
（`plugins/plugin-creator/skills/plugin-structure/references/lifecycle-skills.md`）、プラグインが
**配置先に残す状態**の形式がバージョンアップで変わったときに、旧形式のまま取り残される問題を
救済しない。本機構は、バージョン単位の**状態変換ステップ**を累積（スタック）構成で定義し、
`update` が配置先の現在バージョンから最新バージョンまで中間バージョンを順次経由して適用できる
ようにする。

移行対象となる「状態」の具体:

- 配置先スキルの frontmatter `metadata` スキーマ（キーの追加・改名・構造変更）
- 各ワークスペースの `.spec` 系 frontmatter の書式（id 体系・必須キー等）
- `.claude/<plugin>.local.md` 等の設定ファイル（レジストリ・委譲マトリクス等の構造）

**適用範囲外（責務外）**: スキルファイル本体（SKILL.md・references）の置き換え可否判定。これは
skill-packager が担う（第4節）。

## 2. マイグレーションステップの最小契約

### 2.1 1ステップが持つ属性

| 属性 | 必須 | 内容 |
|---|---|---|
| `from` | ○ | このステップが入力として受け付ける状態の直前の形式変更バージョン（semver）。最初のステップはプラグイン初版（baseline）を指す |
| `to` | ○ | このステップ適用後に到達する形式変更バージョン（semver）。`from < to` |
| `targets` | ○ | 変換対象（例: `deployed:SKILL.md#metadata` / `workspace:.spec/*/frontmatter` / `config:.claude/<plugin>.local.md`）。ファイル横断可 |
| `transform` | ○ | 変換内容の記述（何を→何に）。宣言的に、before/after の例を含める |
| `idempotency` | ○ | 冪等性の根拠。二重適用の判定条件（例:「新キー `x` が既に存在すれば no-op」） |
| `rollback` | ○ | 失敗時の復元手順（原則: 適用前バックアップからの復元。破壊的置換を避け加算的変換を推奨） |
| `verify` | 推奨 | 適用成功の機械確認（例: YAML パース可・新キー存在・旧キー不在） |

### 2.2 冪等性の要件

各ステップは、同一状態に二重適用しても壊れないこと。ステップ本文は「まず適用済みか判定 →
未適用なら変換」の形（guard 付き）で記述する。本リポジトリのマイグレーションは**Markdown 命令文**
（エージェントが解釈実行）であり、実行コードではないため、冪等性は自然には保証されない。ゆえに
guard 条件を各ステップに明記することを必須とする（未解決論点、第7節）。

### 2.3 ファイル配置規約（2案）

**案A: 1ファイル1ステップ** — `references/migrations/<from>-to-<to>.md`（例 `1.2.0-to-2.0.0.md`）

- 利点: 追加が非破壊（差分レビューが小さい、`git revert` 粒度が1ステップ）。ファイル名から
  from/to が機械抽出でき、チェーン解決が容易。スキル自己完結原則（フォルダ単位コピー）と親和。
- 欠点: ステップ数が増えるとファイルが散らばる。全体像の一覧性が低い。

**案B: 単一マニフェスト** — `references/migrations.yaml`（または `.md` の表）にステップを列挙

- 利点: 全ステップを一望でき、チェーンの連続性を目視確認しやすい。
- 欠点: 追加のたび同一ファイルを編集（コンフリクト源）。1ステップの revert が難しい。巨大化。

**推奨: 案A**。理由は「1PR=1関心事」「加算的追加」「チェーン抽出の機械可読性」に合致し、
スキルのフォルダ単位配布とも整合する。全体像は `doctor` 側の一覧表示で補完できる。

## 3. 解決アルゴリズム（チェーン解決）

### 3.1 前提となる規約

ステップは**連続チェーン**として著述する: あるステップの `to` は、次に形式が変わるステップの
`from` と一致する（連結リスト）。`metadata.version` はあらゆる変更で bump されるが、マイグレーション
ステップは**状態形式が実際に変わった版にのみ**存在する（飛び番＝形式無変更の bump は正常で
ステップ不要）。

### 3.2 手順

1. **D** = 配置先状態の記録バージョン（`.claude/<plugin>.local.md` 等、`init`/`update` が
   stamp するプラグイン version の自己記述）、**T** = ライブラリ側のプラグイン version
   （`plugin.json`。3マニフェスト共通）を読む。どちらか欠落・不正形式なら解決不能→
   **安全側停止**（semver 比較の手順は skill-packager の規約を再利用する）。
2. `D >= T` なら移行不要（バージョンアップ自体が無いか降格）→終了。
3. 全ステップを収集し `to` 昇順にソート。**適用候補** = `D < to <= T` を満たすステップ集合。
4. 適用候補が空 → `D`〜`T` 間に形式変更なし→移行不要で終了（正常。飛び番の正当ケース）。
5. **連続性検査**（ここが「中間バージョン欠落」の判定）:
   - 候補の先頭ステップの `from` が「`D` 以下の直近形式境界（無ければ baseline）」と一致しないなら
     →**GAP**。特に **`D` が先頭ステップの `from` より古く、その間を埋めるステップが存在しない**
     場合（＝配置先が古すぎて変換の入り口が無い）は安全側停止。
   - 隣接する候補ステップ `i, i+1` について `step[i].to != step[i+1].from` なら→**GAP**
     （チェーンの断裂）→安全側停止。
6. 連続性 PASS なら、`from` の小さい順に**逐次適用**する。各ステップ適用後に `verify` を実行し、
   失敗したら当該ステップの `rollback` を実行して停止・報告。

### 3.3 安全側動作の具体的判定条件

以下のいずれかで `update` は**変換を行わず停止し、状態と不足を人間へ提示**する（無視して先に
進めない）:

- (a) `D` または `T` が semver として解釈不能
- (b) `D` が最古ステップの `from`（baseline）より古く、その間を接続するステップが無い
- (c) 適用候補チェーンに断裂がある（`step[i].to != step[i+1].from`）
- (d) 任意のステップの `verify` が適用後も不成立、かつ `rollback` 後も回復不能
- (e) 変換対象ファイルが配置先に存在しない／YAML パース不能で入力形式を判定できない

(a)〜(c) は**適用前**（dry-run 段階）に検出でき、書き込み前に停止できる。

## 4. skill-packager との責務境界と実行順序

| 機構 | 担当 | 判定/操作 | 正の場所 |
|---|---|---|---|
| **安全判定**（置き換え可否） | skill-packager | 配置先が別ライブラリ由来でない・配置先 < ライブラリ・直接編集の形跡が無い | `plugins/skill-creator/skills/skill-packager/references/lifecycle.md` |
| **マイグレーション**（状態変換） | 本機構（`update`） | frontmatter/`.spec`/`.local.md` の形式を D→T へ累積変換 | 本設計（plugin-creator の update reference） |

**二重規定を避ける**: 「ファイルを置き換えてよいか」は packager のみ、「置き換え後の状態を
どう直すか」は本機構のみが規定する。バージョン比較（semver）ロジックは packager 側を正とし、
本機構は D/T の読み取りにそれを再利用する。

### 4.1 実行順序（推奨フロー）

```
1. D を配置先の状態記録（.local.md 等）から読む（ファイル置き換え前に必ず捕捉）
2. skill-packager 安全判定（3項目。直接配置の場合）→ FAIL なら停止・報告
3. チェーン解決 + 連続性検査（dry-run、書き込みなし）→ GAP なら停止（第3.3節）
4. 変換内容をユーザーへ提示し承認取得（第5節。特に配置先がリポジトリ外の場合）
5. 状態のバックアップを取得
6. スキルファイルの置き換え（直接配置は skill-packager + 再 stamp、プラグイン導入は
   プラットフォームの plugin update が実施済み）
7. マイグレーションステップを from 昇順に逐次適用 + 各 verify
8. 全ステップ成功後に、状態記録バージョンを D→T へ stamp（コミットマーカー。最後に行う）
9. いずれか失敗 → rollback（バックアップ復元）→停止・報告。D の stamp が未更新のまま残るため
   update は再実行可能で、各ステップ冪等（guard）により修正後の再走で回復する
```

**擬似トランザクションの確定**（2026-07-18 裁定に伴い解決）: 状態記録バージョンの stamp
（手順8）を全ステップの verify 成功後に置くことをコミットマーカーとする。途中失敗時は
D が旧値のまま残るので、再実行すれば同じチェーンが再解決され、guard により適用済みステップは
no-op で通過する。スキルファイル本体の version（packager 軸）と状態記録バージョン
（本機構の軸＝プラグイン version）は別管理であり、ファイルは新・状態は旧の期間が生じても
次回 update の再走で収束する。マイグレーション定義は**ライブラリ側**（新バージョンの
プラグイン同梱物）にあるため、手順7は配置先の旧ファイルに依存しない。

## 5. リポジトリ外書き込みへの配慮

配置先は `~/.claude/skills/`・`~/.gemini/config/skills/`・対象プロジェクトの
`.claude/<plugin>.local.md` 等、**このリポジトリ外**になりうる。AGENTS.md ガードレール
「リポジトリ外への書き込みは事前確認」に該当するため、以下の承認フローを必須とする:

1. **dry-run で差分を生成**: 各ステップの変換対象ファイル一覧・before/after プレビュー
   （frontmatter や設定ファイルの該当区間）を提示。
2. **書き込み範囲の明示**: 書き込み先の絶対パスと、それがリポジトリ内/外のいずれかを明示。
3. **ユーザー承認を取得**: 承認が得られるまで一切書き込まない（バックアップ取得含め、実書き込みは
   承認後）。
4. **バックアップの所在提示**: 失敗時 rollback のためのバックアップ先を提示。

これは packager 側の「変更点の要約を提示して承諾を得る」と同一思想で、本機構ではさらに
**状態ファイルの before/after 差分**まで提示する点が追加要件。

## 6. CORE-CON-008 との関係（統合 vs 独立）

| 案 | 内容 | 利点 | 欠点 |
|---|---|---|---|
| **統合** | CON-008 の受入基準に移行機構を追記 | 「update の契約」が1要件に集約。参照が単純 | CON-008 は既に **verified**。受入基準の追記は要件を再オープンし、verified を巻き戻して再検証が必要。改訂履歴が肥大 |
| **独立**（CORE-CON-009 相当） | 別要件として起票し CON-008 に依存で連結 | verified な CON-008 を触らない。移行機構は**任意能力**（形式変更のあるプラグインのみ実装）で、独自の検証項目（冪等性・安全側停止）を持つため要件分離が自然。単独 revert 可 | 「update の契約」が2要件に分散。参照時に両方を見る必要 |

**裁定: 独立（CORE-CON-009）を採用**（2026-07-18 人間裁定）。理由: (1) CON-008 は verified 済みで、
再オープンのコストと改訂履歴の汚染を避ける。(2) 本機構は全プラグインに必須ではない任意拡張であり、
独自の受入基準（安全側停止・冪等性）を持つため要件として分離した方が検証境界が明確。
(3) SI-CORE-028 も「軽量レーン不可・Design Gate 必須」としており、独立要件として通常フローに
載せる方が整合する。CORE-CON-009 は `derived_from: CORE-CON-008` で連結する。

## 7. 論点の解決記録（Design Gate 裁定・設計確定）

起草時の未解決論点8件は、2026-07-18 の人間裁定（1・3・5）とそれに伴う設計確定（2・4・6・7・8）で
以下のとおり解決した:

1. **要件化の粒度** → **裁定: 独立要件 CORE-CON-009**（第6節）。
2. **バージョン軸の二重性（書き忘れ検出）** → チェーン断裂の静的検出（ファイル名 from/to の
   機械抽出による連続性検査）は将来 release_check へ追加可能とし、本要件のスコープ外とする。
   「形式変更にステップを書き忘れた」ことの機械検出は原理的に困難なため、reference に
   「状態形式を変えた PR はマイグレーションステップを同梱する」規律として明文化する。
3. **定義形式** → **裁定: 宣言的 Markdown + guard 必須**。案A（1ファイル1ステップ、
   `references/migrations/<from>-to-<to>.md`）を採用し、guard（適用済み判定）の記述を必須
   セクションとして書式を厳格化する。実行可能スクリプト化は採らない（プラグインのフォルダ単位
   配布・エージェント解釈というスキル基盤に整合させる）。
4. **トランザクション順序** → 状態記録バージョンの stamp を全 verify 成功後の最終手順に置く
   コミットマーカー方式で確定（第4.1節）。
5. **バージョン軸** → **裁定: プラグイン version**（`plugin.json`、3マニフェスト共通）。
   D は配置先状態の自己記述（`.claude/<plugin>.local.md` 等に `init`/`update` が stamp）から
   読む。skill-packager の安全判定（スキル単位 version）とは軸を分離し、責務境界（第4節）と
   一致させる。
6. **このリポ自身の `.spec` が対象の場合** → 書き込み先がリポジトリ内のためリポジトリ外承認
   フロー（第5節）は適用しないが、dry-run 差分の提示は同様に行い、通常のブランチ規律
   （main 直コミット禁止）に従う。reference に消費側プロジェクトとの区別を明記する。
7. **`doctor` との連携** → 任意（MAY）。安全側停止条件 (a)〜(c) は読み取り専用で判定できるため、
   `doctor` が未適用マイグレーション・チェーン断裂を事前検知して提示してよい（実装は各
   プラグインの任意）。
8. **テスト戦略** → 合成フィクスチャ（旧形式の配置先を模した frontmatter / .local.md）による
   冪等性・安全側停止の確認手順を reference に記載する。機構自体は規定（reference）のみで
   実行コードを持たないため、自動回帰テストは migrations を同梱する実プラグイン側の検証工程
   （sdd-test）で行う。

## 8. 裁定記録

- 2026-07-18 人間裁定（チャット、AskUserQuestion 回答）:
  - 要件化の粒度 = 独立要件 CORE-CON-009
  - 定義形式 = 宣言的 Markdown + guard 必須（案A: 1ファイル1ステップ）
  - バージョン軸 = プラグイン version
  - Design Gate = 再検討のうえ問題なければ実装まで進める（approved 化を承認）
