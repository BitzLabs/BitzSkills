# DSN-003 ライフサイクルスキル横展開の必要操作マトリクス

- **背景 / 課題**: CORE-CON-008（標準ライフサイクルスキル）と CORE-CON-009
  （update 累積マイグレーション機構）は規約制定のみで、先行実装の bitz-env 以外に
  実装プラグインが無い。SI-CORE-031（横展開）/ SI-CORE-032（update 実装）の
  accepted 化を受け、どのプラグインにどの操作を実装するかを本設計で確定する。
  標準の原則は「提供する操作だけを標準名で命名」（全4操作の一律実装は不要）。

- **設計判断（必要操作マトリクス）**:

  | プラグイン | init | doctor | update | uninstall | 判断根拠 |
  |---|---|---|---|---|---|
  | bitz-env | ✅済（env-init） | ✅済（env-doctor） | **⬜ 今回実装（env-update）** | ✅済（env-uninstall） | 配置先状態が最多（permissions・マーカー区間・agents・レジストリ）。SI-CORE-032 の本命 |
  | bitz-sdd | ❌不要 | **⬜ 今回実装（sdd-doctor）** | ⏸保留 | ❌不要 | doctor は依存 bitz-flow>=0.2 と `scripts/spec` ラッパー解決の診断（SI-CORE-010 の確認観点）。init は sdd-core の scaffold と責務重複のため作らない |
  | bitz-ddd | ❌不要 | **⬜ 今回実装（ddd-doctor）** | ❌不要 | ❌不要 | 併用前提の bitz-sdd を `metadata.dependencies` に宣言追加（CORE-FR-013 準拠）し、doctor がそれを診断する |
  | bitz-flow | ❌不要 | **⬜ 今回実装（flow-doctor）** | ❌不要 | ❌不要 | git / gh CLI の前提環境診断のみ。配置物なし |
  | plugin-creator | ❌不要 | ❌不要 | ❌不要 | ❌不要 | リポジトリ内で完結し配置物・外部依存なし。診断対象が無い |
  | skill-creator | ❌不要 | ❌不要 | ❌不要 | ❌不要 | 配置は skill-packager 自身のライフサイクル管理（installed-at stamp / 安全判定）が既に担う。二重規定を避ける |

  - **命名**: 先行実装 bitz-env の接頭辞式に統一する（`sdd-doctor` / `ddd-doctor` /
    `flow-doctor` / `env-update`）。プラグイン名前空間と合わせ `bitz-sdd:sdd-doctor` の形で
    CORE-CON-008 の「`<plugin名>:<操作>`」を満たすと解釈する（bitz-env の先行例を踏襲）。
  - **doctor の共通契約**: 読み取り専用。診断結果は「OK / 問題 + 修正案」の形で報告し、
    `metadata.dependencies` 宣言があるプラグインは有効性 + semver 制約充足を診断する
    （lifecycle-skills.md の最小契約に準拠）。
  - **env-update の範囲**: CORE-CON-009 の受入基準（EARS 7項目）に準拠した機構
    （チェーン解決・安全側停止・冪等 guard・リポジトリ外書き込みの dry-run + 承認）を実装する。
    現時点で形式変更が無いため `references/migrations/` は空で出荷し、
    migration-steps.md の合成フィクスチャ手順で機構の冪等性・安全側停止を検証する。

- **代替案と却下理由**:
  - bitz-sdd への `sdd-update` 同時実装（SI-CORE-032 一次案）: CORE-CON-009 は
    「状態の形式変更を伴わないプラグインには不要」であり、`.spec` 書式の破壊的変更は
    現在計画されていない（docs 旧8章移行は sdd-docs が既に保有）。**形式変更が実際に
    発生した時点で実装**する方が YAGNI に適うため保留（却下ではなく延期。
    発生時は新規 spec-issue で起票する）。
  - 全プラグインへの doctor 一律実装: plugin-creator / skill-creator は診断対象が無く、
    空の doctor は標準の趣旨（独自命名の乱立防止）に貢献しないため実装しない。
  - `init` の bitz-sdd 実装: `.spec/` 初期化は sdd-core の `spec_scaffold.py` と
    sdd-docs の初期化が既に担っており、責務重複になるため作らない。

- **影響範囲・ロールバック**: 新スキル4つの追加と bitz-ddd の依存宣言追加のみで、
  既存スキルの動作変更なし。プラグイン単位（1プラグイン = 1 PR）で独立に revert 可能。
  委任先要件: SDD-FR（sdd-doctor）/ DDD-FR（ddd-doctor）/ FLW-FR（flow-doctor）/
  ENV-FR（env-update）。ルート側は SI-CORE-031/032 の `delegated_to:` で完了追跡する。
