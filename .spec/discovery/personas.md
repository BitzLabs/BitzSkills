# ペルソナとJobs-to-be-Done

> 実利用の根拠は現在の管理者によるドッグフーディングが中心。共同開発者像は
> `[proto / 未検証]` とし、発言や感情を事実として捏造しない。

## Jobs-to-be-Done

- **J1（開始）**: 新しいリポジトリでAIエージェント開発を始めるとき、安全設定、Git規律、
  仕様・検証方法を一から説明せず、BitzLabsの標準状態から開始したい。
- **J2（継続）**: エージェントやセッションを切り替えるとき、開発規律が失われたり解釈が変わったりせず、
  同じ契約と検証方法を引き継ぎたい。
- **J3（変更）**: 標準規律を改善するとき、各プロジェクトを手作業で直さず、バージョン付きプラグインとして
  配布し、安全に移行したい。
- **J4（復旧）**: エージェントの作業が失敗したとき、禁止された破壊的操作に頼らず、変更を局所化してやり直したい。
- **J5（監査）**: 完了を報告するとき、自己申告ではなく検証コマンドの実出力を提示したい。

## ペルソナ

### A — BitzLabs の開発者・標準環境メンテナ（主要、実在）

- **文脈**: 複数のAIエージェントとモノレポ／個別リポジトリを開発し、標準自体も保守する。
- **Jobs**: J1〜J5。
- **Pains**: 環境別の仕様差、同じ規律の再説明、更新ドリフト、危険操作への不安。
- **Gains**: プラグイン導入だけで共通規律を再現し、機械検証で完了を判断できる。
- **根拠**: BitzSkills リポジトリでのドッグフーディング。利用規模は現状 n=1 系統。

### B — BitzLabs の共同開発者（served、proto）

- **文脈**: 既存プロジェクトへ参加し、選択したAIエージェントで作業する。`[proto / 未検証]`
- **Jobs**: J1、J2、J5。
- **Pains**: プロジェクト固有ルールの所在と、完了条件が分からない。`TBD`
- **Gains**: 標準プロファイルとAGENTS.mdから、同じ開始点と検証手順を得る。
- **生の声**: `TBD`（ユーザー調査なし）。

### C — 実行主体としてのAIエージェント（システム参加者）

- 顧客ではないが、標準環境の主要な実行主体。
- 必要条件: 明確な権限境界、段階的に読めるスキル、定型スクリプト、失敗時の安全な停止。
- 成功状態: 利用プラットフォームが変わっても、中核規律に従い同じ検証証跡を残せる。

### アンチペルソナ

- BitzLabs規律を採用せず、既に別の組織標準・ガードレール・開発フローで統制されているチーム。
- OS／クラウド／IDEを含む完全な開発者端末プロビジョニングだけを求める利用者。

## 優先順位付き機会

1. 標準プロファイルの導入手順を短く、3プラットフォームで同じ構造にする。
2. AGENTS.md、スキル、検証スクリプトの責務を明確にし、再説明を減らす。
3. 更新・doctor・migrationで導入後のドリフトを検知する。
4. 実利用プロジェクトが増えた時点でペルソナBの仮説を検証する。
