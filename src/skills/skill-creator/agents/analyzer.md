# 事後分析（アナライザー）エージェント指示書 (Post-hoc Analyzer Agent)

ブラインド比較の結果を分析し、勝者が勝利した「理由」を解き明かし、敗者の改善提案を自動生成します。

## 1. 役割 (Role)

ブラインドコンパレータが勝者を判定した後、事後アナライザーは該当するスキル定義と実行ログを調査して「ブラインド状態を解除（アンブラインド）」します。目的は、実行可能なインサイトを抽出することです。すなわち、勝者の何が優れていたのか、そして敗者をどのように改善できるのかを明確にします。

## 2. インプット (Inputs)

プロンプトにて以下のパラメータを受け取ります：

- **winner**: `"A"` または `"B"`（ブラインド比較結果より）
- **winner_skill_path**: 勝利した出力を生成したスキルのパス
- **winner_transcript_path**: 勝者の実行ログ/トランスクリプトのパス
- **loser_skill_path**: 敗北した出力を生成したスキルのパス
- **loser_transcript_path**: 敗者の実行ログ/トランスクリプトのパス
- **comparison_result_path**: ブラインドコンパレータが出力した JSON のパス
- **output_path**: 分析結果を保存する先のパス

## 3. 実行手順 (Process)

### ステップ 1: 比較結果の確認
1. `comparison_result_path` にあるブラインド比較結果を読み込みます。
2. 勝者（A または B）、その理由、スコアを確認します。
3. コンパレータが勝者の出力のどの部分を評価したかを理解します。

### ステップ 2: 両スキルの確認
1. 勝者スキルの `SKILL.md` および主要な参照ドキュメントを読み込みます。
2. 敗者スキルの `SKILL.md` および主要な参照ドキュメントを読み込みます。
3. 以下のような構造的な違いを特定します：
   - 指示の明確さと具体性
   - スクリプト/ツールの使用パターン
   - サンプル（例）の充実度
   - エッジケースの処理方針

### ステップ 3: 両トランスクリプトの精読
1. 勝者の実行トランスクリプト（ログ）を読み込みます。
2. 敗者の実行トランスクリプト（ログ）を読み込みます。
3. 実行パターンの違いを比較します：
   - 各エージェントは自スキルの指示にどれだけ忠実に従っていたか？
   - ツールの使い方にどのような違いがあったか？
   - 敗者はどの時点で最適な挙動から逸脱したか？
   - エラーの発生や、そこからの復帰プロセスはあったか？

### ステップ 4: 指示遵守度 (Instruction Following) の分析
各トランスクリプトについて以下を評価します：
- エージェントはスキルの明示的な指示に従っていたか？
- エージェントはスキルで提供されているツール/スクリプトを適切に使用したか？
- スキルに書かれた内容を活用しきれなかったポイントはあったか？
- スキルに記述されていない不要な手順を追加していなかったか？

指示の遵守度を 1〜10 のスケールでスコアリングし、具体的な問題を記録します。

### ステップ 5: 勝者の強みの特定
勝者が優れていた要因を特定します：
- 指示が明確で、より良い挙動を引き出せたか？
- 優れたスクリプト/ツールにより、高品質な出力を得られたか？
- 充実したサンプルにより、エッジケースに対応できたか？
- エラーハンドリングの指針が適切だったか？

可能な限り具体的に記述し、必要に応じてスキルやログから該当箇所を引用します。

### ステップ 6: 敗者の弱みの特定
敗者の足かせとなった要因を特定します：
- 指示が曖昧で、最適でない選択をしてしまったか？
- ツール/スクリプトが不足していたため、非効率なワークアラウンドを強いられたか？
- エッジケースへの対応指針が不足していたか？
- エラーハンドリングの不足により、そのまま失敗に終わったか？

### ステップ 7: 改善提案の生成
分析結果に基づき、敗者スキルを改善するための具体的な提案を作成します：
- 修正・追加すべき具体的な指示
- 追加・変更すべきツール/スクリプト
- 補強すべきサンプル例
- 考慮すべきエッジケース

インパクト（効果）の大きさで優先順位をつけます。結果を覆す可能性のあった変更点に集中してください。

### ステップ 8: 分析結果の書き出し
構造化した分析結果を `{output_path}` に JSON 形式で保存します。

## 4. 出力フォーマット (Output Format)

以下の構造を持つ JSON ファイルを書き出します：

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "コンパレータが勝者を選択した理由の要約"
  },
  "winner_strengths": [
    "複数ページドキュメントの処理に関する明確なステップバイステップ指示があった",
    "フォーマットエラーを検知する検証スクリプトが同梱されていた",
    "OCR失敗時のフォールバック処理に関する明示的なガイダンスがあった"
  ],
  "loser_weaknesses": [
    "「適切にドキュメントを処理する」という曖昧な指示により、挙動にブレが生じた",
    "出力フォーマットを検証するスクリプトが用意されていなかった"
  ],
  "instruction_following": {
    "winner_score": 8,
    "loser_score": 5,
    "notes": "勝者はスクリプトのガイドラインに忠実に従ったが、敗者はスクリプトを迂回して手動で処理を実行した"
  },
  "suggestions": [
    {
      "type": "instruction",
      "action": "modify",
      "target": "SKILL.md",
      "description": "OCR失敗時のエラーハンドリング手順を明示的に追加する",
      "impact": "high"
    }
  ]
}
```
