# スキルクリエイター (Skill Creator)

本スキルは、新しいカスタムエージェントスキルの作成、および既存スキルの反復的な検証・改善（リファイン）を支援するためのメタスキルです。

高レベルにおいて、スキル作成プロセスは以下のように進行します：

1. **要件定義**: スキルに何をさせたいか、およびどのように実行すべきかの概要を決定する。
2. **ドラフト作成**: スキルの初回ドラフト（`SKILL.md`）を執筆する。
3. **テスト設計と実行**: 評価用のテストプロンプトを作成し、対象スキルを有効化した状態でエージェントにタスクを実行させる。
4. **評価・レビュー**: 実行結果を定性的および定量的に評価する。
   - バックグラウンドで評価が実行されている間に、必要に応じてアサーション（合格期待条件）を作成または編集する。
   - `eval-viewer/generate_review.ts` スクリプトを使用してダッシュボードを起動し、定性的な出力結果と定量的な評価スコアをブラウザで確認する。
5. **リファクタリング**: 評価結果やユーザーからのフィードバック（または定量的なベンチマークで見つかった欠陥）に基づき、スキルの指示内容を改善する。
6. **最適化**: 満足するまで 4〜5 のサイクルを繰り返し、さらに大規模なテストケースへと拡張する。
7. **トリガーの最適化**: 必要に応じて説明文の最適化スクリプト（`run_loop.ts`）を実行し、スキルの説明文（YAMLフロントマターの `description`）を微調整してトリガーの確度を最大化する。

エージェント（あなた）の役割は、ユーザーが現在このプロセスのどの段階にいるかを確認し、次のステップへと進められるよう伴走することです。

## ユーザーとのコミュニケーションガイドライン

スキルクリエイターは、コマンドラインやプログラミング用語に慣れていない初心者から、熟練のエンジニアまで、幅広いユーザーによって使用されます。相手の理解度や発言の文脈に応じて、説明のレベルを調整してください。
- 専門用語（例: 「アサーション」「JSON」「JSONストリーム」など）を説明なしで使用する場合は、ユーザーがそれらを理解している兆候があるか確認してください。
- 必要に応じて、短い定義や補足を付記することで、ユーザーが迷わずに指示コマンドを実行できるようにしてください。

## 1. スキルの作成プロセス

### ① 意図の把握 (Capture Intent)
現在の会話履歴（過去の実行手順、エラーからの回復、ユーザーの修正、I/O形式など）を読み解き、スキル化したいワークフローを抽出します。
以下の項目を明確にしてください：
1. このスキルを読み込むことで、エージェントは何ができるようになるか？
2. スキルはどのような状況やクエリでトリガーされるべきか？
3. 期待される出力フォーマットはどのようなものか？
4. テストケース（検証用の問題）を設定すべきか？（出力が客観的に検証可能なファイルやコードの場合は設定を推奨。主観的な創作タスクなどの場合は不要となる場合があります）

### ② ヒアリングと設計提案
エッジケース、入力/出力形式、依存関係について事前にユーザーに質問します。ヒアリング後、以下の構成案をユーザーに提示します：
- **name**: スキルの識別名（kebab-case）
- **description**: いつスキルを読み込むべきかのトリガー条件。**※Claudeがトリガーし損ねるのを防ぐため、少し「強気（Pushy）」に、トリガーすべき状況や類義語を具体的に記述してください。**
- **フォルダ構成案**: 500行制限に備え、詳細情報をどのサブファイル（`references/` 等）に分離するかの設計。

### ③ SKILL.md の執筆とガイドライン
承認を得たら、`src/skills/<skill_name>/` 配下にファイルを配置します。

#### スキルのフォルダ構成：
```
src/skills/<skill-name>/
├── SKILL.md (必須: YAMLフロントマターと主要指示)
├── scripts/    - 補助用の実行スクリプト（JavaScript/TypeScript等）
├── references/ - contextに必要に応じて読み込ませる詳細仕様書・ドキュメント
└── assets/     - 出力等に使用するアセットファイルや静的テンプレート
```

#### 段階的開示（Progressive Disclosure）の厳守：
- `SKILL.md` の本文は**500行以内**に収めてください。これを超えそうな場合は、詳細を `references/` ディレクトリの別ファイルに切り出し、SKILL.md から相対リンク（`file:///...`）で参照させてください。これによりトークン消費量を節約できます。

#### 指示の記述スタイル：
- 命令文（〜せよ、〜すること）を優先して使用します。
- 出力テンプレートや実行例がある場合は、Markdownブロックで明確に構造を示してください。

## 2. 評価テスト（Evals）の実行とベンチマーク

### テストケースの用意
テストプロンプトを 2〜3 個作成し、`evals/evals.json` に保存します（スキーマは `references/schemas.md` を参照）。

### ステップ 1: テストの実行 (run_eval)
スキルを適用した状態 (with-skill) と、そうでない状態 (baseline) の実行を行います。
検証用サブプロセスを起動するには、以下のコマンドを実行します：
```bash
npx tsx src/skills/skill-creator/scripts/run_eval.ts "<検証用クエリ>" <スキルパス> [コマンド名]
```
※検証コマンドはデフォルトで `claude` を呼び出しますが、環境変数 `SKILL_CLI_COMMAND` または引数で他のコマンド（`antigravity` など）を指定できます。

### ステップ 2: アサーションの作成
テストの実行中に、結果を機械的に判定するための「合格期待値（アサーション）」を定義し、`evals/evals.json` の `expectations` に書き加えます。アサーションは検証可能で客観的な内容にしてください。

### ステップ 3: 採点と集計 (Grade & Aggregate)
テスト実行結果および生成されたファイルに基づき、採点用エージェント（Grader）を起動してアサーションを採点（`grading.json` の作成）し、実行実時間を `timing.json` に記録します。

その後、以下のコマンドを実行して、ベンチマーク結果を統計的に集計します：
```bash
npx tsx src/skills/skill-creator/scripts/aggregate_benchmark.ts <評価結果のワークスペースパス>
```
実行すると、合格率や平均実行時間が計算され、`benchmark.json` およびサマリーMarkdown（`summary.md`）が生成されます。

### ステップ 4: ダッシュボードの起動 (Launch Viewer)
集計結果を定性的・定量的に確認するためのWebビューアを起動します：
```bash
npx tsx src/skills/skill-creator/eval-viewer/generate_review.ts <評価結果のワークスペースパス> --skill-name <スキル名>
```
自動的にブラウザでダッシュボードが開き、A/Bテストの勝敗や、各クエリに対するアサーションの合否を確認できます。

## 3. 説明文の最適化ループ (run_loop)

スキルの説明文（description）は、エージェントがそのスキルを使用するかどうかを判断するための唯一の境界条件です。
説明文を自動で繰り返し推敲し、テストケースにおける「適切なトリガー率（合格率）」を高めるには、以下のループ実行コマンドを使用します：

```bash
npx tsx src/skills/skill-creator/scripts/run_loop.ts <eval_set.jsonのパス> <スキルパス>
```

このコマンドは内部で `run_eval.ts` と `improve_description.ts` を最大試行回数に達するまで自動でループ実行し、最もトリガー精度の高い説明文を導き出します。また、推移を可視化する `optimization_report.html` を自動生成します。
